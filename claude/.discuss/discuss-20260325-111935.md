## 内容摘要

讨论围绕将只读分析型 skill（如 /review、/summary）改造为 subagent 执行模式展开。核心争议在于改造的必要性——R2 极简主义者认为 allowed-tools 已实现只读控制，无需增加复杂性；R3 性能狂热者用上下文保护（单次 /review 消耗 ~30k tokens）和失败隔离论证了改造价值。最终共识为渐进式改造：先改造 /review 和 /summary 两个最纯粹的分析型 skill，验证效果后扩展到 /bench 和 /retro，/discuss 和写操作型 skill 不改造。所有写操作由主 agent 统筹执行。

## 第 1 轮：各角色独立提出核心论点

### 【第一性原理者 R5 | 先问"为什么"】

论点：在改架构之前，先搞清楚当前模式的真正问题是什么。

当前 skill 模式的工作方式是：用户调用 `/review`，skill 内容被加载到主 agent 上下文中，主 agent 按指令执行。这个模式下，"只读"是通过 `allowed-tools` 字段声明的——review 只允许 `Bash(git:*)` 等读操作，不允许 Edit/Write。

可能的动机有三个：
1. **上下文保护**——skill 执行消耗大量上下文窗口，分离到 subagent 可以保护主 agent 上下文
2. **并行执行**——多个只读分析可以并行跑
3. **关注点分离**——主 agent 只做决策和写操作，subagent 做分析

如果动机是 1，这是真实的痛点。如果是 2 或 3，需要权衡引入的复杂性是否值得。

### 【极简主义者 R2 | 反对无理由的复杂化】

论点：当前模式已经通过 `allowed-tools` 实现了只读控制，没有必要引入 subagent 层。

增加的复杂性包括：
- 需要维护两套定义格式（skill YAML + agent markdown）
- subagent 返回结果需要序列化/反序列化
- 主 agent 需要解析 subagent 输出再决定是否执行写操作
- 调试变得更困难——用户看不到中间过程

### 【性能狂热者 R3 | 关注 token 和延迟】

论点：subagent 模式的核心优势是**上下文隔离**和**并行能力**。

一次 /review 完成后，主 agent 可能已经消耗了 20-30k tokens 的上下文，后续交互的效率和准确性都会下降。subagent 模式让主 agent 上下文保持干净，且支持并行执行多个分析。

### 【维护性倡导者 R6 | 三年后谁来维护？】

论点：skill 定义的可维护性是关键考量。

两种不同的定义格式并存（skills/ vs agents/）会增加认知负担。更新 skill 逻辑时需要判断影响的是 skill 还是 agent，新增 skill 时需要决策类型。

### 【架构师 R14 | 模块边界和概念完整性】

论点：Skill 和 Agent 是两个不同概念，有各自清晰的职责边界。将只读 skill 改成 agent 在概念上合理，但关键问题是信息流设计：上下文传递、结构化输出消费、用户交互处理。

## 最终方案

### 核心决策
选择性地将纯分析型只读 skill 改造为 subagent 模式，渐进实施，主 agent 统筹所有写操作。

### 方案细节

**Phase 1（立即实施）：**
- 改造 `/review` 和 `/summary` 为 subagent 模式
- 创建对应的 agent 定义文件
- skill SKILL.md 保留作为调度器角色
- subagent 执行纯分析，返回结构化报告内容
- 主 agent 收到报告后决定是否写入文件

**Phase 2（验证后扩展）：**
- 评估 Phase 1 效果后扩展到 `/bench`、`/retro`
- `/debug` 暂缓

**不改造的 skill：**
- `/discuss`、`/fix`、`/refactor`、`/feature`、`/improve`、`/design`、`/ship`

### 已解决的分歧
- 是否值得改 → 值得，上下文保护和失败隔离是实际收益
- 改造范围 → 渐进式，先改 2 个最纯粹的
- 自动生成 vs 手动维护 → 手动维护
- subagent 是否可以写文件 → 不可以

### 未解决的权衡（需用户决策）
- SKILL.md 的角色：保留作为调度器 vs 直接删除改用 agent
- /debug 是否纳入改造
- subagent 输出详细度：完整报告 vs 摘要+关键发现
