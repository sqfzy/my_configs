## 内容摘要
讨论聚焦于是否为 skill 系统引入子代理并行化。R3 提出全面并行化 5 个 skill 的方案，但被 R2（复杂度）、R6（维护成本）、R4（缺乏数据）质疑。经 4 轮交锋，各方收敛至"只改 `/ship` 单点突破"方案：Gate 1-4（review/test/bench/doc）通过内联 Agent 调用并行执行，不新增文件。R4 提出验证标准（50% 加速、质量无差异），全组设定退出条件：3 次实际使用不达标则回滚。

## 第 1 轮：各角色独立陈述

**【R3 性能狂热者 | 强烈支持并行化】**

论点：当前 skill 系统最大的瓶颈是串行执行。以 `/ship` 为例，Gate 1（review）→ Gate 2（test）→ Gate 3（bench）→ Gate 4（doc）是严格串行的，但它们之间几乎没有数据依赖。这四个 Gate 完全可以作为独立子代理并行启动，最终由 orchestrator 收集结果后决定 Gate 5（commit）。

具体可并行化的 skill 组合：
1. `/ship`：Gate 1-4 并行，Gate 5 串行等待。预计 4x 加速。
2. `/evolve` Phase 2：功能扫描、API 扫描、健康扫描、可观测性扫描可以 4 路并行。
3. `/review`：文件级分析可以按文件或目录拆分为多个子代理。
4. `/test`：覆盖率分析按模块拆分。
5. `/doc`：API doc、README、CHANGELOG、inline comments 四类独立生成。

---

**【R2 极简主义者 | 反对过度架构】**

论点：当前 17 个 skill 已经是一个不小的维护面。引入子代理意味着编排复杂度、错误处理、上下文割裂、调试困难。更关键的是——当前的瓶颈真的是执行速度吗？并行子代理不能减少总 token 消耗，只是降低墙钟时间。对于个人使用的 CLI 工具，优化 ROI 存疑。

---

**【R6 维护性倡导者 | 关注长期维护成本】**

论点：当前 skill 系统的优势是自包含性。引入子代理后，逻辑散布在多个文件中，修改需要同步更新，prompt 设计需要平衡上下文传递和 token 效率。建议即使要做并行化，也应该只对 1-2 个收益最明显的 skill 做。

---

**【R4 实用主义者 | 要求数据支撑】**

论点：需要先回答——各 skill 平均执行时间？用户使用频率？Agent tool 并行限制？上下文隔离是否降低输出质量？在没有数据之前反对做架构决策。

---

**【R12 系统思维者 | 关注全局影响】**

论点：子代理化改变了整个 skill 系统的执行模型。上下文窗口消耗可能反而增加、一致性问题、错误传播策略、二阶效应（复杂度蔓延）。核心观点：并行化的真正价值在于只读分析任务，有副作用的任务绝不应并行化。

## 最终方案

### 核心决策
同意并行化，但只从 `/ship` 单点突破——Gate 1-4 改为并行子代理执行，不新增文件。

### 方案细节
1. 改造目标：仅 `/ship/SKILL.md`
2. 实现方式：内联 Agent tool 并行调用
3. 子代理要求：结构化 markdown 输出，明确 PASS/FAIL 判定
4. Gate 5 合并：任一 FAIL → 整体 FAIL
5. 架构约束：只读并行、主 agent 合并、保留串行回退
6. 验证标准：50% 加速、质量无差异、错误清晰
7. 推广条件：验证成功后独立论证

### 已解决的分歧
- 是否值得做 → 限定为单点突破，实施成本低
- 是否新增文件 → 不新增
- 质量保证 → 结构化输出 + 验证标准 + 退出条件
- 复杂度蔓延 → 不推广直到验证成功

### 未解决的权衡
- fail-fast vs wait-all：关心 token 成本选 fail-fast；关心全局视图选 wait-all
