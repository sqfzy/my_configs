## 内容摘要

讨论围绕"破坏性重构（API 重塑、架构重组、核心抽象替换）在现有 skill 体系中无对应入口"的问题展开。R5 明确了四类典型场景，确认存在空白。核心争议在于新建 skill vs 扩展现有 `/refactor`——R2 主张组合使用 /design + /refactor，但 R4 和 R5 指出跨 skill 上下文断裂不可接受。最终共识：扩展 `/refactor` 增加 `[breaking]` 参数模式，启用后切换到独立的迁移策略流程，自动创建分支，每个检查点提交，行为变更必须被显式声明和验证。R1 确保 breaking 模式的安全约束比默认模式更严格而非更宽松。

## 第 1 轮：核心论点

### 【第一性原理者 R5 | 先定义问题】
论点："破坏性重构"拆解为四类场景——API 重塑、架构重组、核心抽象替换、技术栈迁移。共同特征：行为可能改变、测试必须更新、需要分阶段迁移策略。这正是 `/refactor` 排除的，也不属于 `/feature`（非新功能）、`/improve`（非打磨质量）、`/design`（非从零设计）。确认存在空白。

### 【极简主义者 R2 | 组合已有 skill】
论点：`/design`（讨论新架构）+ `/refactor`（执行结构变更）+ `/feature`（行为变化部分）的组合已能覆盖。新增 skill 意味着更多维护成本和用户选择困难。

### 【架构师 R14 | 概念空白不可忽视】
论点：组合方案的缝隙在于流程连续性——迁移策略、进度跟踪、测试策略变更恰好落在两个 skill 的交接处。建议扩展 `/refactor` 支持 `breaking` 模式，而非新建 skill。

### 【实用主义者 R4 | 看用户怎么用】
论点：用户说"重构这个模块的设计"时自然反应是 `/refactor`。被踢到其他 skill 的体验是割裂的。无论方案如何，入口必须是 `/refactor`。

### 【风险卫士 R1 | 安全网必须更强】
论点：破坏性重构的安全网锚点不同——不能再用"测试全部通过"证明正确。需要：独立分支、迁移检查点、行为变更清单。

## 最终方案

### 核心决策
扩展 `/refactor` skill，新增 `[breaking]` 参数模式，支持破坏性的设计层面重构。不新建 skill。

### 方案细节

| 维度 | 默认模式 | breaking 模式 |
|------|----------|---------------|
| 核心原则 | 行为不变是铁律 | 行为变更必须被声明和验证 |
| 分支策略 | 当前分支 | 自动创建 `refactor/<name>` 分支 |
| 测试策略 | 测试数量一致、不修改测试 | 旧测试按计划更新，新测试覆盖新行为 |
| 迁移计划 | 无需 | 必须输出分阶段迁移计划 |
| 行为变更清单 | 无需 | 必须声明每项预期行为变更 |
| 提交策略 | 完成后一次提交 | 每个迁移检查点自动提交 |
| commit footer | 无 | 包含 `BREAKING CHANGE:` |

引导路径：默认模式检测到行为变化时提示 `/refactor breaking <意图>`。

### 已解决的分歧
- 新 skill vs 扩展 → 扩展 `/refactor`
- 安全性是否降低 → 不会，breaking 模式有独立的更严格安全机制
- 用户入口 → `/refactor`，通过参数区分模式

### 未解决的权衡
- breaking 模式的评审深度：保持 3 角色快速评审 vs 升级到 5 角色完整讨论
