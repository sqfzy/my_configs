## 内容摘要

讨论围绕 `/refactor` 和 `/migrate` 的功能重叠问题展开。R2 主张合并（70% 流程重复），但 R5 通过"变更驱动力"这一切分轴（内部设计 vs 外部适配）证明了两者的本质区别。R14 从概念完整性、R1 从风险模型差异两个维度支持分离。最终共识：保持独立，在 TRIGGER 说明中互相引导，公共流程可选提取到 shared/ 目录。模糊场景（如框架替换）默认路由到 `/migrate`。

## 第 1 轮：核心论点

### 【第一性原理者 R5 | 找本质区别】
论点：本质区别在于变更驱动力。`/refactor`（含 breaking）是内部驱动——不满意自己的设计。`/migrate` 是外部驱动——外部世界变了需要适应。实际意义：信息源不同，`/refactor` 读代码，`/migrate` 读 CHANGELOG。

### 【极简主义者 R2 | 主张合并】
论点：两个 skill 的执行流程高度相似（分析影响→计划→分步执行→验证→提交→清理），70% 重复。加上 `/refactor breaking`，重叠更严重。建议把 `/migrate` 能力并入 `/refactor` 作为模式参数。

### 【架构师 R14 | 反对合并】
论点：skill 分类应基于用户意图而非实现细节。"升级 tokio"和"重新设计接口"是完全不同的意图。`/migrate` 有特有领域知识（CHANGELOG 读取、依赖版本锁定、兼容层策略、升级命令），硬塞进 `/refactor` 会导致臃肿。

### 【实用主义者 R4 | 看使用频率】
论点：真正模糊的场景很少。actix→axum 是 `/migrate`（外部驱动），HashMap→BTreeMap 是 `/refactor`（内部设计）。用户在实际使用时不会太纠结。

### 【风险卫士 R1 | 风险模型不同】
论点：`/migrate` 风险来自新版本未文档化行为变化和传递依赖不兼容，需要重运行时验证。`/refactor breaking` 风险来自架构缺陷和调用方迁移不完整，需要重架构评审。混在一起安全检查重点会模糊。

## 最终方案

### 核心决策
保持 `/refactor` 和 `/migrate` 独立，不合并。通过明确的触发边界和互相引导消除歧义。

### 本质区分轴：变更驱动力

| | `/refactor` (默认) | `/refactor breaking` | `/migrate` |
|---|---|---|---|
| 驱动力 | 内部，行为不变 | 内部，行为可变 | 外部，适配变更 |
| 典型场景 | 拆模块、重命名 | 替换核心抽象、重塑 API | 升级依赖、换框架 |
| 信息源 | 项目代码 | 项目代码 | CHANGELOG + 迁移指南 |

### 已解决的分歧
- 合并 vs 分离 → 分离（驱动力不同、信息源不同、风险模型不同）
- 流程重复 → 可通过 shared 引用缓解
- 模糊场景 → 默认路由到 `/migrate`

### 未解决的权衡
- 是否现在提取公共流程到 shared/
