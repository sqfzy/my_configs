## 初始化

解析参数：
- `[max-iterations: N]` 或 `[最大迭代: N]`：最大迭代次数，默认 **10**
- `[target: <path>]`：指定关注的文件或模块，未指定则覆盖整个项目

记录开始时间。创建本次 session 的日志目录：
```bash
mkdir -p .discuss
```
日志文件路径：`.discuss/evolution-YYYYMMDD-HHMMSS.md`（使用开始时间戳）

---

## 主循环

以下流程**循环执行**，直到触发终止条件。每次循环为一次"迭代"，记录迭代编号。

---

### Phase 1: 代码审查与议题提取

读取当前代码状态，提取本轮讨论议题。

**第一次迭代**：全面阅读代码，识别所有潜在问题域：
- 正确性（逻辑错误、边界条件、错误处理）
- 健壮性（panic、unwrap、异常路径）
- 性能（热路径、不必要的分配、算法复杂度）
- 可维护性（可读性、结构、命名）
- 安全性（如适用）
- 测试覆盖（缺失的测试场景）

**后续迭代**：聚焦上一轮未解决的问题 + 新引入代码的质量。

将本轮议题整理为一句话，作为讨论输入。

---

### Phase 2: 多角色对抗讨论

针对 Phase 1 提取的议题，执行以下讨论流程。

**角色选择**（固定 5 个，每轮根据议题动态调整）：

从以下角色库中选出本轮最相关的 5 个，确保存在真实价值冲突：

| 代号 | 角色         | 核心偏好                           |
|------|--------------|------------------------------------|
| R1   | 风险卫士     | 失败模式、边界条件、防御性编程     |
| R2   | 极简主义者   | 反对一切不必要的复杂性             |
| R3   | 性能狂热者   | 吞吐量 / 延迟 / 资源效率           |
| R4   | 实用主义者   | 实证优先，拒绝过度设计             |
| R5   | 第一性原理者 | 质疑所有假设，回归根本需求         |
| R6   | 维护性倡导者 | 三年后的可读性和可维护性           |
| R7   | 测试驱动者   | 测试覆盖、可测试性、回归安全       |
| R8   | 激进重构者   | 无视当前约束，追求最优设计         |
| R9   | 安全专家     | 安全性、资源泄漏、未定义行为       |
| R10  | 怀疑论者     | 默认质疑当前实现，要求严格举证     |

**讨论轮数**：根据议题复杂度自动判断（2–6 轮），代码迭代场景优先简洁收敛。

**每轮格式**：
```
【角色名 | 立场】
论点：... （指向具体代码位置，如文件名/行号/函数名）
反驳 [角色]：... ← 第 2 轮起必填
立场修正：... ← 最后一轮必填
```

**讨论输出**：
```
## 本轮讨论结论
- 发现的问题：<列表，每项注明严重程度：critical / major / minor>
- 建议的改动：<具体、可操作的改动列表>
- 搁置的争议：<本轮未达成共识的点，留待后续迭代>
```

---

### Phase 3: 代码实现

根据 Phase 2 的结论，实施改动。

**优先级**：`critical` → `major` → `minor`

**实施原则**：
- 每次改动后立即在心里验证逻辑，不要批量改动后才发现冲突
- 若改动涉及接口变更，同步更新所有调用方
- 新增代码遵循项目已有的风格和约定（从现有代码推断）

---

### Phase 4: 编译 & 测试验证

改动完成后，自动检测项目类型并执行验证：

```
Rust:   cargo build 2>&1 && cargo test 2>&1
C++:    xmake build 2>&1 && xmake test 2>&1  （或 cmake --build . && ctest）
Python: uv run python -m pytest 2>&1         （或 python -m pytest）
Node:   npm run build 2>&1 && npm test 2>&1
Go:     go build ./... 2>&1 && go test ./... 2>&1
```

**验证结果处理**：
- ✅ 全部通过 → 进入 Phase 5
- ❌ 编译失败 → 立即修复，不进入下一 phase，修复后重新验证
- ❌ 测试失败 → 分析失败原因：
  - 若是实现问题 → 修复代码，重新验证
  - 若是测试本身需要更新（接口变更）→ 同步更新测试，重新验证
  - 若无法在当前迭代内修复 → 回滚本轮改动，记录原因，进入 Phase 5

---

### Phase 5: 迭代评估 & 终止判断

**记录本轮摘要**（写入日志文件）：
```markdown
### 迭代 N — <时间戳>
- 议题：<一句话>
- 参与角色：<列表>
- 主要改动：<改动摘要>
- 验证结果：✅ 通过 / ❌ 部分回滚
- 遗留问题：<未解决的点>
```

**终止条件检查**（按优先级，任一满足即终止）：

1. **已达最大迭代次数** → 终止，输出"已达最大迭代次数"
2. **编译或测试无法通过且无法修复** → 终止，输出当前状态和建议
3. **Claude 判断无更多有意义的改进点**：
   - 条件：上一轮讨论中所有角色在最后一轮均无新的 critical/major 问题
   - 且：遗留问题仅为 minor 且属于主观偏好分歧
   - → 终止，输出"代码已达到当前认知下的完善状态"
4. **用户介入**：每次迭代结束后询问用户：
   ```
   ✅ 迭代 N 完成。是否继续下一轮？[继续 / 停止 / 跳过测试继续]
   ```
   → 用户回复"停止"或按 Esc → 终止

---

## 终止输出

```markdown
## Self-Evolution 完成报告

### 执行概况
- 开始时间：...
- 总耗时：...
- 执行迭代：N 轮（最大上限：M 轮）
- 终止原因：...

### 代码改进总结
- 修复的问题：<按 critical/major/minor 分类列出>
- 新增/改动的文件：<列表>
- 测试状态：✅ 全部通过 / ⚠️ 部分通过

### 遗留事项
- <未解决的问题或建议的后续工作>

### 各轮迭代摘要
- 迭代 1：...
- 迭代 2：...
- ...
```

将完整报告和所有迭代日志写入 `.discuss/evolution-YYYYMMDD-HHMMSS.md`，并告知用户：
`✓ 进化日志已保存至 .discuss/evolution-YYYYMMDD-HHMMSS.md`
