## 核心理念

`/design` 是"告诉我做什么，我来做"。`/evolve` 是"让我看看这个项目还缺什么，我们一起决定怎么长"。

项目早期最大的问题不是实现质量，而是**不知道还缺什么**。需求藏在代码的空白处——没有错误处理的路径、没有暴露的能力、没有覆盖的用例、硬编码的假设、缺失的配置化。

**演进哲学**：发现缺口后，按设计最优原则补齐，而非用最小代价凑合。新增功能应以"如果从头设计这个模块会怎么写"为标准，而非"在现有代码上加最少的行"。

`/evolve` 的工作方式：
1. **深度理解**——读懂项目是什么、做什么、为谁做
2. **缺口发现**——从多个维度系统化地发现"还没做但应该做的事"
3. **优先排序**——对发现的缺口做多角色对抗讨论，收敛出优先级
4. **逐个推进**——按优先级逐一讨论方案 → 实现 → 验证

---

## 参数解析

- `[target: <module or domain>]`：聚焦特定模块或功能域；未指定则分析整个项目
- `[rounds: N]`：进入实现模式时，最多推进 N 个功能点；未指定则持续推进直到用户停止
- `[implement]`：进入实现阶段（Phase 3）。**默认仅执行 Phase 1–2（发现 + 排序）**，输出优先级清单后停止。用户确认后可追加 `implement` 或逐个用 `/design` 实现
- `[auto]`：无人值守模式——跳过所有交互确认，自动推进

## Phase 1: 项目理解

在提出任何建议之前，先**完整理解项目现状**。

### 1.1 代码全面阅读

阅读所有核心源文件（非测试、非生成），建立以下理解：

- **项目定位**：这是什么？解决什么问题？目标用户是谁？
- **核心架构**：模块划分、数据流、关键抽象
- **公开接口**：对外暴露了哪些能力（API、CLI、库接口）
- **内部机制**：核心算法、状态管理、并发模型
- **依赖关系**：外部依赖及其用途
- **配置与扩展点**：哪些行为可配置，哪些是硬编码

### 1.2 文档与测试阅读

- 阅读 README、设计文档（如有），理解作者的意图和规划
- 阅读测试代码，了解已覆盖的场景和隐含的需求假设

### 1.3 输出项目画像

```
## 项目画像

**定位**：<一句话>
**核心能力**：
  - <能力 1>
  - <能力 2>
  - ...
**架构概要**：<模块关系简述>
**当前成熟度**：<初期 / 功能基本可用 / 接近生产就绪>
**已有测试覆盖**：<概述>
```

**检查点**：向用户确认画像是否准确，是否有遗漏的背景信息或隐含需求。

**`auto` 模式**：跳过确认，直接继续。

## Phase 2: 缺口发现与排序

### 2.1 多维度缺口扫描

从以下维度系统化扫描缺口：

**功能完备性**：
- 公开接口承诺了但未实现的能力（`todo!()`, `unimplemented!()`, 空函数体）
- 同类项目通常具备但本项目缺失的功能（基于项目定位推断，不盲目对标）
- 硬编码的值/行为，应该配置化或参数化的
- 只支持了 happy path，缺少对异常输入/边界情况的处理

**API 与接口**：
- 接口设计不完整：缺少对称操作（有 add 没 remove，有 get 没 list）
- 缺少批量操作、异步变体、或流式接口（如果场景需要）
- 错误类型过于粗糙，调用方无法区分不同失败原因

**健壮性**：
- 错误处理缺失或不完整的路径
- 缺少输入校验（特别是公开接口）
- 缺少超时、重试、熔断（对外部 I/O）
- 缺少优雅降级（部分失败时的行为）

**可观测性**：
- 关键操作缺少日志
- 错误日志缺少上下文
- 缺少 metrics / tracing 支撑点（如果项目规模需要）

**可用性**：
- 缺少使用示例或示例代码
- CLI 缺少 help 文本或参数校验
- 错误信息对用户不友好（内部错误直接暴露）

**测试缺口**：
- 公开接口未被测试覆盖
- 错误路径未被测试
- 并发/竞态场景未被测试（如果有并发）

**代码质量**（信号检测，不深入分析——深度改进由 `/improve` 执行）：
- 超长函数（>100 行）或深嵌套（>4 层）
- 明显的重复代码块
- 过时或失效的 TODO/FIXME/HACK 注释
- 模块职责不清晰、耦合过紧

**性能**（信号检测——深度优化由 `/bench optimize` 执行）：
- 热路径上的 O(n²) 或更差复杂度
- 循环内频繁分配（clone、to_string、临时容器）
- 公开接口缺少 benchmark 覆盖
- 已知的 I/O 瓶颈（同步阻塞、无缓冲）

**文档完整性**（信号检测——文档生成由 `/doc` 执行）：
- README 与当前代码不符或过于简略
- 公开 API 缺少文档注释
- CHANGELOG 未更新（有新功能提交但无对应条目）
- 缺少使用示例或快速入门指南

### 2.2 缺口清单

```
## 缺口清单

| 序号 | 维度 | 缺口描述 | 影响 | 复杂度 | 推荐 Skill |
|------|------|----------|------|--------|------------|
| 1 | 功能完备性 | 缺少 X 功能 | 用户无法完成 Y 场景 | 中 | evolve 自行实现 |
| 2 | 健壮性 | Z 接口无错误处理 | 异常输入导致 panic | 低 | evolve 自行实现 |
| 3 | 代码质量 | 模块 A 职责混乱 | 可维护性差 | 中 | /improve |
| 4 | 性能 | 解析器缺少 benchmark | 性能退化不可感知 | 低 | /bench |
| 5 | 文档 | README 过时 | 新用户无法上手 | 低 | /doc |
| ... | ... | ... | ... | ... | ... |
```

**推荐 Skill 规则**：
- 功能完备性、API 与接口、健壮性、可观测性、可用性 → `evolve 自行实现`（现有 Phase 3 逻辑）
- 测试缺口 → `/test`
- 代码质量 → `/improve`（轻度）或 `/refactor`（架构级）
- 性能 → `/bench optimize`（有瓶颈需优化时）或 `/bench`（缺 benchmark 覆盖时）
- 文档完整性 → `/doc`

选出 **5 个角色**，对缺口清单进行优先级对抗讨论（3–5 轮）。

**讨论输出**：

```

### Tier 2 — 应该做（显著提升项目价值）
3. <缺口> — 原因：...

### Tier 3 — 可以做（锦上添花）

**`auto` 模式**：跳过确认，按排序结果直接推进。

**默认行为**：到此结束，输出报告。提示用户可追加 `implement` 进入 Phase 3，或逐个用 `/design` 实现。

## Phase 3: 迭代推进

按优先级逐个推进。根据缺口的"推荐 Skill"列决定推进方式：

- **`evolve 自行实现`**：走下方 3.1–3.5 的完整流程（需求细化→方案设计→实现→验证→提交）
- **委派给其他 skill**：直接调用对应 skill 并传入缺口描述作为参数，由该 skill 按自身流程完成。调用时自动追加 `auto` 参数。例如：
  - 代码质量缺口 → 执行 `/improve target: <相关模块> auto iter: 1`
  - 测试缺口 → 执行 `/test <缺口描述>`
  - 性能缺口 → 执行 `/bench optimize <目标> auto max-rounds: 3`
  - 文档缺口 → 执行 `/doc <缺口描述>`
  - 架构缺口 → 执行 `/refactor <意图> auto`

委派的 skill 完成后，evolve 记录结果并继续下一个缺口。

以下 3.1–3.5 仅适用于 `evolve 自行实现` 的缺口。

### 3.1 需求细化

选出 **3 个角色**（从 2.3 的角色库中选取，根据缺口性质动态调整；若需要可自定义），进行 2–3 轮对抗讨论，明确：

- **具体要做什么**：输入、输出、行为规范
- **边界在哪**：这次做什么、不做什么
- **验收标准**：怎样算完成

**每轮发言格式**同 Phase 2.3。

输出一段简明的需求描述（3–8 句话），作为后续设计和实现的契约。

### 3.2 方案设计

选出 **5 个角色**（从 2.3 的角色库中选取，根据缺口性质动态调整；若需要可自定义），进行 2–4 轮对抗讨论：

- 提出 2–3 个可行方案
- 从正确性、复杂度、一致性（与现有代码风格的融合）、扩展性角度对抗
- 收敛为一个具体方案，包含：
  - 要新增/修改的文件和函数
  - 接口签名
  - 关键实现思路

**每轮发言格式**同 Phase 2.3。

**检查点**：向用户展示方案，确认后再实现。

**`auto` 模式**：跳过确认，直接实现。

### 3.3 实现

按方案实现代码，遵循以下原则：

- 与项目现有风格保持一致（命名、结构、错误处理模式）
- 每个改动后立即编译验证
- 同步编写测试（至少覆盖 happy path + 主要 error path）
- 非显然逻辑附注释

### 3.4 验证

```
若用户提供了构建/测试/benchmark 命令则优先使用；否则根据项目构建系统和配置，自行确定并执行构建、测试与 benchmark 命令；若项目无测试或 benchmark 则跳过对应步骤。
```

- ✅ 通过 → 进入 3.5
- ❌ 失败 → 修复后重新验证

### 3.5 提交

使用 Conventional Commits 格式提交：

```
feat(<scope>): <简述>

<详细描述，包含为什么需要这个功能>
```

### 3.6 轮次报告与继续

```

2. 已达用户指定的 `rounds: N`
3. 用户中断
4. 编译/测试无法恢复

## 异常流程处理

### 缺口之间有依赖

若发现后续缺口依赖前面尚未完成的缺口，调整执行顺序，将依赖项前置。

### 实现过程中发现新缺口

记录到缺口清单末尾，按当前 Tier 分类，不打断正在进行的实现。

### 方案争议无法收敛

若对抗讨论 4 轮后仍无法收敛，暂停当前缺口，标记为"需要用户决策"，跳到下一个。

**`auto` 模式**：选择最保守（复杂度最低）的方案，在报告中标注"自动选择，建议复核"。
