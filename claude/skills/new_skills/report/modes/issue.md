## Mode：issue（问题报告 / bug tracker 条目）

**适用**：发现了一个问题 / bug / 异常，要写成可追踪条目让协作者（或未来的自己）接手修。

**与 incident 的区别**：
- `issue` 是**前置**：问题刚被发现，等处理
- `incident` 是**后置**：事故已经发生过，做 post-mortem

**与 fix 目的的区别**：
- `/blueprint --fix` 是**规划 + 实施** 修复
- `/report issue` 是**把问题写成可交接的条目**，不修

---

### 读者画像

- **负责修这个问题的人**（可能是协作者，也可能是未来的自己）：需要足够信息独立开始调查
- **triage 的人**：决定这个 issue 的优先级 / 归类 / 派给谁

**共同特征**：
- **没有当前会话的上下文** —— issue 必须是自包含的
- 时间有限 —— 读完 1 分钟就要能判断是否接手 / 优先级

### 必含章节

```
## 概述
<1-2 句：问题本质>

## 现象
<精确、可观察的描述：
  - 观察到什么
  - 在哪里（模块 / 环境 / 用户场景）
  - 何时开始出现（若知道）>

## 复现步骤
<可执行的最小复现
  1. 做 X
  2. 做 Y
  3. 观察到 Z（期望是 W）>
<若无法复现 —— 明确写"无法稳定复现，触发条件未明"，并列出已尝试但失败的复现>

## 期望行为
<本来应该发生什么 + 依据（文档 / 合约 / 常识）>

## 实际行为
<真实发生了什么 + 相关信息（错误消息 / 日志片段 / 截图引用）>

## 影响范围
- 谁受影响（用户 / 开发 / CI / ...）
- 多严重（阻塞 / 降级 / 次要）
- 是否有 workaround

## 环境信息
- 版本 / commit: <精确到 hash>
- 平台 / OS
- 相关依赖版本
- 复现环境的任何特殊配置

## 已调查 / 已尝试
<作者已经试过什么：
  - grep 过哪些代码（带结果或"空"）
  - 查过哪些文档
  - 尝试的 workaround 和结果
  - 怀疑的根因方向>
<目的：让接手人不重复你做过的事>

## 建议调查方向
<作者的假设：可能是什么原因、从哪入手
 标明"假设"而非"结论" —— 不要误导接手人>

## 建议处理目的
<若这 issue 需要转成 blueprint 处理，推荐走哪个目的：
  - fix（如果是 bug）
  - reshape（如果是结构问题）
  - upgrade（如果是依赖问题）
  - review（如果需要更多核查）>

## 相关
- Commits: <怀疑相关的 commit>
- Issues / PRs: <相关的其他 issue>
- Docs: <相关文档引用>
- Logs: <日志路径 / 截图文件引用（放 .artifacts/ 子目录）>
```

### 素材收集侧重

- **用户输入**：issue 大多需要用户提供核心信息 —— /report 不能凭空知道"发生了什么 bug"
- **git log / blame**：对怀疑相关的文件，看最近改动
- **grep**：搜索相关符号 / 错误消息
- **.artifacts/**：若有相关 review 报告提到过此问题

### 语气 / 深度

- **精确** > 完整：宁可少写信息也不写不确定的
- **事实 vs 假设分明**：用标签明确标出"观察" / "假设" / "已验证"
- **谦虚**：承认"我不确定为什么" —— 好于编造原因误导调查
- **自包含**：读完这个 issue 不需要问作者就能开始调查
- **可操作**：复现步骤是真能跑的命令，不是"大概做一下类似的"

### ASCII 图示要求

按场景选用：
- 若涉及多个组件 / 服务 → **组件关系图**（标出出问题的部分）
- 若是时序 / 并发问题 → **时序图**
- 若涉及状态 → **状态转移图**（标出卡住或异常转移的状态）

单点简单 bug 可以不画图，但复现步骤必须清晰。

### 禁忌

- ❌ "不工作" / "坏了" / "有 bug" —— 必须描述**如何**不工作
- ❌ 没有复现步骤
- ❌ 把假设写成事实（"这显然是 X 引起的" —— 除非已验证，否则标为"假设"）
- ❌ "我觉得跟最近的 X 改动有关" —— 去查 git blame / 读代码确认后再写
- ❌ "重启就好了" 作为解决方案 —— 那是 workaround，不是修复
- ❌ 没有环境信息（版本 / OS / 平台） —— 复现 90% 靠环境
- ❌ 把"帮我修一下"当作 issue 描述
- ❌ 敏感数据直接粘贴（用户 email / 真实姓名 / 内部 token） —— redact 处理
- ❌ 一个 issue 混多个问题 —— 拆成多个独立 issue
