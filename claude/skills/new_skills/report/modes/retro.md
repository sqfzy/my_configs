## Mode：retro（反思复盘 / lessons learned）

**适用**：某段工作告一段落（功能完成、修复完成、项目 milestone、sprint 结束、事故处理后），写一份反思：做对了什么、做错了什么、下次怎么改。

**核心价值**：组织学习。**不是**成果汇报（那是 status）、**不是**决策记录（那是 decision）、**不是**事故根因追查（那是 incident）——而是"从这段经历中提炼可传承的教训"。

---

### 读者画像

- **自己**：未来遇到类似情况时的自我提醒
- **团队**：组织学习，避免重复踩坑
- **新人**：了解项目 / 团队走过的弯路

**共同特征**：
- 期望**真诚**的反思 —— 一眼看穿粉饰
- 关心**可迁移的教训** —— 不只是本次的事实
- 愿意读**情绪和判断**（不只是冷冰冰数据）

### 必含章节

```
## TL;DR
<2-3 句：这段工作做了什么 + 最重要的 1-2 个教训>

## 背景
- 这段工作的时间范围
- 目标是什么
- 上下文约束（时间 / 资源 / 技能 / 外部压力）

## 做了什么（简述）
<不是 status report 的详细清单
 只是让读者知道"什么工作触发了这次复盘"
 3-5 句话概述>

## 做对了（What Went Well）
<每条：
  - 发生了什么（具体）
  - 为什么这是"做对"（带来了什么）
  - 怎么持续保持 / 扩大到其他场景>

## 做错了（What Went Wrong）
<每条：
  - 发生了什么（具体、诚实）
  - 代价是什么（时间 / 质量 / 团队关系 / ...）
  - 为什么会这样（根本原因，不是表象）
  - 下次怎么避免（具体、可操作）>

## 意外收获 / 惊喜
<预期之外出现的积极发现
 例：某个工具比想象中好用 / 某个同事展现了意外的能力 / 某个约束反而促进了简化>

## 教训（Key Takeaways）
<从本次经历提炼出的**可迁移**原则
 不是"这次 A 做错了"
 而是"下次遇到 X 类情况，应该 Y"
 每条：
  - 原则
  - 适用场景
  - 对立面（什么情况下这个原则不适用）>

## 具体 Action Items（给未来的自己 / 团队）
<每条：
  - 具体动作
  - 触发条件（什么情况开始做这件事）
  - 负责人（若涉及团队）>
<和 incident 的 action item 不同 —— retro 的 action 更宽泛，可以是流程 / 心态 / 工具>

## 系统性问题（若有）
<若反思中识别出的问题不是个人或本项目独有，而是团队 / 组织 / 流程层面的
 明确列出，建议升级为组织级讨论>

## 数据 / 指标（可选）
<若能量化本次工作，附上：
  - 计划 vs 实际时间
  - 代码量 / 测试数 / bug 数
  - 性能指标
  - 等>

## 相关
- Commits: <相关 commit 范围>
- .artifacts/: <本次期间生成的其他 report（decision / status / incident）>
```

### 素材收集侧重

- **整个会话历史**：讨论、决策、踩坑、灵光一现
- **git log**：commit message 本身就是"当时在想什么"的证据
- **.artifacts/INDEX.md**：本期间有哪些 report，反映了什么思考轨迹
- **用户补充**：情绪 / 主观判断 / 当时的假设 —— 这些是 retro 的核心，AI 无法完全自动获取

### 语气 / 深度

- **真诚 > 正确**：宁可表达不完美的反思也不粉饰
- **具体 > 抽象**：不是"沟通可以更好"，而是"周三 X 事上我们没达成共识，原因是 Y"
- **无指责**：讨论"系统" / "流程" / "决策"，不点名个人
- **情绪可见**：可以写"当时很挫败" / "这个解法让我惊喜" —— 比冷冰冰的事实更能传达教训
- **反思自己**：作者的角色和贡献也要审视，不只是评价外界
- **避免灾难化**：犯了错不等于一切都是错；避免过度自责
- **避免过度乐观**：做对了不等于下次一定继续做对

### ASCII 图示要求

按需使用：
- **时间线**：本次工作的关键转折点 + 情绪 / 认知的变化
- **因果图**：把某个教训追溯到哪些决策 / 事件的组合结果
- **Before/After 心智图**：开始前以为要做什么 vs 实际做了什么

### 禁忌

- ❌ 变成 status report（只列做了什么，不反思）
- ❌ 只列"做对了"不列"做错了"（假作团结）
- ❌ 只列"做错了"不列"做对了"（自我鞭挞不健康）
- ❌ 点名追责（"是 Alice 的错"）
- ❌ 教训太抽象（"要更早沟通" —— 具体何时？对谁？）
- ❌ Action items 泛化（"多思考" —— 这不是 action）
- ❌ 粉饰 —— "总体来说很成功"掩盖真实问题
- ❌ 只看表面现象 —— "因为时间不够" 通常只是表象，深挖"为什么时间不够 / 为什么估错了"
- ❌ 不让教训可迁移（只是这一次的故事，不能给未来类似场景提供帮助）
