## Mode：release（发布说明 / release notes）

**适用**：版本发布后，写给**终端用户 / 库的调用方**看的"本版本有什么变化"。

**区分**：
- release notes **对外**（用户）：按用户视角组织
- CHANGELOG **对内 + 偏技术**：保留完整的技术变更历史
- 一般项目两者并存，release notes 可以从 CHANGELOG 提炼 + 用户视角改写

---

### 读者画像

- **终端用户 / 库的调用方**：最关心"我能用什么新功能 / 我的代码要不要改"
- **技术决策者**（升级评估者）：关心"升级的代价 / 获得的好处"

**共同特征**：
- **不读项目内部细节**：不关心"重构了 X 模块"除非这影响他们
- **关心影响**：能做什么、不能做什么、怎么升级
- **痛恨"改进了性能"式空话**：要具体数字

### 必含章节

```
## <项目名> vX.Y.Z - YYYY-MM-DD

<版本号 + 日期 + 版本类型（stable / beta / RC）>

## TL;DR
<2-3 句：本版本最重要的变化 + 是否含 breaking + 推荐动作>

## 新能力（Added / Features）
<用户能做什么新事情
 每条：
  - 一句话描述
  - 最小使用示例（若适用）
  - 指向详细文档的链接>

## 改进（Changed）
<已有能力的优化 —— 不破坏兼容
 每条：
  - 一句话描述 + 量化（快了多少 / 内存省了多少 / 容量扩了多少）
  - 是否需要用户做什么（通常不需要）>

## 修复（Fixed）
<修复了哪些 bug，给用户的视角
 不是 "修了 race condition in parser"
 而是 "修复在并发访问时可能产生错误结果的问题">

## 破坏性变更（⚠️ Breaking Changes，若有）
<必须显著标注
 每条：
  - 变更说明
  - 受影响的 API / 配置 / 行为
  - 迁移指引（具体命令 / 代码示例）
  - 升级检查清单>
<如果没有 breaking，明确写"本版本无 breaking changes，可安全升级">

## 废弃（Deprecated，若有）
<还能用但未来会移除的 API
 每条：
  - 废弃的是什么
  - 替代方案
  - 计划在哪个版本移除>

## 移除（Removed，若有）
<彻底移除的 API
 每条：
  - 移除的是什么
  - 从哪个版本开始废弃的（给读者心理准备）
  - 替代方案>

## 安全（Security，若有）
<修复的安全问题
 按 CVE 标准描述：编号 / 影响 / 严重度 / 修复说明>

## 升级指引
<从上一版本升到本版本的步骤
 特别是 breaking 存在时必须有详细步骤
 包括：
  - 升级前准备
  - 具体升级命令
  - 升级后验证
  - 回滚方法>

## 已知问题（若有）
<本版本已知但尚未解决的问题
 每条：描述 + workaround + 预计哪个版本修复>

## 致谢
<贡献者列表（可选，根据项目文化）>

## 相关
- GitHub release: <链接>
- Full changelog: <链接到 CHANGELOG.md>
- Documentation: <链接>
```

### 素材收集侧重

- **git log**：上次 release 到本次的所有 commit（按 Conventional Commits 的 type 分组）
- **CHANGELOG.md**：如果项目有，提炼其中本版本条目
- **breaking change 扫描**：grep 所有带 `BREAKING CHANGE:` 的 commit footer
- **用户输入**：哪些是"面向用户的亮点"（通常不是最大 commit）
- **.artifacts/**：若有 ship report，提取其中 Gate 结果

**版本号**：从 git tag / package 配置 / 用户输入获取

### 语气 / 深度

- **用户视角**：而不是工程师视角
- **具体**：有示例、有数字、有命令
- **对称**：按 Keep a Changelog 分类（Added / Changed / Deprecated / Removed / Fixed / Security）
- **避免内部术语**：用户不知道"Validator trait" / "mod_a" 是什么
- **避免"重构了 X"作为亮点**：重构是实现细节，除非用户能感知
- **不卖弄**：用户关心能力，不关心你做了多少工作

### ASCII 图示要求

按需使用：
- 若有**显著架构变化**影响用户 → 前后对比图
- 若有**性能提升** → 对比柱状图（老 vs 新）
- 若有**多个破坏性变更** → 迁移决策树

简单版本可以完全无图。

### 禁忌

- ❌ "改进了性能" —— 必须有数字
- ❌ "修复了一些 bug" —— 列出具体有哪些（对用户可见的）
- ❌ "重构了 X 模块" —— 除非用户能感知，否则不是 release 亮点
- ❌ Breaking changes 藏在中间 —— 必须显著在第一屏提示
- ❌ 没有迁移指引的 breaking
- ❌ 版本号写错 / 日期写错
- ❌ 用工程术语打击用户（"Mutex contention 降低" → "并发吞吐提升 40%"）
- ❌ 抄 CHANGELOG 不做提炼（CHANGELOG 可能有几十条，release notes 应该浓缩到用户能读完的长度）
- ❌ 没有"是否可以安全升级"的结论（用户最关心这个）
- ❌ 暴露内部路径 / 账号 / 密钥 / 未公开 API
