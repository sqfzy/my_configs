## 目的：fix（修复已知问题）

**适用**：已经观察到 bug、错误、崩溃、测试失败、行为异常，要制定修复方案。

**不适用**：bug 原因不明需要先调查 —— 这种情况先用独立的调试会话定位根因（或在 Phase 2 的"根因分析"里显式展开调研），拿到根因后再进入 Phase 2 后续。

本文件只定义 fix 目的**独有**的约束，通用维度由 SKILL.md 规定。

---

### fix 独有的必展开章节

以下章节**必须**在计划中显式展开：

#### 1. 现象描述

必须精确：
- 触发条件（什么输入、什么环境、什么时序）
- 复现步骤（最小复现用例，优先用代码片段 / 命令）
- 影响范围（哪些用户 / 数据 / 下游功能被影响）
- 首次发现时间 / 被报告的版本（若已知）

**陷阱**：模糊的"有时候会出错"是没法修的。计划前必须把现象约束到可复现。

#### 2. 根因分析

必须回答三个问题：
- **哪一层的问题**（逻辑 / 边界 / 并发 / 资源 / 假设错误 / 外部依赖）
- **为什么现在才出现**（新上线的改动引入？一直存在但未触发？环境变化？）
- **是否还有同类问题**（相似代码模式在别处是否也有？需要一起修吗？）

若根因尚未完全确认，**必须在 Phase 2 内先做定位调研**（阅读代码、grep 同类模式、加临时日志验证假设），把根因锁定后再继续方案设计。不允许"根因猜一猜就写修复方案"。

**陷阱**：只定位到"这一处出问题"是不够的。必须问"同类模式别处有吗"——fix 常常一次只修露出水面的一个点，同类隐藏问题下次又爆。

#### 3. 修复方案

必须给出 **2 种**候选的对比（哪怕最终选哪个已经明确）：
- **最小改动**：改到不报错即可，范围最小
- **根本修复**：从设计层面消除此类问题的可能性

对比后明确选哪个 + 理由。若选"最小改动"，必须在计划中：
- 显式写出技术债条目
- 标注后续补齐的触发条件（下次迭代 / 遇到同类问题 / 某个 Tier 任务）

**陷阱**：永远选最小改动会累积结构性问题；永远选根本修复会阻塞紧急发布。**必须在计划里给出选择依据**。

#### 4. 回归防护

**强制要求**：fix 的计划必须包含回归测试。不带回归测试的 fix 不算 fix。

必须给出：
- 哪些测试用例需要新增（覆盖最小复现用例 + 至少 1 个同类边界场景）
- 测试应该放在哪个测试文件 / 层级（单元 / 集成 / e2e）
- 若能加 assertion / invariant 让同类问题"崩得早、崩得响"，在计划里列出（如 `debug_assert!` / 不变量检查）

#### 5. 影响评估

必须核对：
- 修复改动的波及面（调用方、公共 API、外部接口、持久化格式）
- 是否引入**任何行为变化**（哪怕是错误信息字符串改了）——若有，列入计划的"行为变化清单"
- 是否可能影响现有测试（哪些测试要跟着改、哪些应该保持不动）

**陷阱**：fix 引入的悄悄行为变化是最危险的——下游依赖方 / 客户 / 监控告警 / 文档可能全部基于旧行为。

---

### fix 必含 ASCII 图示（按场景选用）

- 若涉及跨模块 / 多文件 → **影响范围图**（谁调用谁、哪些受影响）
- 若涉及行为变化 → **修复前后行为对比**
- 若是并发 / 时序问题 → **时序图**（bug 触发路径 + 修复后路径）
- 若根因链条长 → **根因追踪图**（从现象到根因的因果链）

单点逻辑 bug 可以不画图，但修复方案和回归测试必须文字讲清。

---

### fix 实施计划的特殊约束（施工契约）

**阶段划分原则（强制顺序）**：

```
阶段 1: 定位验证（若根因已 100% 确认可跳，否则强制）
  - 具体动作：grep 同类模式；阅读关键代码；加临时日志或断言；复现用例
  - 交付物：根因报告（在 plan 的"根因分析"章节补全）
  - 验收：能稳定复现 bug + 根因定位到具体代码行

阶段 2: 先写回归测试（应当失败）
  - 具体动作：按"回归防护"章节在对应测试文件写测试用例
  - 交付物：新增测试文件 / 测试函数
  - 验收：新测试运行**失败**（证明它能捕获 bug）
  - 施工纪律：这一步的 commit message 明确"add failing regression test for #XXX"

阶段 3: 修复实施
  - 具体动作：按选定方案（最小 / 根本）改代码
  - 交付物：修复补丁
  - 验收：阶段 2 的新测试**通过**；同时所有现有测试仍然通过

阶段 4: 同类问题扫描 + 加固（若根因分析已识别同类）
  - 具体动作：按"是否还有同类问题"章节修复同类；添加 assertion / invariant
  - 交付物：同类修复 + 加固代码
  - 验收：全量测试通过；新加 assertion 在非 bug 场景不会误报

阶段 5: 全量回归验证
  - 具体动作：运行全部测试 + lint；若有 bench，对关键路径跑一遍
  - 验收：全量通过；bench 无显著退化
```

**阶段 2 和阶段 3 禁止合并**——先改代码再加测试容易忘或测不到。先写失败测试再让它通过是更严谨的顺序。

若修复跨多个模块，**阶段 3 应进一步拆分**，每个模块独立成一个阶段，每阶段独立可验证。

### fix 的生产级重点（施工时必须贯穿）

8 维度里 fix 要特别强化：

- **正确性与边界**：本就是 fix 的核心——边界条件必须覆盖
- **可观测性**：修复时机是加日志 / 指标的好时机，让下次同类问题能快速定位（在阶段 3 或阶段 4 加入）
- **测试策略**：必须有回归测试，无例外
- **不引入新问题**：每阶段末必须运行全量测试，发现任何新失败立即停下

---

### fix 的施工纪律（写入计划的"施工纪律"字段）

- **阶段 2 的 commit 独立**，message 注明"regression test for ..."
- **阶段 3 的 commit 独立**，message 注明"fix: ..."（Conventional Commits）
- **阶段 4（若有）的 commit 独立**，message 注明"harden: ..."或"fix: (同类修复)..."
- 不允许"顺手改一下"——发现任何无关改动停下，移到独立 fix / reshape plan
- 不允许悄悄改行为——任何行为变化必须在计划中已声明

---

### fix 的反模式（计划里出现即不合格）

- ❌ 没有根因分析，直接跳到修复方案
- ❌ 没有回归测试条目
- ❌ 没有"是否有同类问题"的审视
- ❌ 没有行为变化审视（静悄悄改行为是最危险的 fix）
- ❌ 没有影响范围评估
- ❌ 阶段 2 和阶段 3 合并（先写失败测试的纪律必须保持）
- ❌ 把无关改动夹带进修复

---

### fix 的风险与回滚

- 单阶段失败时的回滚粒度：回到上阶段末的 commit
- 预警信号：
  - 阶段 3 修复后新测试仍然失败 → 方案有误，停下回 `/blueprint --fix` 重估根因
  - 阶段 3 修复后老测试开始失败 → 修复引入回归，立即回滚重评
  - 阶段 4 发现的同类问题数量远超预期 → 停下，考虑转 `/blueprint --reshape`（结构性问题）
