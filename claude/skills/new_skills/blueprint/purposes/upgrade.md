## 目的：upgrade（外部依赖 / 版本 / 框架迁移）

**适用**：由**外部变化驱动**的变更——依赖主版本升级、语言版本 / edition 升级、框架替换、构建系统切换、平台迁移。

**与 reshape 的关键区别**：reshape 是**内部主动**改造结构；upgrade 是**被外部变化逼着**改。判据：如果今天外部依赖不变，还会做这事吗？不会——就是 upgrade。

本文件只定义 upgrade 目的**独有**的约束，通用维度由 SKILL.md 规定。

---

### upgrade 独有的必展开章节

以下章节**必须**在计划中显式展开：

#### 1. 升级目标

必须精确：
- **从什么到什么**：具体的版本号 / 框架名 / 配置（如 `sqlx 0.7.x → 0.8.0`、`Rust 2021 → 2024 edition`、`webpack 4 → Vite 5`）
- **为什么升**（必须给出具体驱动因素）：
  - 安全：CVE 编号
  - 新特性：需要的特性名 + 当前替代方案的痛点
  - 生态跟进：关键依赖已不再支持旧版
  - EOL：旧版停止维护的时间点
  - 性能：预期收益 + 数据支持

**陷阱**：没有具体驱动因素的"想升级一下试试"不是 upgrade——是 reshape 或 feat。

#### 2. Breaking Changes 清单（核心章节）

必须系统性列出：

**从官方文档 / changelog 提取**所有 breaking changes，然后 grep 项目代码，每一项标注：
- 变更内容（一句话）
- 本项目是否受影响（是 / 否 / 部分）
- 受影响的具体位置（文件 / 符号，grep 结果）
- 迁移方案

```
| # | Breaking Change | 受影响？ | 位置 | 迁移方案 |
|---|-----------------|---------|------|---------|
| 1 | API X 重命名为 Y | 是 | src/a.rs:42, src/b.rs:89 | sed 替换 |
| 2 | 错误类型改动 | 是 | 整个 error 模块 | 适配层 |
| 3 | 移除 deprecated 方法 Z | 否 | - | - |
```

**陷阱**：跳过系统清单、凭"改着看"，会在升级后期遇到一串意外爆炸。

#### 3. 废弃 API 使用情况

即使不是 breaking，若目标版本标注了某些 API 将在下一个主版本废弃，也要清单化：
- 调用点
- 目标替代 API
- 是否这一轮一起改掉

**原则**：upgrade 窗口是一次性处理废弃 API 的最佳时机，不要"升完再说"。

#### 4. 兼容层策略

必须明确是否需要兼容层（adapter / feature flag / 双轨代码）：

| 策略 | 何时用 | 代价 |
|------|-------|------|
| 无兼容层，直接切 | 项目规模小、调用方少、可一次性原子切换 | 无 |
| Adapter 临时桥接 | 调用方多，不能一次性全改 | 需后续清理 |
| Feature flag 灰度 | 生产风险高，需要回滚能力 | 需双轨代码 + 清理周期 |
| 新旧双轨并存 | 高可用场景、需要运行时对比验证 | 最复杂、最长周期 |

选定后必须给出：
- 兼容层存在周期（多久后清理）
- 清理阶段的触发条件

**陷阱**：兼容层极易成为永久代码。必须在计划里明确清理时间点和触发条件，写进实施计划的后续阶段。

#### 5. 升级路径

必须从三种中选一种并说明理由：

- **big-bang**：一次性切换。适用小项目 / 风险可控 / 能原子完成。
- **incremental**：模块逐个升级。适用大项目 / 依赖图复杂 / 不同模块可独立升级。
- **渐进共存**：新旧双轨跑一段后切换。适用高可用 / 需要对比验证 / 有业务连续性要求。

#### 6. 基线与验证策略

**强制要求**：升级前必须捕获基线，升级后必须对照。

升级前基线：
- 构建基线：版本号 / 构建时间 / 产物大小
- 测试基线：全部测试通过 + 测试数量 + 耗时
- Benchmark 基线（若有）：关键 benchmark 数值
- 行为基线（若可捕获）：关键端点的样本请求 / 响应

升级后验收：
- 构建：通过 + 产物大小变化 < 容忍阈值
- 测试：全部通过 + 数量一致（除非 breaking 涉及）+ 耗时无显著退化
- Bench：无退化（或已解释的可接受退化）
- 行为：样本请求响应一致（或已解释的可接受变化）

#### 7. 回滚策略（强制）

upgrade 风险高，**必须**有回滚计划：
- 每个里程碑的**回滚点**（具体是哪个 commit / branch）
- 回滚后能回到的状态
- 回滚的触发条件（什么情况就回滚，不是"看心情"）
- 兼容层时代的回滚特别要小心（新旧代码交织时如何干净回退）

### upgrade 必含 ASCII 图示（在通用要求之上追加）

**强制必含**：
- **受影响点全景**（文件 / 符号 / 调用链的分布图，按模块 / 按 breaking 类别）
- **迁移路径时序**（阶段 1 → 阶段 2 → … → 清理）

按场景追加：
- 涉及共存策略 → **新旧并存阶段的数据流**（两套代码怎么共存、切换点在哪）
- 涉及回滚 → **回滚决策树**（不同失败点的回滚操作）

### upgrade 实施计划的特殊约束（施工契约）

**阶段骨架（强制，可按规模拆分子阶段）**：

```
阶段 0: 升级前准备
  - 具体动作：
    1. 创建升级分支
    2. 运行当前全量测试 + 记录结果
    3. 运行 bench（若有）+ 记录基线
    4. 捕获行为基线（若适用：样本请求/响应、端点对比）
    5. 准备回滚点（当前分支/tag）
  - 验收：基线完整，所有后续阶段都可对照

阶段 1: 基础设施升级
  - 具体动作：升级 Cargo.toml / package.json / 构建配置 中的目标版本号
  - 验收：能 build（可能有 warning，允许；有 error 需要阶段 N 解决）

阶段 2..N: 按升级路径逐模块 / 逐 breaking 修复
  - 每个阶段对应一个 breaking change 类别 或 一个模块
  - 具体动作：按"Breaking Changes 清单"的"迁移方案"逐条执行
  - 每阶段后全量测试必须通过

阶段 N+1: 废弃 API 清理（若包含）
  - 具体动作：替换废弃 API 为新 API
  - 验收：grep 废弃 API 为空

阶段 N+2: 兼容层清理（若使用了兼容层）
  - 触发条件：所有调用方已迁移完成（上面阶段已保证）
  - 具体动作：移除 adapter / feature flag / 废弃 API 桥接
  - 验收：grep 相关桥接代码为空

阶段 N+3: 最终验证
  - 具体动作：
    1. 全量构建 + 测试
    2. Bench 回归对比基线
    3. 行为对比基线
    4. 产物大小对比
  - 验收：全部对照通过，或有已解释的可接受差异
```

**强制**：每阶段必须独立可编译、可运行、可回滚。若某阶段内部规模大，再拆子阶段。

**禁止**：
- ❌ "一口气全改完再测" —— 违反独立可编译
- ❌ "兼容层之后再删" 而不指定清理阶段 —— 兼容层会永久化
- ❌ 没有回滚点 / 回滚触发条件

### upgrade 的生产级重点（施工时必须贯穿）

8 维度里 upgrade 要特别强化：

- **正确性与边界**：目标版本的新默认值 / 新行为可能引入静默 bug，必须审视
- **可观测性**：升级过程中的埋点（让运行时知道是旧行为还是新行为，便于对比）
- **安全**：新引入的依赖必须审视（传递依赖也算，不是直接依赖就不看）
- **性能**：必须有 bench 前后对比，或在计划里写明"本项目无 bench 覆盖，接受性能为盲区但在集成环境观察"
- **测试策略**：升级前补齐测试覆盖（若不足），升级后必须全过

**特别强调**：**每个阶段的中间态必须是可交付状态**，不能"中间阶段功能缺失无所谓反正没发布"——生产级要求升级过程中任何时点都能安全停下。

---

### upgrade 的施工纪律（写入计划的"施工纪律"字段）

- 每个里程碑（每个 breaking change 修复、兼容层清理、最终验证）独立 commit
- 每阶段末跑全量测试，失败立即停下
- 关键阶段后跑 bench 对比基线
- 兼容层阶段的代码用显式注释标记（`// COMPAT: remove after phase N+2`）
- 不允许夹带 feat / reshape 改动 —— 除非是 breaking change 的直接后果
- 若升级中途发现有显著难度（breaking 多于预期、迁移卡住），停下，回 `/blueprint --upgrade` 重新拆阶段

---

### upgrade 的反模式（计划里出现即不合格）

- ❌ 没有具体的驱动因素（CVE / 特性 / EOL 时间）
- ❌ 没有 breaking changes 系统清单，凭"试着跑看哪里报错"
- ❌ 没有基线 / 验证对照
- ❌ 没有回滚策略
- ❌ 兼容层没有清理计划
- ❌ 把 "顺便重构" 夹带进 upgrade —— reshape 和 upgrade 不要混，混了出问题分不清是谁的锅
- ❌ 用户未明确驱动因素时就规划升级

---

### upgrade 的风险与回滚

- 单阶段失败：回到该阶段初的 commit
- 中大型升级失败：回到阶段 0 的准备分支
- 预警信号：
  - 某阶段后测试失败数量远超预期 → 停下评估是否应该转渐进共存
  - Bench 退化大于计划容忍阈值 → 停下调查，可能需要阶段内进一步优化
  - 发现计划未列出的新 breaking → 停下补充清单，重排阶段
  - 兼容层清理阶段意外失败 → 说明还有调用方未迁移，回去补
