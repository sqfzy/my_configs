## 目的：bench（性能基线 / profile / 对比 / 优化）

**适用**：性能核查与优化——建立基线、剖析热点、前后对比、目标驱动优化。

**四种子任务**（由计划自动组织，不需用户选）：
- **建立基线**：首次跑 bench 记录，作为后续参照
- **Profile**：剖析热点，找"慢在哪"（不改代码）
- **前后对比**：同一套 bench 在某次改动前后的差异（不改代码）
- **目标驱动优化**：按性能目标迭代优化代码（**会改代码**）

本文件只定义 bench 目的**独有**的约束，通用维度由 SKILL.md 规定。

---

### bench 独有的必展开章节

#### 1. 目标声明

**必须在计划最显眼位置写一句**：

```
bench 子任务：
  [ ] 建立基线
  [ ] Profile 剖析
  [ ] 前后对比（<改动 A> vs <改动 B>）
  [ ] 目标驱动优化（目标：<指标> <比较符> <数值>，如 "p99 < 100ms" / "吞吐 > 10k QPS" / "内存 < 200MB"）
```

**陷阱**：
- 没有明确指标就"优化性能"—— 永远不知道何时停
- "优化优化"式的 bench 不是 bench，是 reshape 或 feat 的附带工作

#### 2. Benchmark 基础设施

必须明确：
- **测试框架**：cargo bench / pytest-benchmark / go test -bench / Criterion / …（沿用项目既有）
- **bench 对象**：哪些函数 / 场景 / 端点被测
- **关键指标**：时间（mean / p50 / p99）/ 吞吐 / 内存 / 分配数 …
- **数据集**：输入规模 / 分布（小 / 中 / 大），明确到具体数字
- **环境隔离**：
  - 关闭 turbo / 固定 CPU 频率（若可）
  - 排除首次冷启动数据
  - 足够的迭代次数（statistical significance）
- **复现命令**：完整的 build + bench 命令

**陷阱**：bench 结果无法复现就无意义——环境 / 数据集 / 命令必须全部明确。

#### 3. 基线数据（建立基线 / 对比 / 优化子任务必有）

捕获：
- Commit hash + 时间戳
- 构建配置（release / 优化等级）
- 编译器 / 工具链版本
- 每个 bench 的 mean / p50 / p99 / 标准差
- 迭代次数、总耗时

基线必须带**复现上下文注释**（commit / 时间 / 编译器 / 构建命令 / bench 命令），让将来任何人都能精确复现。

#### 4. Profile 数据（Profile / 目标驱动优化子任务必有）

使用项目适合的剖析工具（perf / valgrind / flamegraph / pprof / cargo-flamegraph / py-spy / …）：
- 热点函数 Top N（按 self time / total time）
- 调用栈关系
- 内存分配热点（若适用）
- 关键火焰图 / 调用图（若能生成）

剖析数据必须**导出成文件**并在计划中引用（火焰图路径、callgrind 文件路径），不要只截图描述。

#### 5. 前后对比（对比 / 优化子任务必有）

对比表必须：

```
| Benchmark | 基线 | 当前 | 变化 | 状态 |
|-----------|------|------|------|------|
| X | 100ms | 80ms | -20% | ✅ 提升 |
| Y | 50ms | 55ms | +10% | ⚠️ 退化 |
| Z | 1ms | 1ms | 0% | — 持平 |
```

**退化阈值**：
- < 5%：在噪声内，可忽略
- 5%–15%：轻度退化，需要解释
- ≥ 15%：显著退化，必须回滚或解决

**提升验证**：不只看 mean，看 p99 和标准差——mean 好 p99 差是常见陷阱。

#### 6. 优化策略（目标驱动优化子任务必有）

迭代优化的约束：
- **正确性铁律**：每轮优化后全量测试必须通过（性能提升不能以牺牲正确性换取）
- **目标收敛**：每轮优化后 bench 对比基线与目标——达到目标即停
- **回滚机制**：优化阶段若退化全局其他 bench ≥ 15%，该轮回滚
- **解释能力**：每轮优化必须有明确假设（"这个改动为什么会提升"），不允许试错式改代码

### bench 必含 ASCII 图示（按场景选用）

- **热点分布**（Profile / 优化必含）：Top N 热点的占比
- **前后对比图**（对比 / 优化必含）：每个 bench 的柱状对比
- **优化轨迹**（优化必含）：每轮优化后相对基线和目标的进度

### bench 实施计划的特殊约束（施工契约）

**bench 是否施工类，取决于子任务**：
- 建立基线 / Profile / 前后对比：**只读**（跑 bench / profile，不改代码）
- 目标驱动优化：**施工类**（会改代码）

**各子任务的阶段骨架**：

#### 建立基线

```
阶段 1: 基础设施就绪
  - 具体动作：确认 bench 框架可用、bench 函数已定义、数据集已准备
  - 验收：能跑 bench 且有稳定结果

阶段 2: 正式跑基线
  - 具体动作：在隔离环境下跑 N 次迭代，采集统计数据
  - 交付物：基线数据文件（带复现上下文注释）
  - 验收：标准差合理（通常 < 5% mean）；结果稳定

阶段 3: 归档
  - 具体动作：基线数据保存到 `.artifacts/bench-baseline-*.txt`（由 /report 归口）
  - 交付物：可引用的基线记录
```

#### Profile

```
阶段 1: 剖析运行
  - 具体动作：用 profiler 跑目标场景，导出剖析数据
  - 验收：热点清晰可见；数据已导出成文件

阶段 2: 热点分析
  - 具体动作：分析热点调用栈、分配点、同步点
  - 交付物：热点报告（Top N + 分析）
  - 验收：每个热点有"为什么热"的解释

阶段 3: 建议清单
  - 交付物：潜在优化点清单（每项：位置 / 假设 / 预期收益 / 成本）
```

#### 前后对比

```
阶段 1: 对比运行
  - 具体动作：在两个 commit / 分支分别跑相同 bench
  - 验收：两组数据都稳定

阶段 2: 差异分析
  - 具体动作：对照表 + 退化 / 提升的解释
  - 交付物：对比报告
  - 验收：每项变化有解释
```

#### 目标驱动优化（施工）

```
阶段 0: 准备
  - 具体动作：
    1. 运行全量测试 + 记录通过
    2. 跑基线 bench + 记录
    3. 确认目标指标（如 "p99 < 100ms"）
    4. 跑 profile 找初始热点
  - 验收：基线、目标、热点清晰

阶段 1..N: 迭代优化（每轮一个独立假设）
  - 具体动作：
    1. 针对某个热点提出改进假设
    2. 实施改动
    3. 运行全量测试（必须全过，否则立即回滚）
    4. 跑 bench 对比基线和目标
    5. 判断：达标 → 结束；提升但未达标 → 进入下轮；退化 → 回滚
  - 每轮独立 commit（message 明确"perf: <改动> — <bench 变化>"）
  - 验收：
    - 测试全过
    - 目标指标的 bench 有进展
    - 其他 bench 无显著退化（< 15%）

阶段 N+1: 最终验证
  - 具体动作：
    1. 全量测试
    2. 完整 bench 套件（不只目标指标）
    3. 对比基线确认无其他 bench 退化
  - 验收：达到目标；无其他 bench 退化 ≥ 15%；测试全过
```

**每阶段的 bench 结果必须保留**（作为后续再优化或排查的历史）。

### bench 的生产级重点（施工时必须贯穿）

- **正确性优先**：性能提升不能以牺牲测试通过为代价
- **可观测性**：bench 结果可复现、可追溯、可对比
- **不过度优化**：没有 benchmark 证据不做"优化"；有 bench 证据才改
- **最小改动原则**：每轮优化只改一个假设的点，便于定位贡献
- **优化代码也是生产级代码**：不为了性能牺牲可读性、安全性、错误处理

---

### bench 的施工纪律（优化子任务专用）

- 每一轮优化一个独立 commit（message 明确"perf: <改动> — <bench 变化>"）
- 每轮 commit 前全量测试必须通过
- 每轮 commit 后 bench 对比基线
- 不允许"先一口气优化完最后再测"
- 不允许"只跑目标 bench 不跑其他"—— 必须全套 bench 跑，防止一处提升一处退化
- 每轮有明确假设，试错式改代码不 OK
- 若改动涉及算法 / 数据结构大改，应该是 `/blueprint --reshape` 而不是 `/blueprint --bench --optimize`

---

### bench 的反模式（计划里出现即不合格）

- ❌ 没有明确目标指标的"优化性能"
- ❌ 没有基线就对比
- ❌ 只看 mean 不看 p99 / 标准差
- ❌ bench 结果无法复现（缺环境 / 命令 / 数据集说明）
- ❌ 用 debug 构建跑 bench
- ❌ 优化阶段发现 bug 不停下继续优化
- ❌ 为了提升某个 bench 隐式退化其他 bench
- ❌ 试错式优化，没有每轮的假设
- ❌ bench 代码本身有 bug（测错了对象、被编译器优化掉了、fixtures 污染）

---

### bench 的风险与回滚

- 某轮优化后全量测试失败：立即回滚该轮
- 某轮优化后其他 bench 退化 ≥ 15%：回滚该轮，评估改动是否本质有问题
- N 轮后仍未达到目标：
  - 评估目标是否现实（可能原始假设过高）
  - 评估是否应升级为 `/blueprint --reshape`（算法 / 数据结构本身需要换）
- Bench 结果噪声过大（标准差 > 15% mean）：
  - 环境隔离不足（先修环境再继续）
  - 数据集太小（增大迭代次数）
- Profile 数据和直觉矛盾：相信数据，不相信直觉；先重新 profile 确认
