## 核心理念

`/fix` 是灭火，`/ship` 是出门前检查，`/improve` 是定期体检。

代码能跑、测试通过、没有 bug——不代表代码好。`/improve` 解决的是"没坏但可以更好"的问题：模块设计不合理、职责划分不清、耦合过紧、API 难以扩展、抽象层级错位、命名有歧义、测试覆盖有盲区、性能有优化空间。

**改进哲学**：追求设计最优，一步到位，而非最小改动或修补现有。发现设计不合理时，推荐正确的设计方案，而非在错误设计上打补丁。不要因为"改动范围大"就降级为局部修补——improve 的目标就是让代码达到理想状态。

流程分三段：
1. **全局审视**——建立问题全景，按严重程度排优先级
2. **迭代改进**——多角色对抗讨论 + 实施 + 验证，循环直到收敛
3. **沉淀复盘**——把改进过程中的教训、决策、洞察写下来，供未来参考

## 参数解析

- **目标**（必填）：指定要改进的文件、模块、或功能域
- `[iter: N]`：最大迭代次数上限；未指定则**不设上限**，持续迭代直到收敛（Phase 1 的 Critical/Major 全部消除且连续两轮无新问题）
- `[depth: quick]`：快速模式——审查 + 1 轮迭代 + 简要复盘，适合小模块或时间有限时；默认 `full`
- `[auto]`：无人值守模式——跳过迭代确认自动继续；性能退化时自动回滚该轮改动并继续；迭代陷入循环时自动终止

## Phase 1: 全局审视

在改任何代码之前，先建立完整的问题全景。

### 1.1 建立基线

运行构建和测试，建立基线：

- ✅ 通过 → 记录测试数量，继续
- ❌ 失败 → 终止：`❌ 基线未通过。请先修复现有问题（/fix），再运行 /improve。`

按 Bench 感知约定执行基线检查。

### 1.2 架构建模

在逐行审查之前，先建立目标代码的**结构性理解**。设计问题藏在组件关系中，不是单个文件里。

**依赖图**：梳理目标范围内的模块/类/trait 之间的依赖关系，输出文本依赖图：

```
## 架构地图

### 依赖图
A → B        A 调用 B 的公共接口
A → C        A 直接访问 C 的内部类型
B → C        B 依赖 C 的 trait
D → A, B     D 同时依赖 A 和 B
C → A        ⚠️ 循环依赖

### 职责映射
| 模块 | 当前职责 | 职责是否单一 |
|------|----------|-------------|
| A | 解析 + 校验 + 序列化 | ❌ 混杂 |
| B | 网络 I/O | ✅ |
| C | 配置管理 | ✅ |
| D | 编排（调用 A, B） | ✅ |

### 信息流
用户输入 → D(编排) → A(解析) → B(发送) → 外部服务
                   ↘ C(配置) ↗

### 耦合热点
- A↔C 循环依赖：A 引用 C 的配置类型，C 反向引用 A 的解析结果
- D 同时持有 A 和 B 的具体类型，无法替换实现
```

**公共 API 快照**：记录目标范围内所有公共接口签名（函数签名、trait 定义、导出类型），作为后续"无损改进"的契约基线。

```
### 公共 API 快照
- `pub fn parse(input: &str) -> Result<Ast, ParseError>`
- `pub trait Validator { fn validate(&self, ast: &Ast) -> Result<(), Vec<Error>>; }`
- `pub struct Config { pub timeout: Duration, pub max_retries: u32 }`
- ...
```

此快照在 Phase 2 验证环节用于检查接口契约是否被意外破坏。

### 1.3 深度阅读

全面阅读目标代码，从以下维度审查：

**正确性**：
- 逻辑错误、边界条件、off-by-one
- 错误处理：`unwrap()` / 静默忽略 `Result` / 空 `catch` 块
- 并发安全：共享状态是否有正确的同步
- 资源管理：所有路径（含错误路径）是否正确释放

**安全性**：
- 输入校验、注入风险、敏感数据泄漏
- unsafe 块是否有 safety 注释且不变量成立
- C++ 未定义行为：悬垂引用、use-after-move

**性能**：
- 热路径上不必要的开销
- 循环内堆分配、可避免的 `clone()`
- 算法复杂度是否合理

**设计**：
- 模块职责划分是否清晰，是否存在"上帝对象"或职责混杂
- 模块间耦合度：是否存在不合理的循环依赖或越层调用
- 依赖方向：高层模块是否依赖了低层细节（依赖倒置）
- API 设计：接口是否表意、一致、难以误用（pit of success）
- 扩展性：新增同类功能是否需要修改核心代码（开闭原则）
- 状态管理：可变状态的作用域是否最小化，是否存在隐式全局状态

**可维护性**：
- 命名是否准确传达意图
- 抽象层级是否合理（过度抽象 vs 重复代码）
- 非显然逻辑是否有注释解释 **why**

**代码风格统一性**（横向扫描，跨文件汇总）：
- **命名约定**：模块/类型/函数/变量/常量是否遵循一致的命名风格（snake_case / camelCase / PascalCase / SCREAMING_SNAKE）；同一概念是否用同一名字（如 `id` vs `identifier` vs `key` 混用）；动词选择是否统一（`get_*` vs `fetch_*` vs `load_*`、`build_*` vs `create_*` vs `make_*`）
- **错误处理模式**：是否统一（`Result<T, E>` 单一错误类型 vs 多种 / panic / 静默忽略并存）
- **日志/tracing 风格**：调用模式、字段命名、级别使用是否一致
- **导入/模块组织**：导入顺序、re-export 风格、模块划分粒度是否一致
- **注释/文档风格**：doc comment 格式、TODO/FIXME 标记规范是否统一
- **格式化细节**：行宽、空行、括号风格——若已有 formatter 配置则忽略，否则记录

**测试覆盖**：
- 公共函数是否有测试
- 错误路径和边界条件是否被覆盖
- 测试命名是否描述场景

**可观测性**：
- 关键路径是否有日志
- 错误日志是否包含足够上下文

### 1.4 问题全景

除了"问题清单"，**额外输出"风格不一致清单"**——单点都不算 bug，但累积起来侵蚀可读性，必须统一治理。

```
## 审查结果

### 问题清单
| 序号 | 严重程度 | 类型 | 位置 | 描述 | 建议 |
|------|----------|------|------|------|------|
| 1 | 🔴 Critical | 正确性 | `src/parser.rs:87` | 空输入导致 panic | 改用 `?` 传播 |
| 2 | 🟡 Major | 测试 | `src/core.rs` | `process()` 完全无测试 | 补充 happy + error path |
| 3 | 🟡 Major | 性能 | `src/io.rs:42` | 循环内重复打开文件 | 提到循环外 |
| 4 | 🔵 Minor | 命名 | `src/util.rs:15` | `do_thing()` 命名不明确 | 改为 `normalize_path()` |
| ... | ... | ... | ... | ... | ... |

### 风格不一致清单（跨文件汇总）
| 维度 | 当前混用情况 | 建议 canonical 形式 | 影响范围 |
|------|--------------|---------------------|----------|
| 命名：标识符 | `id` (12 处) / `identifier` (3 处) / `key` (5 处) 指代同一概念 | 统一为 `id` | 20 处 |
| 命名：构造动词 | `build_*` (8 处) / `create_*` (4 处) / `new_*` (6 处) 语义无差异 | 工厂用 `build_*`，构造函数沿用 `new` | 18 处 |
| 错误类型 | `MyError` / `String` / `anyhow::Error` 共存 | 统一为 `MyError`，外层用 `anyhow` | 3 个模块 |
| 日志字段 | `user_id` / `userId` / `uid` 混用 | `user_id` | 14 处 |
| TODO 格式 | `// TODO`、`// todo:`、`// FIXME` 并存 | `// TODO(name): ...` | 9 处 |
| ... | ... | ... | ... |

### 优先级排序
1. [Critical] <描述> — 不修可能导致运行时错误
2. [Major] <描述> — 测试盲区，隐患
3. [Major] <描述> — 性能问题
4. [Minor] <描述> — 可读性改善
5. [风格统一] <一组相关项> — 累积侵蚀可读性，建议作为独立迭代批量处理
...

### 总体评价
<2–3 句话：代码整体质量判断、最需要关注的领域>
```

**风格统一专项处理原则**：
- 风格不一致项**不要逐条夹在普通问题清单里**——会被淹没。单独成表，并在 Phase 2 用一个独立迭代批量统一
- 选定 canonical 形式时倾向于**项目中已是多数派**的写法，除非该多数派违反语言官方约定（如 Rust 的 snake_case）
- 自动 formatter 能处理的（行宽、缩进、引号）不写进清单——委托给 formatter，记录"建议引入/启用 formatter"即可

Phase 1 的输出（问题清单 + 架构地图 + 公共 API 快照）是 Phase 2 的输入。

## Phase 2: 迭代改进

以下流程**循环执行**，每次循环为一次迭代。

### 2.1 议题提取与分类

**第一次迭代**：从 Phase 1 的问题清单中，取最高优先级的 1–3 个问题作为本轮议题。

**后续迭代**：上一轮遗留问题 + 新引入代码可能暴露的新问题。

**分类**：将本轮议题标记为以下三类之一：

- **设计类**：涉及模块职责、依赖方向、接口边界、耦合关系等结构性问题——需要整体性改动，不能逐 issue 修补
- **局部类**：正确性 bug、性能热点、单点命名、测试缺失等——可独立修复，不影响整体架构
- **风格统一类**：来自 1.4 的"风格不一致清单"——一组跨文件的小改动（重命名、错误类型替换、日志字段对齐等），逐项极小但必须**作为一个迭代整体处理**，否则容易半途而废

**分流规则**：
- 若本轮议题包含任何设计类问题 → 整轮走**设计类流程**（2.2a）
- 否则若全部为风格统一类 → 走**风格统一流程**（2.2c）
- 否则 → 走**局部类流程**（2.2b）

### 2.2a 设计类流程：目标架构收敛 → 拓扑序迁移

设计问题的本质是"当前架构不合理"，逐个修 issue 只会制造更多问题。正确做法是：先收敛出目标架构，再按依赖拓扑序迁移。

**Step 1：目标架构确定**

选出 **5 个角色**（**必选 R14 架构师**），进行 3–6 轮对抗讨论，收敛出目标架构：

- R14 负责提出目标架构方案（模块划分、依赖方向、接口边界）
- 其他角色从各自立场质疑和补充
- 讨论必须基于 Phase 1.2 的架构地图，指出"从 X 迁移到 Y"

**讨论输出**：
```
## 目标架构

### 当前 vs 目标对比
| 模块 | 当前职责 | 目标职责 | 变化 |
|------|----------|----------|------|
| A | 解析+校验+序列化 | 仅解析 | 拆分出 Validator 和 Serializer |
| B | 网络 I/O | 不变 | — |
| ... | ... | ... | ... |

### 目标依赖图
A(解析) → B(发送)
Validator → A（依赖 AST 类型）
Serializer → A（依赖 AST 类型）
D(编排) → Validator, Serializer, B

### 迁移步骤（拓扑序）
1. 定义新接口（trait/interface），不改现有实现
2. 拆分 A → A + Validator（A 无调用方变化）
3. 拆分 A → A + Serializer（同上）
4. 更新 D 的依赖，从直接依赖 A 改为依赖 trait
5. 清理旧代码

### 接口契约
- 每步迁移后，公共 API 签名不变（或变化是有意为之且已标注）
- 每步迁移后，所有测试通过
```

**Step 2：逐步迁移**

按拓扑序逐步实施，每步：
1. 实施改动
2. 编译 + 测试验证
3. 接口契约检查（见 2.4）
4. 提交（一步一 commit，便于回滚）

**若迁移某步失败且无法修复**：回滚到上一步的 commit，记录失败原因，重新讨论该步的迁移策略。

### 2.2c 风格统一流程：批量整改

风格统一改动单点小、数量多，特征是**一致性 > 单点最优**——选定 canonical 形式后机械替换，不要每处都重新讨论。

**Step 1：canonical 形式确认**

向用户呈现"风格不一致清单"中本轮要处理的项，对每项给出推荐 canonical 形式 + 一句话理由。用户确认后冻结，整轮迭代不再讨论替代方案。

**`auto` 模式**：直接采用 1.4 清单中的 canonical 形式。

**Step 2：批量替换**

按维度逐项执行，每个维度独立成 commit：

1. 列出所有需替换位置（grep / LSP rename 辅助）
2. 执行替换（重命名优先用工具的 rename symbol，避免误伤字符串/注释中的同名词）
3. 编译 + 测试验证
4. 通过 → commit；失败 → 排查是否漏掉调用方或误改了非目标位置

**Step 3：防回归**

若项目支持，配置 lint/format 规则把 canonical 形式固化（如 clippy 自定义 lint、pyproject.toml 的 ruff 规则、.editorconfig）；否则在 CLAUDE.md 或贡献指南里明确写出约定，避免下次又混乱。

**禁忌**：
- 不要在风格统一迭代中夹带功能修改——破坏 commit 原子性，回滚困难
- 不要因为某处"特殊场景"就破例保留旧形式——要么 canonical 形式选错了重新讨论，要么就贯彻到底

### 2.2b 局部类流程：多角色对抗讨论

针对本轮议题，选出 **5 个角色**进行对抗讨论。

!`cat ~/.claude/skills/shared/roles.md`

根据本轮议题动态调整角色选择——正确性问题侧重 R1/R10/R13，性能问题侧重 R3/R2/R4，设计/结构问题侧重 R14/R5/R6/R8。

**讨论轮数**：根据议题复杂度 2–6 轮，优先简洁收敛。

**讨论输出**：
```
## 迭代 N 讨论结论
- 发现的问题：<列表，注明 critical / major / minor>
- 建议的改动：<具体、可操作>
- 搁置的争议：<留待后续迭代>
```

### 2.3 实施改动

按 `critical → major → minor` 优先级实施。

**实施原则**：
- 每次改动后立即验证编译通过，不批量堆积
- 接口变更同步更新所有调用方
- 遵循项目已有风格和约定
- 非显然逻辑附注释

### 2.4 验证

**构建 + 测试**：

运行构建和测试验证。按 Bench 感知约定执行回归对比。

**接口契约检查**：

将当前公共 API 签名与 Phase 1.2 的公共 API 快照对比：

```
### 接口契约检查
| 接口 | 状态 | 说明 |
|------|------|------|
| `parse(input: &str) -> Result<Ast, ParseError>` | ✅ 不变 | — |
| `Validator::validate(&self, ast: &Ast)` | 🆕 新增 | 从 A 拆分出的新 trait |
| `process(input: &str) -> Output` | ⚠️ 签名变化 | 返回类型从 `Output` 改为 `Result<Output, Error>` |
```

- ✅ 不变 / 🆕 有意新增 → 通过
- ⚠️ 签名变化 → 必须确认：这是有意的设计决策还是意外破坏？有意变化需在迭代摘要中标注理由；意外变化则修复
- ❌ 接口消失 → 除非目标架构明确要求移除，否则视为回归

**结果处理**：
- ✅ 全部通过 → 若存在 benchmark 且本轮涉及性能代码，运行 benchmark 检查回归。进入 2.5
- ❌ 编译失败 → 立即修复，重新验证
- ❌ 测试失败 → 分析原因：
  - 实现问题 → 修复代码
  - 测试需同步更新（接口变更）→ 更新测试
  - 无法修复 → 回滚本轮改动，记录原因，进入 2.5
- ❌ 接口契约意外破坏 → 修复或回滚

### 2.5 迭代评估与终止判断

**记录本轮摘要**（使用清晰的视觉格式）：
```
╔══════════════════════════════════╗
║  迭代 N — <时间戳>               ║
╚══════════════════════════════════╝

  议题       ： <一句话>
  参与角色   ： <列表>
  改动       ： <摘要>
  验证       ： ✅ / ❌ 部分回滚

  ── 问题消化 ──
  ✅ <消除的问题 1>
  ✅ <消除的问题 2>

  ── 遗留 ──
  ⏳ <未解决的点>
```

**终止条件**（按优先级，任一满足即终止）：

1. **Phase 1 的 Critical 和 Major 问题全部消除，且连续两轮无新的 Major+ 问题** → 收敛
2. **已达用户指定的迭代上限**（若指定了 `iter: N`）→ 终止，输出未完成的问题清单
3. **编译或测试无法恢复** → 终止，输出当前状态
   - 跳过 Phase 3 复盘，直接终止（复盘在最终完成时再做）
5. **用户中断** → 终止

每轮结束询问：
```
┌─────────────────────────────────────────┐
│  ✅ 迭代 N 完成                          │
│                                         │
│  已消除 ： X / Y 个 Critical+Major 问题  │
│  本轮改动 ： +A / -B 行，C 个文件        │
│  验证     ： ✅ 构建通过  ✅ 测试通过     │
│                                         │
│  [继续] / [停止]                         │
└─────────────────────────────────────────┘
```

**`auto` 模式**：跳过询问，自动继续。仅通过条件 1–3 判断终止。

## depth: quick 模式

快速模式精简为：

1. **Phase 1**：正常执行，但审查维度限于正确性 + 测试覆盖（跳过性能、可观测性深度分析）
2. **Phase 2**：仅 1 轮迭代，聚焦 Critical 和 Major 问题
3. **Phase 3**：仅输出 TL;DR + Top 5 教训 + 遗留问题

报告文件名加 `-quick` 后缀。

适用场景：小模块快速打磨、PR 前的最后一遍检查、时间有限但想走一遍流程。

## 异常流程处理

### 迭代陷入循环

若连续 3 轮迭代修复的问题数为 0（每轮发现新问题但无法收敛）：

```
⚠️ 迭代似乎陷入循环——连续 3 轮未消除问题。

可能原因：
  - 问题之间存在相互依赖，需要更大范围的重构
  - 当前抽象层级不足以支撑改进

建议：
  - 使用 /discuss 对核心设计分歧做深度对抗讨论
  - 使用 /refactor 进行结构性调整后重新 /improve
```

暂停，保存当前进度到报告。

**`auto` 模式**：自动终止并保存报告，不暂停。

### Benchmark 退化

迭代过程中发现性能退化：

```
⚠️ 迭代 N 引入性能退化：
  - <benchmark>：<基线> → <当前>（退化 X%）

[回滚本轮并跳过该优化 / 接受退化继续 / 尝试优化]
```

用户选择后继续或回滚。所有退化记录（无论接受与否）都写入最终报告。

**`auto` 模式**：自动回滚本轮改动并跳过该优化，在报告中记录退化详情，继续下一轮迭代。
