## 核心理念

`/fix` 是灭火，`/ship` 是出门前检查，`/improve` 是定期体检。

代码能跑、测试通过、没有 bug——不代表代码好。`/improve` 解决的是"没坏但可以更好"的问题：模块设计不合理、职责划分不清、耦合过紧、API 难以扩展、抽象层级错位、命名有歧义、测试覆盖有盲区、性能有优化空间。

流程分三段：
1. **全局审视**——建立问题全景，按严重程度排优先级
2. **迭代改进**——多角色对抗讨论 + 实施 + 验证，循环直到收敛
3. **沉淀复盘**——把改进过程中的教训、决策、洞察写下来，供未来参考

---

## 参数解析

- **目标**（必填）：指定要改进的文件、模块、或功能域
- `[iter: N]`：最大迭代次数上限；未指定则**不设上限**，持续迭代直到收敛（Phase 1 的 Critical/Major 全部消除且连续两轮无新问题）
- `[depth: quick]`：快速模式——审查 + 1 轮迭代 + 简要复盘，适合小模块或时间有限时；默认 `full`
- `[auto]`：无人值守模式——跳过迭代确认自动继续；性能退化时自动回滚该轮改动并继续；迭代陷入循环时自动终止

## Phase 1: 全局审视

在改任何代码之前，先建立完整的问题全景。

### 1.1 建立基线

```
若用户提供了构建/测试/benchmark 命令则优先使用；否则根据项目构建系统和配置，自行确定并执行构建、测试与 benchmark 命令；若项目无测试或 benchmark 则跳过对应步骤。
```

- ✅ 通过 → 记录测试数量，继续
- ❌ 失败 → 终止：`❌ 基线未通过。请先修复现有问题（/fix），再运行 /improve。`

若存在与目标相关的 benchmark，运行并记录基线。

### 1.2 深度阅读

全面阅读目标代码，从以下维度审查：

**正确性**：
- 逻辑错误、边界条件、off-by-one
- 错误处理：`unwrap()` / 静默忽略 `Result` / 空 `catch` 块
- 并发安全：共享状态是否有正确的同步
- 资源管理：所有路径（含错误路径）是否正确释放

**安全性**：
- 输入校验、注入风险、敏感数据泄漏
- unsafe 块是否有 safety 注释且不变量成立
- C++ 未定义行为：悬垂引用、use-after-move

**性能**：
- 热路径上不必要的开销
- 循环内堆分配、可避免的 `clone()`
- 算法复杂度是否合理

**设计**：
- 模块职责划分是否清晰，是否存在"上帝对象"或职责混杂
- 模块间耦合度：是否存在不合理的循环依赖或越层调用
- 依赖方向：高层模块是否依赖了低层细节（依赖倒置）
- API 设计：接口是否表意、一致、难以误用（pit of success）
- 扩展性：新增同类功能是否需要修改核心代码（开闭原则）
- 状态管理：可变状态的作用域是否最小化，是否存在隐式全局状态

**可维护性**：
- 命名是否准确传达意图
- 抽象层级是否合理（过度抽象 vs 重复代码）
- 非显然逻辑是否有注释解释 **why**

**测试覆盖**：
- 公共函数是否有测试
- 错误路径和边界条件是否被覆盖
- 测试命名是否描述场景

**可观测性**：
- 关键路径是否有日志
- 错误日志是否包含足够上下文

### 1.3 问题全景

```
## 审查结果

### 问题清单
| 序号 | 严重程度 | 类型 | 位置 | 描述 | 建议 |
|------|----------|------|------|------|------|
| 1 | 🔴 Critical | 正确性 | `src/parser.rs:87` | 空输入导致 panic | 改用 `?` 传播 |
| 2 | 🟡 Major | 测试 | `src/core.rs` | `process()` 完全无测试 | 补充 happy + error path |
| 3 | 🟡 Major | 性能 | `src/io.rs:42` | 循环内重复打开文件 | 提到循环外 |
| 4 | 🔵 Minor | 命名 | `src/util.rs:15` | `do_thing()` 命名不明确 | 改为 `normalize_path()` |
| ... | ... | ... | ... | ... | ... |

### 优先级排序
1. [Critical] <描述> — 不修可能导致运行时错误
2. [Major] <描述> — 测试盲区，隐患
3. [Major] <描述> — 性能问题
4. [Minor] <描述> — 可读性改善
...

### 总体评价
<2–3 句话：代码整体质量判断、最需要关注的领域>
```

Phase 1 的输出是 Phase 2 的输入——迭代改进按此优先级逐一攻克。

## Phase 2: 迭代改进

以下流程**循环执行**，每次循环为一次迭代。

### 2.1 议题提取

**第一次迭代**：从 Phase 1 的问题清单中，取最高优先级的 1–3 个问题作为本轮议题。

**后续迭代**：上一轮遗留问题 + 新引入代码可能暴露的新问题。

### 2.2 多角色对抗讨论

针对本轮议题，选出 **5 个角色**进行对抗讨论。

**角色库**：

| 代号 | 角色 | 核心偏好 |
|------|------|----------|
| R1 | 风险卫士 | 失败模式、边界条件、防御性编程 |
| R2 | 极简主义者 | 反对一切不必要的复杂性 |
| R3 | 性能狂热者 | 吞吐量 / 延迟 / 资源效率 |
| R4 | 实用主义者 | 实证优先，拒绝过度设计 |
| R5 | 第一性原理者 | 质疑所有假设，回归根本需求 |
| R6 | 维护性倡导者 | 三年后的可读性和可维护性 |
| R8 | 激进探索者 | 无视当前约束，追求最优设计 |
| R10 | 安全专家 | 安全性、资源泄漏、未定义行为 |
| R11 | 怀疑论者 | 默认质疑，要求严格举证 |
| R13 | 测试驱动者 | 测试覆盖、可测试性、回归安全 |

**选角要求**：5 个角色必须存在真实价值冲突，不选立场相近的组合。根据本轮议题动态调整——正确性问题侧重 R1/R10/R13，性能问题侧重 R3/R2/R4，结构问题侧重 R5/R6/R8。

若预定义角色无法覆盖领域需求，可临时创建自定义角色：
```
【自定义】<角色名>
核心偏好：<判断标准>
预期冲突：<与其他角色的分歧点>
```

**讨论轮数**：根据议题复杂度 2–6 轮，优先简洁收敛。

**每轮格式**：
```
【角色名 | 立场】
论点：...（指向具体代码位置）
反驳 [角色]：... ← 第 2 轮起必填
立场修正：... ← 最后一轮必填
```

**讨论输出**：
```
## 迭代 N 讨论结论
- 发现的问题：<列表，注明 critical / major / minor>
- 建议的改动：<具体、可操作>
- 搁置的争议：<留待后续迭代>
```

### 2.3 实施改动

按 `critical → major → minor` 优先级实施。

**实施原则**：
- 每次改动后立即验证编译通过，不批量堆积
- 接口变更同步更新所有调用方
- 遵循项目已有风格和约定
- 非显然逻辑附注释

### 2.4 验证

```
若用户提供了构建/测试/benchmark 命令则优先使用；否则根据项目构建系统和配置，自行确定并执行构建、测试与 benchmark 命令；若项目无测试或 benchmark 则跳过对应步骤。
```

**结果处理**：
- ✅ 通过 → 若存在 benchmark 且本轮涉及性能代码，运行 benchmark 检查回归。进入 2.5
- ❌ 编译失败 → 立即修复，重新验证
- ❌ 测试失败 → 分析原因：
  - 实现问题 → 修复代码
  - 测试需同步更新（接口变更）→ 更新测试
  - 无法修复 → 回滚本轮改动，记录原因，进入 2.5

### 2.5 迭代评估与终止判断

**记录本轮摘要**：
```
### 迭代 N — <时间戳>
- 议题：<一句话>
- 参与角色：<列表>
- 改动：<摘要>
- 验证：✅ / ❌ 部分回滚
- Phase 1 问题消化：<消除了哪些问题>
- 遗留：<未解决的点>
```

**终止条件**（按优先级，任一满足即终止）：

1. **Phase 1 的 Critical 和 Major 问题全部消除，且连续两轮无新的 Major+ 问题** → 收敛
2. **已达用户指定的迭代上限**（若指定了 `iter: N`）→ 终止，输出未完成的问题清单
3. **编译或测试无法恢复** → 终止，输出当前状态
   - 跳过 Phase 3 复盘，直接终止（复盘在最终完成时再做）
5. **用户中断** → 终止

每轮结束询问：
```
✅ 迭代 N 完成（已消除 X/Y 个 Critical+Major 问题）。
   [继续 / 停止]
```

**`auto` 模式**：跳过询问，自动继续。仅通过条件 1–3 判断终止。

## Phase 3: 沉淀复盘

迭代结束后，对整个改进过程做结构化复盘。

**`depth: quick` 时简化为**：TL;DR + Top 5 教训 + 遗留问题，跳过时间线重建和决策记录。

### 3.1 时间线重建

从 Phase 2 的迭代记录中重建改进时间线：

```
## 改进时间线

Phase 1 审查 — 识别 N 个问题（X Critical, Y Major, Z Minor）
  │
  ├─ 迭代 1 — <议题>，消除 <问题>
  │     └─ ⚠️ <若有回滚或方向调整>
  │
  ├─ 迭代 2 — <议题>，消除 <问题>
  │
  └─ 迭代 N — 收敛

总计：<N 轮迭代，M 个问题消除，+X/-Y 行>
```

### 3.2 坑与教训

对改进过程中踩过的坑（回滚、错误判断、意外发现），用以下结构记录：

```
### 坑 N：<一句话>

**背景**：<当时在做什么>
**现象**：<出了什么问题>
**根因**：<为什么>
**教训**：<下次怎么避免>
**代码位置**：`<file>:<line>`
```

### 3.3 心得与洞察

记录正向收获——设计上的顿悟、发现的好模式、改进后特别满意的地方：

```
### 心得 N：<一句话>

**发现场景**：<什么时候意识到的>
**核心洞察**：<具体内容>
**应用条件**：<什么时候适用>
```

### 3.4 关键决策

记录改进中做出的重要取舍：

```
### 决策 N：<标题>

**选项**：A vs B
**选择**：A
**原因**：<为什么选 A 不选 B>
**现在回看**：<对不对>
```

## depth: quick 模式

快速模式精简为：

1. **Phase 1**：正常执行，但审查维度限于正确性 + 测试覆盖（跳过性能、可观测性深度分析）
2. **Phase 2**：仅 1 轮迭代，聚焦 Critical 和 Major 问题
3. **Phase 3**：仅输出 TL;DR + Top 5 教训 + 遗留问题

报告文件名加 `-quick` 后缀。

适用场景：小模块快速打磨、PR 前的最后一遍检查、时间有限但想走一遍流程。

## 异常流程处理

### 迭代陷入循环

若连续 3 轮迭代修复的问题数为 0（每轮发现新问题但无法收敛）：

```
⚠️ 迭代似乎陷入循环——连续 3 轮未消除问题。

可能原因：
  - 问题之间存在相互依赖，需要更大范围的重构
  - 当前抽象层级不足以支撑改进

建议：
  - 使用 /discuss 对核心设计分歧做深度对抗讨论
  - 使用 /refactor 进行结构性调整后重新 /improve
```

暂停，保存当前进度到报告。

**`auto` 模式**：自动终止并保存报告，不暂停。

### Benchmark 退化

迭代过程中发现性能退化：

```
⚠️ 迭代 N 引入性能退化：
  - <benchmark>：<基线> → <当前>（退化 X%）

[回滚本轮并跳过该优化 / 接受退化继续 / 尝试优化]
```

用户选择后继续或回滚。所有退化记录（无论接受与否）都写入最终报告。

**`auto` 模式**：自动回滚本轮改动并跳过该优化，在报告中记录退化详情，继续下一轮迭代。
