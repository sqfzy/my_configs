## Mode：incident（事故复盘 / post-mortem）

**适用**：系统出过事故、故障、生产问题，要写一份 post-mortem 给团队 / 管理层 / 客户看。

**核心职责**：透明、准确、可追溯地记录**时间线 + 根因 + 影响 + 缓解 + 预防**，作为组织学习材料（而不是追责工具）。

---

### 读者画像

- **SRE / 工程团队**：关心时间线 + 根因 + 技术细节 + 预防措施
- **客户 / 用户**（对外版本）：关心影响 + 缓解 + 以后不会再发生的保证
- **管理层**：关心业务影响 + 资源需求 + 系统性问题

**共同特征**：
- 情绪敏感 —— 事故刚发生过，人都紧张
- 对"粉饰"极其敏感 —— 一旦嗅到粉饰立即失去信任
- 技术深浅差别大 —— 一份报告可能多种读者

### 必含章节

```
## TL;DR
<2-3 句：发生了什么 + 影响范围 + 已解决 / 进行中 + 根因简述>

## 事故摘要
- 事故 ID：<编号 / 内部标识>
- 发生时间：<UTC 时间戳，精确到分钟>
- 发现时间：<何时被检测到 —— 监控告警 / 用户报告 / 工程师发现>
- 解决时间：<恢复正常的时间>
- 总影响时长：<N 小时 M 分钟>
- 严重程度：<P0 / P1 / P2 / P3 或项目既定分级>
- 状态：<已解决 / 缓解中 / 监控中>

## 影响范围
- 受影响的用户 / 客户数
- 受影响的功能 / 服务
- 数据损失 / 完整性（若有）
- 业务影响（收入 / SLA / 合规）
- 外部可见性（公开 / 内部）

## 时间线（关键节点）
<严格按时间顺序，UTC 时间戳>
- HH:MM  <事件描述 + 谁做了什么 + 根据什么信号>
- HH:MM  ...

<时间线必须包括：首次发生 / 首次告警 / 首次确认 / 各次缓解尝试 / 恢复 / 确认稳定 />

## 根因（Root Cause）
<用"5 Whys" 或因果链追到最深可操作的原因
 不允许停在"某人操作失误" —— 那是表象，必须问"为什么系统允许这种错误 / 为什么没有防护">

## 促发因素（Contributing Factors）
<除根因之外让事故变严重的因素
 例：监控缺失、runbook 过期、人员不足、依赖脆弱>

## 做对的事（What Went Well）
<事故处理过程中做得好的地方
 例：告警及时、缓解手段有效、沟通顺畅
 **必须有** —— 事故复盘不是自我批判大会，也要承认做对的>

## 做错的事（What Went Wrong）
<客观列出 —— 不追责个人，但不回避系统性问题>

## 缓解措施（已完成）
<事故期间已采取的恢复措施>

## 预防措施（Action Items）
<防止同类事故再发生的具体行动
 每条：
  - 行动描述
  - 负责人
  - 目标完成时间
  - 优先级（immediate / short-term / long-term）
  - 关联的 issue / PR / ticket>
<每条必须有负责人和时间 —— 没有负责人的 action item 不算 action item>

## 时间线图（ASCII 必含）
<从首次问题到完全恢复的可视化时间线>

## 技术细节（给工程师）
<根因的详细技术说明
 含代码位置、配置、相关 commit、监控截图路径引用>

## 相关
- 相关 commits: <导致事故的 commit + 修复 commit>
- 相关 issues / PRs: <链接>
- Runbook: <执行的应急流程文档>
- 监控: <相关仪表盘 / 告警>
```

### 素材收集侧重

- **用户输入**：事故描述、发现方式、严重度评估、业务影响（/report 不能自己知道这些）
- **git log**：事故前的 commit（寻找触发改动）+ 事故期间的 commit（修复与缓解）
- **.artifacts/**：若有相关 fix / incident 历史
- **外部工具**（用户粘贴）：监控截图、日志片段、告警记录

### 语气 / 深度

- **透明**：不回避任何事实
- **无责备**（blameless）：不点名个人 "Alice 犯了错"；而是 "系统允许了 X，没有守住 Y"
- **精确**：时间戳精确到分钟，数字有单位
- **诚实**：承认未知 —— 如果某些时间点的动作记录缺失，明确标注"记录缺失"
- **避免术语堆砌对外版本**：给客户看的版本用中性语言，技术细节放附录

### ASCII 图示要求

**强制**：
- **时间线图**：首次故障 → 告警 → 诊断 → 缓解尝试 → 恢复 → 稳定（带时间戳）

按场景追加：
- **根因因果链**：从现象到根因的因果图
- **影响范围图**：哪些系统 / 用户受影响
- **架构失效点**：在架构图上标注失败的组件

### 禁忌

- ❌ **点名追责**："Alice 把配置改错了" —— 改为 "配置变更流程缺少 review 环节"
- ❌ 粉饰严重度（真实 P0 写成 P2）
- ❌ 遗漏"做对的事"（看起来不专业也不健康）
- ❌ Action items 没有负责人或时间（等于没有）
- ❌ 时间线含糊（"大约下午三点左右" —— 必须精确到分钟）
- ❌ 根因停在表层（"代码有 bug" —— 必须继续问"为什么 bug 没被测试 / review 抓到"）
- ❌ 对客户版本透露内部系统细节 / 密钥 / 内部 IP
- ❌ 把预防措施写成"加强培训" / "更小心" —— 这些不是具体的可追踪 action
- ❌ 让事故报告变成自我表扬（"团队表现卓越" —— 留给做对的事章节，不要充斥报告）
