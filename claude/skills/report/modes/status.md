## Mode：status（周报 / 月报 / 绩效周期总结）

**覆盖任意周期**：周报 / 双周报 / 月报 / 季度报告 / 绩效周期总结。时间跨度由用户指定或自动推断（默认上次 status report 至今）。

---

### 读者画像

- **经理 / 团队 lead**：最关心"进度是否符合预期 + 有没有风险"
- **跨部门 / 管理层**：最关心"交付了什么 + 对业务的贡献"
- **HR / 绩效评审**（长周期时）：最关心"做了什么 + 带来什么价值 + 量化指标"

**共同特征**：
- 时间宝贵，第一屏要给结论
- 不需要所有技术细节，但需要量化
- 关心"风险 / 阻塞"：下一步会不会出问题

### 必含章节

```
## TL;DR
<2-3 句：本周期状态（进展顺利 / 有风险 / 已交付 X）+ 核心数字>

## 周期信息
- 时间范围：<YYYY-MM-DD 至 YYYY-MM-DD>
- 报告类型：<周报 / 月报 / 季度 / 绩效>
- 对比基线：<上一周期 / 目标>

## 完成情况
<本周期真正完成的事，按重要性排序
 每条：
  - 一句话摘要
  - 量化（commits / issues / 功能 / 性能指标 / ...）
  - 锚点（commit hash / PR / 相关 .artifacts/）
  - 价值 / 影响（对用户 / 业务 / 团队）>

## 进行中
<本周期开始但未完成的事
 每条：进度百分比 + 预计完成时间 + 当前阻塞（若有）>

## 未开始 / 延后
<原计划本周期做但未开始 / 延后的事
 每条：原因 + 新计划>

## 风险与阻塞
<下一周期可能出现的风险 / 已经遇到的阻塞
 每条：描述 + 影响 + 当前应对>

## 下一周期计划
<本周期结束后接下来的事
 按优先级列 Top 3-5 项>

## 量化指标（若适用）
<按项目相关指标给出数字：
  - 代码量变化（+/-）
  - 测试数 / 覆盖率
  - 性能指标（p99 / 吞吐 / ...）
  - 可靠性（错误率 / 可用性）
  - 速度指标（PR throughput / 上线次数）>

## 相关
- Commits: <short hash 范围或列表>
- Releases: <若本周期有 release>
- Incidents: <若本周期有事故，引用 incident 报告>
- .artifacts/: <相关报告>
```

### 素材收集侧重

- **git log**：时间范围内的 commit（按类型分组：feat / fix / refactor / docs / ...）
- **git diff stats**：代码量变化
- **.artifacts/INDEX.md**：本周期内的所有 report 条目
- **相关 issue / PR**（若有 gh CLI 或用户补充）
- **用户补充**：业务价值描述、阻塞说明、外部事件（会议决定 / 合作方变化 / ...）

**时间范围推断**：
- 显式参数 > 上次 status report 至今（查 INDEX.md）> 最近 7 天（默认周报）> 1 个月 / 3 个月 / 完整年（按用户语境）

### 语气 / 深度

- **结论先行**：第一屏让读者知道"是否 on track"
- **量化**：每个成就 / 风险都要有具体数字
- **诚实**：不粉饰进度；延后就说延后，卡住就说卡住
- **避免流水账**：不是 commit 清单，是"做成了什么"
- **对价值的解读**：每项工作说"这意味着什么"，不只是"做了什么"

### ASCII 图示要求

- **进度时间轴**：按周 / 月展示完成项 + 阻塞
- **完成度概览**：计划 vs 实际的对比条形图
- **量化趋势**（绩效周期）：指标随时间变化
- **风险矩阵**（若有多个风险）：按概率 × 影响象限

### 禁忌

- ❌ "本周做了很多事" —— 没有量化
- ❌ "遇到一些问题" —— 没有具体说明
- ❌ 只列 commit 不列价值（读者不想自己读代码）
- ❌ 粉饰延后（"基本完成" 实际没完成）
- ❌ 把工作量等同于价值（写了 1000 行代码 ≠ 交付了价值）
- ❌ 无风险部分（永远有风险；没写 = 没思考过）
- ❌ 技术术语轰炸经理（除非经理是技术背景）
- ❌ 绩效周期报告里没有量化（绩效评审最怕"我做了很多"）
