## Mode：decision（决策记录 / 技术提案）

**覆盖时态**：
- **已决**（ADR-style）：记录"为什么做了这个决策"，给未来的自己 / 接手人
- **未决**（Proposal-style）：推销"为什么应该做这个决策"，给决策者 / 架构委员会

时态由计划 / 讨论内容自动推断；也可以同时覆盖（"这是我们决定的 + 以下是说服团队的理由"）。

---

### 读者画像

- **已决场景**：6 个月后回来读这份文档的自己 / 接手项目的新同事 / 写出类似代码的其他团队
- **未决场景**：有权拍板的人（tech lead / 架构委员会 / 决策者）+ 需要被说服的同事

**共同特征**：
- 有技术背景，但未必了解本项目细节
- 关心"为什么"远多于"是什么"
- 痛恨"我们决定采用 X"而没有理由的记录

### 必含章节

```
## TL;DR
<1-2 句：决策内容 + 核心理由>

## 背景（Context）
- 遇到什么问题 / 面对什么选择
- 约束是什么（时间 / 资源 / 技术栈 / 团队能力 / 外部依赖）
- 假设是什么

## 候选方案
<至少 2 个候选，每个：
  - 方案摘要
  - 优势
  - 劣势
  - 适用条件>
<若是已决场景，还必须列出"曾经考虑过但最终否决"的方案及理由>

## 决策
<明确写出选了哪个；若是未决场景，写"推荐 X"并注明待批准>

## 理由
<为什么选这个而不是其他：
  - 按哪些维度权衡（如复杂度 / 可维护性 / 性能 / 团队熟悉度）
  - 每个维度本方案如何胜出
  - 被否决方案输在哪>

## 代价与权衡
<选了这个会承担什么 / 放弃什么
 透明地承认代价，不粉饰>

## 后续影响
- 对架构的影响
- 对其他模块的级联影响
- 对团队的要求（是否需要学习新技术 / 新工具）

## 何时重新评估
<什么条件下这个决策应该被重新审视
 例："当 X 依赖升级到 Y 版本" / "当团队规模超过 N" / "当性能需求变化到 Z">

## 相关
- Blueprint plan: <会话中的 plan / 其他 .artifacts/ 文件引用>
- Commits: <short hash 列表>
- Issues / PRs: <若适用>
```

### 素材收集侧重

- 会话中的 pax plan（尤其是"关键决策记录"、"讨论摘要"、"候选方案对比"章节）
- 会话中的讨论内容（尤其是 --deep 多角色对抗的收敛路径）
- git log 中和本决策相关的 commit
- 用户补充的约束 / 假设 / 历史上下文

### 语气 / 深度

- **技术深入**，可以用术语
- **诚实**：承认代价，不回避劣势
- **中立**：决策已定不要演销售；未决场景才可以带推销感
- **精确**：每个声明有数据或引用支撑，不空谈"性能更好"

### ASCII 图示要求

按需使用：
- 候选方案对比矩阵（固定 4 维度 + 动态补充，和 pax 一致）
- 架构影响图（若决策影响架构 → 画前后对比或新架构图）
- 决策依赖链（若本决策依赖或阻塞其他决策）

### 禁忌

- ❌ "我们决定用 X"然后没有理由
- ❌ 只列选中的方案，不列被否决的（6 个月后别人不知道你们想过什么）
- ❌ 粉饰代价（"几乎没有缺点"—— 永远不可信）
- ❌ 用"业界最佳实践"作为理由（具体是什么实践？为什么适用本项目？）
- ❌ 决策理由和代价矛盾（如果代价这么大为什么还选）
- ❌ 未决场景说成已决 / 已决场景说成还在讨论（混淆时态会误导读者）
