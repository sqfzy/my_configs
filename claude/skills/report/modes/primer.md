# Mode: primer

理解并解释一个既有项目 —— 从**开发者角度**（想上手改 / 扩展代码的人）或**终端用户角度**（想使用这工具的人）或两者合一，产出一份能让读者**快速建立心智模型**的导览。

---

## 与相关 skill 的边界

| 你在做什么 | 去哪里 |
|-----------|-------|
| 写**会提交到 repo 的**架构文档 / README / CONTRIBUTING | `/pax --doc`（写入 repo，走 doc purpose 的流程） |
| 写 onboard 时的**私人理解笔记**，仅存 `.artifacts/` 备忘 | **本 mode** |
| 找代码问题 / 技术债 / 演进路线 | `/pax --review` |
| 记录为什么 onboard 时选了某条学习路径 / 某个心智模型 | `/report decision` |

**铁律**：primer 是**纯描述**，不评价、不给建议、不推演进。看到代码里有问题，止步于"标注 gotcha"，不写"应该改成 X"（那是 review 的事）。

---

## 参数

```
/report --primer [--audience dev|user|both] [<主题或补充素材>] [--auto]
```

- `--audience dev`：开发者视角（默认，若未指定）
- `--audience user`：终端用户视角
- `--audience both`：两个视角各一块，共享项目定位一节

---

## 读者画像

| audience | 画像 | 心智模型标准 |
|----------|------|-------------|
| dev | 要改代码 / 扩展 / debug 的工程师；可能是 3 个月后的自己 / 新接手的同事 | 读完能独立做小改动；知道核心抽象在哪、模块怎么依赖、怎么 build-test-run、已知陷阱在哪 |
| user | 要用这工具做事的人；不一定读代码 | 读完能独立完成常见任务；知道安装、典型用例、不支持场景、去哪查文档 |
| both | 两类人都有 | 同上，但一份文档同时覆盖，以共享"项目定位"开头 |

## 必含章节

### dev audience

1. **项目定位**（1-3 句）
   - 这是什么？解决什么问题？为谁而做？

2. **目录结构**（ASCII 树，必含）
   - 顶层目录 + 每个目录一句话职责
   - 不是 `tree` 原始输出 —— 省掉 node_modules / target / .git 之类

3. **模块依赖图**（ASCII，必含若 ≥ 3 个模块）
   - 关键模块之间的调用 / 依赖方向
   - 标出"枢纽模块"（被大量依赖的那个）

4. **关键抽象与契约**
   - 核心 trait / interface / 基类：名字 + 一句话契约 + 一个典型实现
   - 核心数据结构：字段 + 不变量
   - 内联关键代码片段（3-10 行，足够看懂就行）

5. **数据流 & 控制流**（至少一张 ASCII 图）
   - 主干业务的输入 → 处理 → 输出路径
   - 关键状态机（若存在）

6. **Build / Test / Run**
   - 从 clone 到跑起来需要几条命令？每条做什么？
   - 常见构建失败原因
   - 如何单独跑某类测试

7. **扩展点**
   - 想加一种新的 X（如新 codec / 新后端 / 新命令），要改哪几处？
   - 有 plugin / trait-based 扩展机制吗？

8. **已知陷阱 & 反模式**（标"gotcha"，不给改法）
   - 新人容易踩的坑
   - 代码里留下的历史包袱位置
   - 性能 / 并发 / 资源边界的隐含约定

9. **不覆盖的范围**
   - 这份 primer 没讲清的（规模太大 / 需要单独研读）
   - 这项目故意不解决的问题（避免读者期望错位）

### user audience

1. **它是什么 / 不是什么**
   - 一句话定位 + 一句话明确不做什么（降预期错位）

2. **适用场景**
   - 3-5 个"典型你会用它的情况"
   - 如果用户场景不在这里 → 他应该找别的工具

3. **安装**
   - 平台支持矩阵
   - 逐步命令（从依赖装起）
   - 验证安装成功的命令

4. **5 个典型用例**（含命令示例 + 预期输出）
   - 覆盖 80% 使用场景
   - 每个示例内联粘贴可执行，用户复制就能跑

5. **常见问题 / 不支持的场景**
   - FAQ 前 5 条
   - 明说哪些事情它不支持（避免踩坑）

6. **去哪查更多**
   - 官方文档链接
   - 社区 / issue tracker
   - 相关工具对比

### both audience

- 顶部共享一节：**项目定位**（两类读者都先看这个）
- 之后分两大块：`## 开发者指南` + `## 用户指南`，各含上述相应章节

---

## 素材收集侧重

相比其他 mode，primer 的素材来源**以读代码 / 读配置 / 读构建脚本为主**，不以 git log / .artifacts/ 历史为主。

**必须主动读**：
- `README.md` / `CONTRIBUTING.md` / `ARCHITECTURE.md`（若已存在，作为底稿）
- 构建文件：`Cargo.toml` / `package.json` / `pyproject.toml` / `xmake.lua` / `CMakeLists.txt` / `Makefile`
- 入口文件：`main.rs` / `index.js` / `__main__.py` / `cli.cpp` 等
- 关键 trait / class 定义（扫 `pub trait` / `class` / `interface` / `template`）
- 测试入口（从测试反推使用方式）
- CI 配置（`.github/workflows/` / `.gitlab-ci.yml`）
- LICENSE / CHANGELOG（版本现状）

**仓库规模 > 某阈值时**（粗略：> 50 文件 / > 10k LoC），不尝试全读：
- 从入口文件做**限深 BFS**（最多 2-3 层调用）
- 识别"枢纽模块"后只深读枢纽，不逐文件扫
- 在报告里明确写"本 primer 基于扫描深度 N，未深入到 X / Y"

## 铁律

- **纯描述，不评价**：见到坏设计 / 技术债 —— 标为 "gotcha" 不建议改法。评价归 review。
- **内联 + 锚点双轨**：所有引用的代码块内联 3-10 行 snippet + 附 `file.rs:42-48` 精确定位。不能只给一个 link，读者追链 3 次就不合格（§2.7 自包含）
- **动词精确**：不写"处理数据" / "做一些事" —— 具体："把入站 packet 按 flow-id 分流到对应 stream，丢弃超过 reasm buffer 容量的分片"
- **版本锚**：报告顶部必标 `Commit: <hash>` / `Inspected at: <date>` —— primer 会随代码过时，读者要能判断是否已失效
- **not-a-spec**：primer 描述的是"我读后理解到的"，有理解错误的风险。不要把它当权威规范。
- **承认边界**：读不全 / 不懂某处 —— 在"不覆盖的范围"显式承认，比写错更好

## ASCII 图示要求

primer 是 ASCII 图示**密集型**的 mode：

| 图示 | 条件 |
|------|------|
| 目录结构树 | dev audience 必含 |
| 模块依赖图 | dev audience 且项目 ≥ 3 模块时必含 |
| 主干数据流 | dev / user 都建议含（dev 版偏实现，user 版偏用户行为） |
| 状态机 | 项目有明显状态流转时必含 |
| 典型用例流 | user audience 必含（"用户输入 → 工具做什么 → 输出"） |

---

## 必含锚点

- **顶部**：Commit hash + 扫描日期 + 扫描深度简述
- **每个内联代码块**：精确的 `file:line_range`
- **每个模块 / 抽象**：其定义位置

没有这些锚点，primer 就是"某个大语言模型对这代码的印象"，无法审计、无法验证、无法检测过时。

---

## 禁忌

- ❌ 把 `tree` 输出直接粘贴当目录结构（含 node_modules 等噪声）
- ❌ 复述代码注释当理解（复述不是解读）
- ❌ "这段代码做了什么" 纯机械翻译（没加入为什么 / 与其他部分的联系）
- ❌ 把一个模块的作用写成"处理 X 相关的业务逻辑"（这不叫理解）
- ❌ 对着 README 抄一遍（README 已经说的不用再说一遍 —— 要补充 README 没讲的）
- ❌ 写"这部分以后可以优化为..." / "建议重构..."（越界 review）
- ❌ 读到一半猜测 —— 要么读懂要么在"不覆盖范围"里坦承

---

## auto 模式

- `--audience` 未指定时默认 `dev`（项目 onboard 的最常见场景）
- 扫描范围若未给 → 默认入口文件 + 枢纽模块（按导入关系 rank）
- 读不懂的地方**不猜测** —— 写入"不覆盖范围"并 `[待补充]` 标注
- 所有 `[待补充]` 在 Phase 6 汇总

## 集成点

- **输入**：用户给项目根路径或未给（当前目录）。可附加 "聚焦 X 模块" 缩窄范围。
- **输出衔接**：
  - 若扫出有待解问题 → 提示用户 `/report issue` 记下
  - 若扫出来发现可改项 → 提示用户 `/pax --review` 做正式审查
  - 若想把本 primer 固化成 repo doc → 提示 `/pax --doc` 基于此稿改写
