## 预定义角色库

| 代号 | 角色         | 核心偏好                               | 适用场景                     |
|------|--------------|----------------------------------------|------------------------------|
| R1   | 风险卫士     | 识别风险、边界条件、失败模式           | 架构决策、系统设计           |
| R2   | 极简主义者   | 反对一切不必要的复杂性                 | 代码设计、API 设计           |
| R3   | 性能狂热者   | 吞吐量 / 延迟 / 资源效率优先           | 性能敏感场景、底层优化       |
| R4   | 实用主义者   | 只接受实证支撑的方案，拒绝纯理论       | 技术选型、方案评估           |
| R5   | 第一性原理者 | 从根本需求出发，质疑所有假设           | 需求分析、产品决策           |
| R6   | 维护性倡导者 | 三年后的可读性和可维护性优先           | 代码审查、长期项目           |
| R7   | 用户代言人   | 终端用户体验为唯一标准                 | 产品/UX 决策                 |
| R8   | 激进创新者   | 无视当前约束，追求理论最优解           | 架构探索、技术前瞻           |

| R10  | 安全专家     | 安全性、隐私、合规为最高优先级         | 安全设计、数据处理           |

### 自定义角色

若预定义角色库无法覆盖议题需求，可根据议题临时创建角色。格式：

```
【自定义】<角色名>
核心偏好：<该角色的判断标准和优先级>
立场预期：<与其他角色的主要冲突点>
```

例如：在讨论游戏设计时可自定义"玩家体验设计师（关注沉浸感和叙事）"；
在讨论医疗系统时可自定义"临床医生代言人（关注工作流和实际可操作性）"。

### 选角要求

- 角色之间必须存在**真实的价值观冲突**，不能选立场相近的组合
- 优先构建多维张力（保守 vs 激进、短期 vs 长期、性能 vs 可维护性等）
- 说明每个角色的选择理由及预期与其他角色的冲突点

### 发言格式

每轮之间用分隔线和轮次标题分隔，每个角色的发言用缩进区块呈现，确保终端中易读：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  第 N 轮
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

┌─ R1 风险卫士 | 反对 ─────────────┐

  论点：...

  反驳 R3：...   ← 第 2 轮起必填
  联盟 R6：...   ← 可选
  立场修正：...  ← 仅最后一轮必填

└──────────────────────────────────┘

┌─ R2 极简主义者 | 支持 ──────────┐

  论点：...

  反驳 R1：...

└──────────────────────────────────┘
```

**格式要求**：
- 每轮之间用 `━` 粗线分隔，标注轮次编号
- 每个角色用 `┌─ ... ─┐` / `└─┘` 盒子包裹，盒子标题含角色代号、名称、立场标签
- 角色盒子之间空一行
- 论点与反驳之间空一行，保持段落感
- 禁止所有角色的发言挤在一起无分隔

### 质量要求

- 每个角色须有明确立场，"一方面…另一方面…"而无结论是失败模式
- 反驳必须指向对方的具体论点，不得打稻草人
- 轮数较多时，中间轮允许聚焦子议题深入讨论，不必每轮都覆盖全局
