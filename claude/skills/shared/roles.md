### 目的 → 推荐默认角色组合（pax 用）

当 `--deep` 触发且用户未显式列角色时，按 pax 目的选以下组合（默认 5 人，`--roles: N` 可覆盖）：

| pax 目的  | 推荐角色（按优先序）                             | 主冲突张力                  |
|-----------|--------------------------------------------------|-----------------------------|
| `feat`    | R14 架构师 / R2 极简 / R1 风险 / R8 激进 / R6 维护 | 激进 vs 保守、扩展 vs 极简  |
| `fix`     | R11 怀疑 / R1 风险 / R13 测试 / R5 第一性 / R6 维护 | 根因严谨 vs 修得快          |
| `reshape` | R14 架构师 / R6 维护 / R8 激进 / R11 怀疑 / R1 风险 | 保行为 vs 重设计            |
| `upgrade` | R4 实用 / R1 风险 / R9 成本 / R6 维护 / R11 怀疑   | 彻底迁移 vs 兼容层          |
| `review`  | R11 怀疑 / R10 安全 / R13 测试 / R6 维护 / R3 性能 | 覆盖面广（审查本就多视角）  |
| `test`    | R13 测试 / R11 怀疑 / R2 极简 / R1 风险           | 覆盖充分 vs 维护成本        |
| `bench`   | R3 性能 / R4 实用 / R9 成本 / R11 怀疑            | 极致优化 vs ROI             |
| `doc`     | R16 文档 / R7 用户 / R6 维护 / R2 极简            | 详尽 vs 简洁                |
| `ship`    | R1 风险 / R10 安全 / R11 怀疑 / R9 成本 / R13 测试 | 发布稳健                    |
| `loop`    | R11 怀疑 / R1 风险 / R4 实用 / R9 成本 / R12 系统 | 终止条件严谨 / 二阶效应     |

**追加规则**：
- 若议题明显触及某维度（数据处理 → R10 安全；高并发服务 → R3 / R15；跨模块改造 → R12 / R14），**把该维度代言人追加进默认组合**，不拘泥于表格。
- 若议题完全不涉及某维度，可从组合里去掉对应角色（例如 `doc` 里 R10 安全可省）。

### 自定义角色

若预定义角色库无法覆盖议题需求，可根据议题临时创建角色。格式：

```
【自定义】<角色名>
核心偏好：<该角色的判断标准和优先级>
立场预期：<与其他角色的主要冲突点>
```

例如：游戏设计 → "玩家体验设计师（关注沉浸感和叙事）"；医疗系统 → "临床医生代言人（关注工作流和实际可操作性）"。

---

### 选角要求

- 角色之间必须存在**真实的价值观冲突**，不能选立场相近的组合
- 优先构建多维张力（保守 vs 激进、短期 vs 长期、性能 vs 可维护性等）
- 说明每个角色的选择理由及预期与其他角色的冲突点
- 默认 5 人（与 pax `--roles: N` 默认一致）；复杂度高或跨多生产级维度时可扩至 7 人

---

### 发言格式

每轮之间用分隔线和轮次标题分隔，每个角色的发言用 ASCII 盒子包裹，确保终端易读：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  第 N 轮
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

┌─ R1 风险卫士 | 反对 ─────────────┐

  论点：...

  反驳 R3：...   ← 第 2 轮起必填
  联盟 R6：...   ← 可选
  立场修正：...  ← 仅最后一轮必填

└──────────────────────────────────┘

┌─ R2 极简主义者 | 支持 ──────────┐

  论点：...

  反驳 R1：...

└──────────────────────────────────┘
```

**格式要求**：
- 每轮之间用 `━` 粗线分隔，标注轮次编号
- 每个角色用 `┌─ ... ─┐` / `└─┘` 盒子包裹，盒子标题含代号、名称、立场标签
- 角色盒子之间空一行
- 论点与反驳之间空一行，保持段落感
- 禁止所有角色发言挤在一起无分隔

---

### 质量要求

- 每个角色须有**明确立场**（支持 / 反对 / 有条件支持）；"一方面…另一方面…"而无结论是失败模式
- 反驳必须**指向对方的具体论点**，不得打稻草人
- **涉及代码的论点必须引用具体位置**（文件:行号 / 符号名）——与 pax Phase 1.5 保持一致
- 轮数较多时，中间轮允许聚焦子议题深入讨论，不必每轮都覆盖全局
- 最后一轮每个角色**必须声明立场修正**（坚持原立场也要明说）
- 收敛判据：达到 pax 定义的终止条件（复杂度对应的"连续 1-2 轮无新论点"）即结束，不硬凑轮数

---

### 反模式（任一命中即讨论失败）

- ❌ 所有角色立场相近 —— 选角失败，缺乏对抗
- ❌ 讨论结束未收敛但也未把分歧写入计划 —— 违反 pax Phase 1.5 "未收敛必须写入计划由用户裁决"
- ❌ 角色发言不引用具体代码位置（议题涉及代码时）
- ❌ 最后一轮无"立场修正"声明
- ❌ 所有角色发言挤在一起无分隔 / 盒子缺失
- ❌ 反驳打稻草人（歪曲对方论点）
- ❌ 用"你说得都对"式的和稀泥代替真实反驳
- ❌ 把议题涉及的生产级维度完全漏掉（例如讨论 auth 改造但无 R10 安全代言）
