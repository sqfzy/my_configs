## 最终成果可视化原则

> **铁律：任何带计划/设计性质的输出，必须让用户"看见"做完后的样子，而不仅是"听到"过程的步骤。**

计划/设计的核心价值不是罗列要做什么，而是把**最终形态**清晰地呈现出来——结构、关系、数据流、状态变化。文字描述容易留下歧义，图示强制把每个组件的位置和关系明确化。

### 必须包含的成果维度

凡是产出 Plan / 设计文档 / 重构方案 / 迁移方案 / 收敛结论的章节，至少覆盖下列**适用项**（按场景取舍，不必全选）：

- **结构形态**：文件 / 模块 / 目录的最终布局（目录树或框图）
- **依赖关系**：组件之间的调用方向、依赖层次（ASCII 框图 + 箭头）
- **数据流向**：核心数据在系统中的流动路径（流水线图或时序图）
- **关键交互**：用户↔系统、模块↔模块的交互序列（必要时用泳道/时序图）
- **状态对照**：改造类工作必须给出**现状 vs 目标**的并排对比，让差异一目了然

### ASCII 图示偏好

**优先使用 ASCII 图示**，原因：
- 视觉化的结构远比段落文字易于理解
- ASCII 在 plan mode、终端、各种 Markdown 渲染器、Git diff 中都能正常显示，无需额外渲染依赖
- 强制作者把布局和关系想清楚——画不出来通常意味着没想清楚

只有当图示不可避免地超过 ~40 行、或维度超过 3 个时，才退回到表格 + 文字。

### 推荐图示模板

**目录结构**：

```
src/
├── core/
│   ├── engine.rs      ← 核心调度器（纯逻辑，无 I/O）
│   └── state.rs       ← 共享状态
├── adapters/
│   ├── http.rs        ← HTTP 适配层
│   └── db.rs          ← 持久化适配
└── lib.rs             ← 公共 re-export
```

**模块依赖**（箭头表示"依赖"或"调用"方向）：

```
┌──────────┐     ┌────────────┐
│   CLI    │────▶│   Engine   │
└──────────┘     └─────┬──────┘
                       │
            ┌──────────┼──────────┐
            ▼          ▼          ▼
       ┌────────┐ ┌────────┐ ┌────────┐
       │ Parser │ │ State  │ │ Output │
       └────────┘ └────────┘ └────────┘
```

**数据流**（流水线）：

```
Input ──▶ [Parser] ──AST──▶ [Optimizer] ──IR──▶ [Codegen] ──▶ Output
                              │
                              ▼
                          [Validator]
```

**改造前后对比**：

```
   现状（耦合）                         目标（分层）
   ─────────────                       ─────────────
   ┌──────────────┐                    ┌──────────────┐
   │  mod_a       │                    │   api/       │ ← 对外接口
   │  ├─ I/O      │                    ├──────────────┤
   │  ├─ 业务逻辑 │       ───▶         │   core/      │ ← 纯逻辑
   │  └─ 状态管理 │                    ├──────────────┤
   └──────────────┘                    │   adapters/  │ ← I/O 隔离
   ┌──────────────┐                    └──────────────┘
   │  mod_b       │
   │  └─ ...      │
   └──────────────┘
```

**时序 / 交互**（轻量泳道）：

```
User           CLI            Engine          Storage
 │              │               │                │
 │──"run"─────▶│               │                │
 │              │──parse───────▶│                │
 │              │               │──load─────────▶│
 │              │               │◀────data──────│
 │              │◀──result─────│                │
 │◀──output────│               │                │
```

### 反模式（必须避免）

- ❌ 只列"步骤 1: 创建 X，步骤 2: 创建 Y"，不展示**最终结构长什么样**
- ❌ 把图示放在最后作为"附录"——应该在每个关键决策章节就地出现
- ❌ 用纯文字描述"模块 A 调用模块 B 调用模块 C"——画出来
- ❌ 改造类工作只画"目标"不画"现状"——用户判断不了改动幅度
- ❌ 图示与文字描述不一致——以图示为准，文字应当是图示的注解

### 适用 skill

所有带计划/设计性质的章节，包括但不限于：
- `/blueprint` Phase 3 的最终 plan
- `/design` Phase 2 的设计方案（含候选方案）
- `/discuss` Step 3 的收敛输出
- `/refactor [breaking]` 的迁移方案
- `/cleanup` 的重设计方案
- `/migrate` 的迁移路径
- `/evolve` 每轮 design 子流程的产出
