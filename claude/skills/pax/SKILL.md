# /pax

ASCII 可视化原则：!`cat ~/.claude/skills/shared/ascii-viz.md`

目标：$ARGUMENTS

---

## 定位

`/pax` 是**事前阶段的唯一入口**。职责：在动手之前把目的、假设、方案、架构聊清楚，产出一份**可直接施工的计划**。

**计划即施工契约**：pax 产出的计划足够具体，审批通过后 AI 就按计划推进——用户说"完成整个计划"或"执行阶段 N"即可。pax 不只是"想清楚"，还定义"怎么一步步做"。

**输出形态**：plan mode 里的最终计划，经 `ExitPlanMode` 呈给用户审批。**不落盘**——计划是会话本地契约。如需永久留存决策，审批后用 `/report decision`。产物统一由 `/report` 负责。

**阶段边界**：
- **事前（本 skill）**：只读、只想、只讨论，进 plan mode
- **施工**：按 pax 审批通过的计划推进，不需要独立 skill
- **事后（`/verify`）**：审查 / 测试 / 平时检查 / 发布闸
- **产物沉淀（`/report`）**：永久留存

---

## 核心要求：计划必须面向生产级落地

用户重视代码达到生产级别——这一要求必须贯穿事前规划和施工的每一步。计划里对下列维度的考量必须显式存在（具体展开深度由议题决定）：

**正确性与边界 / 并发与资源 / 可观测性 / 安全 / 性能 / 可维护性 / 测试策略 / 文档**

议题规模小时可以合并几项；但**绝不能完全跳过**。计划里看不到生产级考量，就是不合格的计划。

---

## 二维设计：目的 × 框架能力

### 目的（purpose）—— 决定 plan 骨架和施工约束

目的决定要规划什么事、要覆盖哪些维度、施工时如何推进（分阶段、风险、验证关卡、回滚点）。pax 覆盖**从事前规划到事中施工到事后核查的全阶段**，所以目的也分三类：

**施工类（产出计划 + 施工修改代码）**：

| 目的 | 适用 |
|------|------|
| `feat`    | 新功能 / 新模块 / 新项目 |
| `fix`     | 修复已知问题 |
| `reshape` | 代码结构改造（小幅重构 → 激进重设计全谱系） |
| `upgrade` | 依赖 / 版本 / 框架 / 构建系统迁移 |

**核查类（产出报告 / 发现，默认不改代码）**：

| 目的 | 适用 | 是否生成代码 |
|------|------|-------------|
| `review`  | 局部代码审查（diff / 模块 / PR）+ 全局项目审视 + 演进规划元计划 | ❌ 只读，产出发现和演进路线 |

**生成类（产出并写入对应产物）**：

| 目的 | 适用 | 产物 |
|------|------|------|
| `test`       | 测试覆盖盲区分析 + 生成 | 测试代码（必然生成，不可选） |
| `bench`      | 性能基线 / profile / 对比 / 优化 | 基线数据 / profile 报告 / 对比；--optimize 会改代码 |
| `doc`        | 文档生成 / 一致性核查 | 文档（必然生成，不可选） |
| `experiment` | 受控实验 / 假设检验 / A/B / 消融 / 带显著性的对比 | 预注册假设 + 数据 + 统计结论（p + CI + effect size） + 复现包；需要统计严谨性而非单维度测量时用此而非 bench |

**组合类**：

| 目的 | 适用 | 内部组成 |
|------|------|---------|
| `ship` | 发布闸 —— 准备 release / 打 tag / push | 依次跑 review + test + bench + doc + 提交 + tag + push，每个 Gate 不可跳过 |
| `loop` | 周期性重复 —— 自主演进 / 质量打磨 / 测试逼近 / 性能逼近 / 文档跟进 等"跑一晚"场景 | 规划循环契约（BODY 伪代码 + UNTIL 终止 + LENS 镜头 + 失败策略 + 资源边界）；施工期按契约反复执行（直接模式或 subagent 模式），每轮可调任意 pax 子目的 / `/report` / `/script` / shell |

每个目的**只定义自身独有的关注点、施工约束、陷阱和反模式**（见 `purposes/<purpose>.md`）。通用骨架由本 SKILL.md 统一规定。

### 框架能力（capability）—— 横切增强

- `--deep`：Phase 2 前插入多角色对抗讨论子阶段
- 对比候选方案不作为独立模式——在各目的的 Phase 2 里按需展开（议题真有候选时用矩阵对比）

### `--auto` 语义

**激活判据**：`$ARGUMENTS` 字面含 `--auto` 才激活，无则走默认互动路径。不看宿主 auto mode、不看上下文、不做推断。

激活后语义：**全自动、无需任何批准、自动处理任何情况**——用于嵌入 `/autopilot`、`/gsd:autonomous`、后台任务等无人值守工作流。

**自动化覆盖范围**：

- **Phase 0 EnterPlanMode**：**不调用**。plan mode 只为"给用户审批"存在，auto 下无审批，进 plan mode 反而在 Phase 4 触发宿主级对话框阻塞。auto 下靠 AI 自律在 Phase 0-3 期间不做任何文件写操作。
- **Phase 0.5 假设列举**：按保守解释自动锁定（通常偏向"多做不多错、少破坏现有"），不等 confirm
- **Phase 1.5 deep 多角色讨论**：
  - 未收敛时取多数派方案；平票取更保守方案
  - 被否决方案写入"讨论摘要"作为警告
- **Phase 2 逐维度对话**：每维度直接采纳推荐项
- **Phase 2.2 候选方案对比**：自动选推荐项
- **Phase 4 ExitPlanMode**：**不调用**。因为未进 plan mode，无状态要退出。计划以普通消息 Markdown 输出（开头标 `⚠️ --auto 模式生成，未经审批直接进入施工`），然后直接进 Phase 5。
- **施工阶段**：Phase 5 不等用户触发词，直接进入施工，按计划顺序跑完所有阶段

**特殊情况的处理**（全部"警告后继续自主决策"，不停下等用户）：

| 情况 | auto 模式的处理 |
|------|----------------|
| 某假设置信度很低 | 警告 + 按保守解释继续 |
| 候选方案没有明显推荐 | 警告 + 选"复杂度最低、风险最小"的那个 |
| deep 未收敛 | 警告 + 多数派；平票取更保守 |
| 施工中某阶段测试失败 | 警告 + 根据失败类型自主修复：若是明显代码错误直接修；若是计划缺陷自动补丁并记入"施工日志"；若无法自主修复，按回滚点回滚该阶段并标红该阶段为"需人工介入"后继续剩余阶段 |
| Bench 退化 ≥ 5% | 警告 + 记录退化详情；不回滚；继续剩余阶段 |
| 行为破坏（保行为 reshape 时出现） | 警告 + 标记该阶段"行为偏离"，继续剩余阶段，最终汇报 |
| 发现计划无法实施的根本缺陷 | 警告 + 在会话内重新 pax（仍 auto），用修正后计划继续；若再次失败，警告 + 终止并汇报 |
| 用户提供的描述不够清晰 | 警告 + 按最保守 / 最常见解读继续，不反问 |

**auto 模式不停下**的兜底原则：
- 能自动决策就自动决策
- 决策点的依据永远倾向"保守 / 可回滚 / 影响最小"
- 一切异常都以**警告 + 继续**的形式处理，最后在汇报中汇总
- 只有"物理上无法继续"（如无法修复的编译错误、无法写入文件系统、用户主动打断）才会真正停下

**最终交付**：
- 计划显著标注 `⚠️ --auto 模式`
- 施工完成后给出总汇报：完成阶段 / 警告项清单 / 行为偏离 / bench 变化 / 需人工复核的项
- 计划的假设、决策、异常处理全部存入"施工日志"章节，供事后审阅

## 流程

```
Phase 0     EnterPlanMode + 上下文收集
Phase 0.3   需求澄清（WHAT 不清 → 非 auto 必问）
Phase 0.5   假设列举（HOW-前提，所有目的强制）
Phase 1     目的分流 + 加载目的骨架
Phase 1.5   [--deep] 多角色对抗讨论（若启用）
Phase 2     逐维度对话（通用骨架 + 目的独有维度）
Phase 2.5   关键决策用户确认（HOW 争议 → 非 auto 必问）
Phase 3     一致性检查 + 生产级自查 + 可施工性自查
Phase 4     ExitPlanMode 呈交计划
Phase 5     下一步提示（如何启动施工）
```

**"问"的两个阶段**：Phase 0.3 问 **WHAT**（用户到底想做什么），Phase 2.5 问 **HOW**（设计选型 / 破坏性决策）。两者不可合并 —— WHAT 没问清就进设计是南辕北辙，WHAT 清楚后再把 HOW 的决策权交还用户。

## Phase 0: 进入 plan mode + 上下文收集

**第一句输出必须是参数扫描声明**（不得省略、不得改写、不得加修饰词）：

```
▶ --auto: 否 —— 参数 "<原样复述 $ARGUMENTS>" 不含 --auto
```
或
```
▶ --auto: 是 —— 参数 "<原样复述 $ARGUMENTS>" 含 --auto
```

**禁用任何形式的继承论**。以下措辞直接判违规，遇到立刻判"否"重来：

- ❌ "会话继承 auto 模式" / "session inherits auto"
- ❌ "沿用 / 延续 / 继承上轮的 auto"
- ❌ "既然在 auto 里" / "上下文是 auto"
- ❌ "宿主是 auto mode 所以..."
- ❌ 参考系统 reminder / 历史消息判断 --auto

宿主级 Auto Mode、上轮对话、系统 reminder、Claude Code 权限模式 —— **都不是参数**。判据只认 `$ARGUMENTS` 字符串的字面子串匹配。

然后：

- **声明为"否"**：第一件事调用 `EnterPlanMode`，此点到 `ExitPlanMode` 之间只可读取
- **声明为"是"**：不调 `EnterPlanMode`（否则 Phase 4 ExitPlanMode 对话框卡死），靠自律保持 Phase 0-3 只读

按议题需要阅读相关代码、配置、关键接口。读什么、读多少由议题决定。

---

## Phase 0.3: 需求澄清（非 auto 必做，auto 跳过）

**定位**：在做任何设计前，先确认用户描述的**核心需求**是否足够明确。遇到**需求级歧义**（关于 WHAT，不是关于 HOW）必须立即问，不准假设着继续。

**必问**（命中即问）：

| 类别 | 例 |
|------|----|
| 核心概念定义不清 | "连接三路" —— 三条网络连接？IPC 通道？fiber paths？ |
| 关键行为描述模糊 | "发生断连则..." 具体是重连、告警、退出、还是重置状态？ |
| 核心对象范围不明 | "所有 path" —— 代码里有 5 种 path，指哪些？ |
| 成功标准不具体 | "必要的统计信息" —— 哪些字段是必要的？ |
| 动词多义 | "合并" —— 物理合并文件？合并接口？合并功能点？ |

**不问**（留给 0.5 / 2 / 2.5）：
- 技术选型 / 实现路径（HOW）
- 环境与代码现状的前提（HOW 准备）
- 有 2+ 候选但 WHAT 本身清楚

**判据**：若你在 Phase 0.5 打算把某条写成"置信度：低 —— 用户意图不确定"，它其实应该在 Phase 0.3 问掉 —— **用户意图的歧义不该"假设着继续"**。

**问法**：

- `AskUserQuestion` 打包 1-5 题。超过 5 题说明议题定义本身太松，应该告诉用户"需求太模糊，请先把 XX 讲清楚"再重新进 /pax。
- 每题格式：**原句里的歧义短语** + **AI 给出的 2-3 种典型解读** + **"哪种"**（无推荐）
- 不要反问"你是什么意思" —— 要主动列解读让用户选择

**auto 模式**：跳过本阶段。所有歧义按最保守 / 最常见解读锁定，并在 Phase 0.5 写为低置信度假设（**但 auto 模式下用户无法 confirm**，风险自担）。

## Phase 0.5: HOW 假设列举（所有目的强制）

**铁律：每次 plan 都必须走这一步**。范围限定在 **WHAT 已确认后关于 HOW 的前提**：环境 / 代码现状 / 工具链 / 依赖选型默认值。

**什么写这里**：
- 环境类：假设运行在 Linux x86_64 / 用户有 root / python ≥ 3.10
- 代码现状类：假设项目已有 tracing setup / 假设 `logger` 符号来自 `utils::log`
- 工具链类：假设 build 用 cargo workspace / 假设测试框架是 pytest
- 非关键默认：假设沿用既有目录结构 / 假设不需要单独的迁移脚本

**不写这里**：
- 用户意图的不确定（归 Phase 0.3 去问）
- 关键设计决策（归 Phase 2.5 去问）
- 明显事实（看过代码就知道的，不算假设）

每条标注：内容 / 置信度（高 / 中 / 低）/ 若错误的影响范围。非 auto 模式呈给用户 confirm / reject / clarify；低 / 中置信度的项进 Phase 2.5 问卷。auto 模式按保守解释锁定，写入"假设"章节作为契约。

---

## Phase 1: 目的分流 + 加载目的骨架

1. 确认目的（显式指定或自动推断）
2. **铁律**：立即 `Read ~/.claude/skills/pax/purposes/<purpose>.md`，**不允许跳过**。该文件定义目的独有的必含章节 / 陷阱 / 反模式 / 图示要求——不读就出不了合格 plan。文件不存在视为目的未实现，报错并列出支持的目的。"我已经知道 <purpose> 该怎么做" 是自信幻觉，骨架会更新，训练数据不一定最新。
3. 根据 `--deep` 决定是否走 Phase 1.5

骨架只写**独有**内容；通用维度 / 图示 / 实施计划格式 / 施工纪律全在本 SKILL.md 的 Phase 2 与 Phase 4。

## Phase 1.5: 多角色对抗讨论（`--deep`）

**原则**：复杂度越高讨论越深；必须收敛到可落地共识；未收敛分歧写入计划由用户裁决。

### 复杂度与轮数

| 复杂度 | 特征 | 最低轮数 | 终止条件 |
|--------|------|---------|---------|
| 低 | 方案基本明确 | 3 | 连续 1 轮无新论点 |
| 中 | 2-3 个合理方案 | 5 | 连续 1 轮无新论点 |
| 高 | 方案根本分歧、约束模糊 | 8 | 连续 2 轮无新论点 |

`--rounds: N` 覆盖自动判断。

### 角色选择

!`cat ~/.claude/skills/shared/roles.md`

角色组合按议题特征动态选定，不预设。

### 讨论要点

- 每轮发言含：核心论点（指向具体决策）+ 反驳（中间轮必须）
- 最后一轮：每个角色声明立场修正
- 涉及代码的论点必须引用具体位置

### 未收敛处理

- **默认**：询问用户裁决
- **`--auto`**：取多数派；平票取更保守的；否决方案写入"讨论摘要"

**收敛后强制走 Phase 2**：把共识翻译为可执行形态。不允许讨论结束后直接 ExitPlanMode。

## Phase 2: 逐维度对话（通用骨架 + 目的独有维度）

Phase 2 是 pax 的核心。通用骨架和所有目的共享；Phase 1 读入的目的骨架文件补充该目的独有的维度和约束。

### 2.1 通用关注维度（每个目的都要覆盖）

所有目的都必须在计划中明确以下通用维度，具体展开深度按议题规模调整：

- **定位与边界**：要解决什么问题、in/out scope、成功标准
- **现状理解**：相关代码 / 接口 / 数据 / 约束的当前状态（feat 目的下现状可能为空）
- **目标形态**：完成后代码应该是什么样（结构、接口、行为）
- **效果三件套**（施工类 feat/fix/reshape/upgrade / ship / bench --optimize 必含；只读 review 跳过；test/bench/doc/experiment 默认跳过）：
  - **最终效果（用户视角）**：施工完用户**能直接观察到**什么变化？用"用户会发现..."句式，不允许写"重构了 X 函数"这类纯实现描述。
    - **表达风格**：**直白 / 可视化 / 易读**
      - 直白：短句 + 主语是"用户"或具体动作，不用"实现了..."、"优化了..."这类被动 / 工程腔
      - 可视化：优先 before/after ASCII 对照 / 命令行终端示例（带 `$ prompt` + 输出）/ 屏幕响应描述；禁用大段散文
      - 易读：列表分点 > 段落；每点一行；内嵌具体数值（"启动 3s → 0.8s"）而非形容词（"启动更快了"）
    - feat: 用户现在能做什么新事？给最短可跑的用法示例（命令 / API 调用）
    - fix: 原来哪个现象消失了？怎么验证它真消失（复现步骤 → 修复后期望结果）
    - reshape: 对外行为不变是核心承诺；展示前后结构对照（ASCII）
    - upgrade: Breaking changes 清单 + 升级方需要做的具体步骤
  - **影响范围（工程视角）**：文件 / 模块 / 调用方 / 公共 API / 外部协议 / 部署 / CI / 依赖
  - **不影响的（负向承诺）**：**明确列出保证不变**的接口 / 行为 / 性能阈值 / 文档位置 / 配置键。施工期间违反这里的承诺 = 施工事故，停下汇报。
- **生产级 8 维度**（必须有着落，不能跳过）：
  - 正确性与边界
  - 并发与资源
  - 可观测性
  - 安全
  - 性能
  - 可维护性
  - 测试策略
  - 文档
- **风险与回滚**：可能出什么问题、怎么发现、怎么回滚
- **分阶段实施**：每阶段独立可编译、可验证、可提交

目的骨架文件会在此基础上**追加独有维度**（如 fix 必须有"根因分析"、reshape 必须有"不变量清单"、upgrade 必须有"Breaking changes 清单"、review 全局模式必须有"Tier 分层"、loop 必须有"BODY / UNTIL / LENS / STATE / FAILURE / BOUNDS / MODE"）。

### 2.2 候选方案对比（按需展开）

若某个决策点真的有 2+ 合理候选（选型 / 架构 / 迁移路径等），展开矩阵对比：
- 固定 4 维度：复杂度 / 可维护性 / 性能 / 扩展性
- 按议题动态补充相关维度（宁少勿多）
- 单元格用"低/中/高"或定量指标，哪个更有信息量用哪个
- 附推荐理由

没有真正的候选时不强走矩阵。

### 2.3 对话模式（每维度统一）

```
## <维度名>

### Q1: <问题>
选项：
  A. ... — <优劣>
  B. ... — <优劣>
推荐：<X>，因为 <理由>

请逐一回答，或说"按推荐"。
```

**推荐哲学**：追求设计最优，一步到位，而非改动最小或兼容现有。**迁移成本是施工阶段的问题，`/pax` 的规划阶段只关注"最终应该是什么样"**。

**auto 模式**：不等用户，直接采纳推荐项。

### 2.4 通用图示要求（所有目的都要满足）

所有计划至少包含以下图示之一（按议题规模和目的补齐）：

- **结构**：目录树 / 模块组织（静态布局）
- **关系**：模块依赖 / 调用方向 / 数据传递
- **动态**：关键流程控制流 / 状态机 / 时序
- **对照**：改造类场景的"现状 vs 目标"并排对比

目的骨架可能追加特定图示的强制要求（如 reshape 必含"现状 vs 目标"、upgrade 必含"迁移路径时序"）。

**判据**：用户看完图示应能回答"做完后长什么样？数据怎么流？改了哪些地方？"——回答不上就是图不够。

### 2.5 实施计划章节必含字段（所有目的统一）

**计划即施工契约**——任何目的的最终计划的"实施计划"章节必须足够具体使 AI 能直接照做。每阶段必须包含以下字段：

```
### 阶段 N: <标题>
- 目标：<此阶段要达成什么>
- 具体动作：<要改哪些文件 / 做哪些事，列表形式，足够细使 AI 能直接执行>
- 交付物：<此阶段结束时能看到的具体产出>
- 验收标准：<可客观验证的条件：测试通过 / 接口可调 / 无回归 / bench 不退化 …>
- 施工纪律：<每步小提交 / 每步验证 / 每阶段末 bench …>
- 风险等级：<低 / 中 / 高>
- 风险点 & 预警信号：<做到哪一步要警觉、看到什么信号要停下>
- 依赖的前置阶段：<阶段号或"无">
- 回滚点：<若此阶段失败如何回滚到干净状态>
- 生产级自检要点：<本阶段必须检查的生产级维度>
```

**每阶段必须独立可编译、可验证、可提交**。不允许出现"阶段 3 完成前代码都不可编译"这种跨越式阶段。

**施工纪律的通用底线**（各目的骨架可强化）：
- 小步提交，每步可独立回滚
- 每步改动后立即编译 + 运行相关测试
- 接口变更同步更新所有调用方

## Phase 2.5: 关键决策用户确认（非 auto 必做，auto 跳过）

Phase 0.5 / Phase 2 过程中累积的"必须由用户拍板"的问题，在 ExitPlanMode 之前**显式问出来**。不问 = pax 代用户做关键决策 = 不合格。

**铁律**：用户之前说过"按推荐"**不免**此阶段。"按推荐"只覆盖 Phase 2 的逐维度推荐，不覆盖本阶段列出的关键争议。

**必问项**（任一命中即入问卷）：

| 类别 | 触发条件 |
|------|---------|
| 低置信度假设 | Phase 0.5 标"低"或"中"的假设 |
| 争议决策 | 2+ 候选方案矩阵对比后 tradeoff 接近 / 没有明显推荐 |
| 破坏性选择 | 删除或重命名公共 API / 去除既有接口 / 换核心依赖 |
| 目的独有关键岔路 | purposes/<purpose>.md 里标为"关键"的决策点（如 reshape 的改造幅度、upgrade 的 breaking change 策略、feat 的核心抽象选型） |

**不问**（直接用默认或推荐）：
- 沿用项目既有 convention 的低风险默认
- Phase 2 逐维度 Q&A 里推荐明显的项（"按推荐"能覆盖）
- 可事后调整的细节（命名微调 / 目录布局 / 日志格式 / 注释风格）

**问法**：

- 用 `AskUserQuestion` 工具**一次性打包** 1-5 题（超过 5 题说明议题太大，拆掉 Phase 2.5 回 Phase 1 缩范围）
- 每题给**原始选项**（A / B / C / 自己写），并列各自 tradeoff
- **不暗示推荐** —— 此处不是"给我们的推荐盖章"，是用户做真实选择
- 每题标注"若忽略此题会怎样"（便于用户判断重要性）

**用户答后**：回到 Phase 2 把答案折进"假设"与"关键决策记录"章节，再走 Phase 3。
- 不引入未在计划中声明的改动（若发现计划缺陷，停下回到 `/pax` 修正）
- 遵守生产级代码标准（8 维度）
- 若项目有 benchmark 基线，关键阶段后对比，退化 ≥ 5% 停下评估

## Phase 3: 一致性检查 + 生产级自查 + 可施工性自查

全局自检：

**一致性**：
- **骨架已读 + 已落地**：计划里必须出现目的独有章节（feat 的技术选型 / fix 的根因 / experiment 的预注册 等），且用词与 `purposes/<purpose>.md` 对齐。若本节空缺或只有通用内容 → Phase 1 的 Read 被跳过，**立即回 Phase 1 补读**，重做计划。
- **效果三件套齐备**（施工类 / ship / bench --optimize 必检）：
  - 最终效果：
    - 视角：用"用户会发现..."句式，不允许"重构了 X 函数" / "改了 Y 模块"的工程描述
    - 风格：**直白 / 可视化 / 易读** —— 短句、列表分点；有 ASCII 前后对照或命令行示例；具体数值（"启动 3s → 0.8s"）代替形容词（"变快了"）。整段散文 / 形容词堆砌 → 重写
  - 影响范围：文件 / 模块 / 调用方 / 公共 API / 部署 / CI / 依赖 是否列全？
  - 不影响的：负向承诺是否显式列出？**空缺视为未思考，不算合格**。
- **Phase 0.3 已问**（非 auto 必检）：原始输入里的需求级歧义（核心概念 / 行为 / 范围 / 成功标准 / 多义动词）是否都问过？若计划的"定位与边界"里出现对用户原话的**额外解读**而没有对应的 Phase 0.3 问答支撑 → Phase 0.3 被跳过，**立即回 Phase 0.3 问用户**。
- **Phase 2.5 已问**（非 auto 必检）：所有命中触发条件的决策是否都进过 AskUserQuestion？用户答案是否已折进"假设"与"关键决策记录"？若假设章节里还留着"低置信度"标签 / 关键决策记录里出现"pax 自主裁定"、"默认选 X" 等字样 → Phase 2.5 被跳过，**立即回 Phase 2.5 问用户**，不得直接进 Phase 4。
- 决策之间矛盾？（如选了 async 框架但接口全同步）
- 遗漏？（如定义了错误类型但没说谁负责转换）
- 命名约定贯穿所有接口？
- 实施计划匹配架构？
- Phase 0.5 的假设是否全部被 Phase 2 的决策覆盖？

**生产级自查**（8 维度逐项核对）：
- 正确性与边界：边界条件、错误处理、失败路径都规划到了？
- 并发与资源：共享状态 / 锁 / 资源释放 / 超时 / 重试策略清楚？
- 可观测性：关键路径日志 / 埋点 / metrics 规划了？
- 安全：外部输入校验 / 敏感信息处理 / 注入防护考虑了？
- 性能：复杂度估算 / 热路径识别 / 批量化策略想过？
- 可维护性：命名 / 抽象层级 / 职责单一性有方案？
- 测试策略：happy + 边界 + 错误路径的覆盖计划？
- 文档：公共 API 契约、关键决策记录？

**可施工性自查**：
- 每阶段的"具体动作"字段足够细使 AI 能直接执行？（没有"完成功能"这种空话）
- 每阶段都有完整字段（目标 / 动作 / 交付物 / 验收 / 施工纪律 / 风险 / 预警 / 依赖 / 回滚 / 生产级自检）？
- 每阶段独立可编译可验证？
- 前置依赖无环？
- 用户说"完成整个计划"或"执行阶段 N"时，AI 有足够信息一次做对，不会卡在信息不够？

发现不一致 / 遗漏 → plan mode 内修正草稿。

## Phase 5: 下一步提示

- **含 `--auto`**：跳过本阶段的提示文本，直接按计划顺序执行所有阶段（完全等同于用户说了"完成整个计划"）
- **默认**：审批通过后，根据目的给出启动施工的提示（见下）

### 施工类 / 生成类 / ship 目的

```
✅ Plan 已进入会话上下文，可直接施工。

下一步：
  • 说"完成整个计划" —— 按所有阶段顺序推进
  • 说"执行阶段 N"  —— 单独跑某个阶段
  • 说"从阶段 N 开始" —— 从某阶段开始往后做

施工中如发现计划有缺陷：停下，重新 /pax 调整。不要偏离计划临时发挥。

如需永久留存本次决策：
  /report decision
```

### review 目的（全局 Tier 分层产出元计划时）

review 的局部产出直接汇报给用户；产出全局 Tier 分层时，它是**元计划**，不直接施工——而是引导后续多次 pax：

```
✅ Review plan（元计划）已产出。

这是一份元计划，指引后续多次 pax。

下一步建议：
  针对 Tier 1 的每一项独立 /pax --<目的> 做细化计划，再按计划施工
  建议从 <Tier 1 第一项> 开始

如需永久留存本次审视结果：
  /report decision
```

### loop 目的

loop 产出的是**循环契约**，审批后由 loop 执行层直接启动循环，不走普通"说完成整个计划"施工流程：

```
✅ Loop 契约已就绪。

契约摘要：
  镜头：<LENS 或"无">
  循环体：<BODY 一句话摘要>
  终止条件：<UNTIL>
  模式：<直接执行 / subagent（批次 batch-until: <...>）>

下一步：
  • 说"启动循环" / "Go" —— 立即进入循环
  • 说"修改契约" —— 回到 plan mode 调整 BODY/UNTIL/模式
  • 说"dry-run 1 轮" —— 试跑 1 轮再决定是否正式启动

铁律：循环启动后，只有 UNTIL / BATCH_UNTIL 满足、BOUNDS 触发、或用户打断才能停。

循环结束后默认自动调 /report retro 做复盘；如不需要请在契约里关闭 --report-on-exit。
```

审批未通过：留在 plan mode 按用户意见调整，再次 `ExitPlanMode`。

## 施工阶段（pax 审批通过后）

pax 审批通过后进入施工阶段。施工不是另一个独立 skill，而是**按计划直接推进**。

**用户可能的触发语句**：
- "完成整个计划" / "按计划做" / "Go"
- "执行阶段 N"
- "从阶段 N 开始"
- "继续" / "下一阶段"

**AI 的施工纪律**：

1. **严格按计划的阶段和具体动作执行**——每阶段的字段（动作、交付物、验收、施工纪律、回滚点、生产级自检）是强制约束
2. **每阶段末必须做验收**：运行验收标准里的每项验证，全部通过才能进入下一阶段
3. **小步提交**：按计划的"施工纪律"字段执行（通常是每个具体动作完成后编译 + 相关测试 + commit）
4. **生产级代码标准贯穿**：8 维度全程适用，不存在"先写能跑的后面再补"
5. **发现计划有缺陷时的处理**：
   - **默认模式**：停下向用户说明，选择回 `/pax` 修正或继续施工
   - **auto 模式**：警告 + 在会话内自动修正计划（仍 auto）继续；再次失败则警告 + 终止并汇报
6. **失败时的处理**：
   - **默认模式**：按回滚点回滚，不留半完成状态；停下让用户决定是否继续
   - **auto 模式**：警告 + 按失败类型自主决策（代码错误直接修；行为破坏 / bench 退化警告后记录继续；回滚后仍无法前进则标红该阶段"需人工介入"后继续剩余阶段）
7. **auto 模式不停下**：一切异常以"警告 + 继续"处理，最后汇总汇报；只有物理上无法继续（无法修复的编译错误、无法写入文件系统、用户主动打断）才真正停下

**每阶段完成后的标准输出**：

```
✅ 阶段 N 完成

改动总结：
  <ASCII 文件清单 / 结构变化 / 接口变化 / 行为变化 / 故意未改>

验收：
  ✅ <验收项 1>
  ✅ <验收项 2>

生产级自检：
  ✅ <本阶段自检要点 1>
  ✅ <本阶段自检要点 2>

Commit: <hash>

下一阶段：阶段 N+1 <标题>
```

全部阶段完成后给出总改动总结，并提示：

```
🎉 整个计划已完成。

总改动：<跨所有阶段的累计 ASCII 改动总结>

建议下一步：
  /verify              事后检查
  /verify --ship       准备发布
  /report              留存本次施工的决策和结果
```

## 异常处理

### 用户中途要求直接开工（绕过 pax）

1. 询问议题复杂度：
   - 只是一个小改动 / 明确的独立动作 → 仍建议先走 pax，理由是"即使小改动也需要假设、生产级、回滚考量"
   - 用户坚持直接动手 → 提示：`/pax` 目的可自动推断，`--auto` 模式几乎不打断你，值得一用
2. 用户仍坚持 → 允许跳过，但告知"跳过 pax 等同跳过假设确认和生产级自查，出问题责任在跳过"

### 中途发现需求不清晰

1. 停止推进
2. 向用户说明哪部分无法解读
3. 用户补充后回到相应 Phase；或中止

### 议题爆炸

1. Phase 0 就与用户确认"本次只覆盖 <核心维度>"
2. out-of-scope 写入计划"未覆盖项"章节
3. 避免一个 session 把所有事定死——plan 是会话本地的

### 施工中发现计划缺陷

施工阶段发现的计划缺陷**必须停下**，不能临时改方案。选项：

1. 用户给出小幅调整 → AI 在计划内补丁并继续，补丁记入"施工日志"章节
2. 缺陷影响整体方向 → 回 `/pax --<目的>` 重新规划
3. 紧急情况需要先打补丁 → 允许施工，但必须在计划里记录"紧急偏离"，事后回归

## 与其他 skill 的衔接

- **施工**：pax 审批通过后，AI 按计划直接推进——不需要独立 skill 触发。
- **`/verify`**：事后检查 / 平时检查 / 发布闸。施工全部完成后或任何需要审视时使用。
- **`/report decision`**：永久留存决策。`/pax` 自身不落盘，产物归 `/report`。
