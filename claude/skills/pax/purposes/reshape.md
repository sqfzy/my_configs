## 目的：reshape（代码结构改造）

**适用**：代码能跑、功能正确，但结构不合理——模块划分、依赖方向、接口设计、抽象层级、命名一致性、重复代码、技术债…需要重新组织。

**覆盖全谱系**：从"小幅重构 / 提取函数 / 重命名"到"激进的整体重设计 / 从零思考"。由 plan 本身说明改造幅度，不靠外部模式区分。

本文件只定义 reshape 目的**独有**的约束，通用维度由 SKILL.md 规定。

---

### reshape 独有的必展开章节

以下章节**必须**在计划中显式展开：

#### 1. 现状诊断（定位到具体符号，不空谈）

必须精确到：
- 具体文件 / 行号范围 / 符号名
- 具体问题类型：
  - 模块过大（给出行数 / 职责数量）
  - 循环依赖（画出依赖环）
  - 职责混杂（列出该模块混的几件事）
  - 抽象泄漏（哪个接口暴露了不该暴露的内部细节）
  - 命名不一致（列出同概念的几种叫法 + 出现次数）
  - 重复代码（给出重复块的位置和相似度）
  - 公共 API 设计缺陷（列出难用 / 易误用的点）

**陷阱**：reshape 最常见的失败是"凭感觉觉得乱就改"。没有具体定位 + 具体问题类型的诊断，改造方向注定飘忽。

#### 2. 改造幅度声明

**必须在计划最显眼的位置写一句**：

```
改造幅度：
  [ ] 保行为重构（外部行为不变，仅结构调整）
  [ ] 允许行为变化（breaking change）
```

勾选前者时：
- 必须列出**不变量清单**（见下条）
- 所有现有测试必须在改造后仍然通过，且不允许修改测试本身
- 若改造过程中发现测试需要修改，stop，重新评估是否应该转为"允许行为变化"

勾选后者时：
- 必须列出**行为变更清单**（见下条）
- 必须显式列出 BREAKING CHANGE 条目，供 commit footer 使用
- 需要明确升级指引（调用方如何适配）

#### 3. 不变量清单（保行为重构必含）

逐项声明"改造前后必须保持什么"：

```
| 不变量 | 验证方式 |
|--------|---------|
| 公共 API 签名 | grep + 类型系统 |
| 行为语义 | 现有测试 + 新增等价性测试 |
| 错误类型 / 错误信息 | 保持或列出例外 |
| 性能特征 | bench 不退化（或注明预期变化） |
| 对外协议 / 持久化格式 | 保持 |
```

每条不变量要给出**客观验证方式**，不是"感觉没变"。

#### 4. 行为变更清单（允许行为变化时必含）

逐项声明"改了什么"：

```
| 变更 | 旧行为 | 新行为 | 受影响方 | 升级指引 |
|------|--------|--------|----------|---------|
```

每条必须有受影响方和升级指引，不能留给调用方自己猜。

#### 5. 迁移策略

必须从三种中选一种（或组合）并说明理由：

- **一次性**：改完全部再提交。适用小范围、单人可掌控、原子性强。
- **渐进共存（strangler）**：新旧接口共存，逐个调用方迁移。适用调用方多、风险高、新旧并存周期明确。
- **分阶段分批（checkpoint）**：每阶段独立交付、独立验证、可回滚。适用跨模块、激进重设计。

选定后必须说明：
- 为什么选这种（为什么别的不合适）
- 若选渐进共存，给出**调用方迁移顺序**（树状 / 列表）
- 若选分阶段分批，每阶段独立 commit、每里程碑强制全验证

#### 6. 调用方迁移顺序（涉及接口变更时必含）

给出：
- 调用方清单（文件 / 符号级）
- 迁移顺序（通常按依赖拓扑：被依赖少的先改）
- 每个调用方的迁移工作量估算（小 / 中 / 大）
- 哪些调用方在项目外（若有，单独标注需要通知 / 文档 / 迁移窗口）

### reshape 必含 ASCII 图示（在通用要求之上追加）

**强制必含**：
- **现状 vs 目标结构并排对比**（目录 / 模块 / 依赖关系对比）
- **目标架构图 + 模块依赖图**

按场景追加：
- 涉及核心流程变化 → **关键流程前后控制流对比**
- 涉及接口重塑 → **接口前后签名对比**
- 涉及渐进迁移 → **迁移时序图**（哪个阶段什么在哪里）

### reshape 实施计划的特殊约束（施工契约）

**阶段划分原则（强制执行拓扑序）**：

```
先改底层 → 再改中间层 → 最后改上层
先建新接口 → 再让旧接口委托新接口 → 再逐个调用方迁移 → 最后删除旧接口
```

每阶段**必须独立可编译、可测试、可提交、可回滚**。禁止出现"阶段 3 完成前代码跑不起来"的跨越式阶段。

**阶段骨架示例（具体视迁移策略调整）**：

```
阶段 0: 基线捕获
  - 具体动作：记录当前测试数量、bench 数值、公共 API 签名（作为不变量验证基线）
  - 验收：基线存档

阶段 1: 新抽象 / 新接口引入（不动旧代码）
  - 具体动作：新增 trait / 类型 / 模块，不删除旧代码；确保编译通过
  - 验收：新增代码有基础单元测试；所有现有测试不受影响

阶段 2..N: 逐项迁移
  - 若选一次性：单阶段做完
  - 若选渐进共存：每个调用方（或一批）独立阶段
  - 若选分批：按模块拓扑拆
  - 每阶段后：全量测试 + 不变量验证（保行为）/ 变更清单对照（允许变化）

阶段 N+1: 清理
  - 具体动作：删除旧代码 / 旧接口 / 过渡 re-export
  - 验收：无 dead code；grep 旧符号应为空

阶段 N+2: 最终验证
  - 运行全量测试 + lint + bench
  - 对照不变量清单（保行为）或变更清单（允许变化）逐项核对
```

**保行为重构的每阶段后**：
- 现有测试必须全过
- 不允许修改测试本身（若测试必须改，说明行为变了，停下重评）

**允许行为变化的每阶段后**：
- 测试要么全过，要么修改范围已在"行为变更清单"中预告
- 每个 breaking 点对应一个独立 commit，commit footer 含 `BREAKING CHANGE: ...`

### reshape 的生产级重点（施工时必须贯穿）

8 维度里 reshape 要特别强化：

- **可维护性**：reshape 的目标就是提升可维护性
- **可观测性**：改造不能退化观测——若旧代码有日志 / tracing，新结构必须至少保留同等级别
- **测试策略**：保行为的前提是测试覆盖——改造前先补齐测试（若不足，必须在计划里列出"前置补测试"阶段）
- **性能**：不能退化；bench 覆盖的路径必须改造后 bench 通过

**特别强调**：**每阶段的代码仍需达到生产级，不存在"中间态丑陋、改完再补"**。生产级是每阶段的标准，不是终点的标准。

---

### reshape 的施工纪律（写入计划的"施工纪律"字段）

- 每阶段独立 commit
- 每阶段末运行全量测试 + lint
- 保行为阶段：不允许 touch 测试文件（除非新增等价性测试）
- 允许变化阶段：测试改动必须和对应 breaking 声明一起 commit
- 激进重设计（选 checkpoint 策略）：每个里程碑跑 bench 对比基线，退化 ≥ 5% 停下评估
- 不允许夹带无关修复 / 无关 feat 改动——发现就停下独立 `/pax`

---

### reshape 的反模式（计划里出现即不合格）

- ❌ 没有定位到具体符号 / 文件的"感觉乱"诊断
- ❌ 没有明确"保行为 vs 允许行为变化"的声明
- ❌ "先全改完再测" —— 违反每阶段独立可验证
- ❌ "中间态有段时间代码跑不了" —— 违反独立可编译
- ❌ 没有调用方迁移顺序
- ❌ 没有不变量 / 行为变更清单中任意一个
- ❌ 对激进重设计没有给出分批 checkpoint 方案
- ❌ 夹带 fix / feat 改动

---

### reshape 的风险与回滚

- 单阶段失败时的回滚粒度：回到上阶段末的 commit
- 保行为改造的预警信号：
  - 某阶段后现有测试失败 → 行为已变，停下评估是否应转"允许行为变化"
  - 性能 bench 退化 → 停下，评估是否继续
- 允许行为变化的预警信号：
  - 测试失败不在预告的变更清单里 → 意外行为变化，停下修复
  - 调用方迁移出现未预料的依赖链 → 停下补充调用方清单
- 激进重设计预警：
  - 某里程碑后代码显著更乱 → 方向错了，停下回 `/pax --reshape`
  - 多个不变量同时被破坏 → 已经不是保行为，停下重声明改造幅度
