## 目的：feat（新功能 / 新模块 / 新项目）

**适用**：要做一个之前没有的东西——新功能、新模块、新组件、新项目。

本文件只定义 feat 目的**独有**的约束，通用维度（定位 / 现状 / 目标 / 影响 / 生产级 8 维度 / 图示 / 实施计划字段）由 SKILL.md 规定。

---

### feat 独有的必展开章节

以下章节**必须**在计划中显式展开，不能合并不能跳过：

#### 1. 技术选型

若涉及新选型（语言 / 框架 / 核心依赖 / 构建工具 / 测试框架 / 日志/观测库），每一项必须：
- 明确写出选择
- 给出至少一条选择理由（为什么选这个而不是其他）
- 若项目已有默认选型，必须明确"沿用项目既有 X"

**陷阱**：不要在 feat 中把"选型"偷偷带过——后续施工时每一个依赖引入都是决策点，前置明确才不会反复返工。

#### 2. 架构设计

必须给出：
- **模块划分**：每个新模块的名称、职责、边界
- **依赖方向**：模块间的依赖关系（箭头方向清晰，不允许循环依赖未标注）
- **核心抽象**：关键 trait / interface / 基类及其契约
- **数据模型**：关键类型定义
- **并发 / 异步模型**：同步 / 异步 / Actor / Channel / 线程池 …
- **对外接口形态**：CLI / API / 库 / 服务

#### 3. 接口设计

必须给出：
- **公共 API 签名**：至少关键函数的签名（参数类型、返回类型、错误类型）
- **错误体系**：错误类型定义、分类、调用方如何处理
- **关键数据结构**：入参 / 出参 / 配置类型

#### 4. 扩展点预留

feat 目的必须主动回答："未来同类需求如何扩展？"
- 是否有 trait / plugin / hook 设计
- 若明确不需要扩展，在计划中显式写出"本功能不预留扩展点，理由：<…>"

**陷阱**：feat 最常见的问题是当前够用但未来同类需求来时发现无法扩展只能重写。必须前置回答。

### feat 实施计划的特殊约束（施工契约）

**阶段划分原则（强制拓扑序）**：

```
阶段 1: 骨架与核心抽象
  - 建立模块骨架（空文件 + 模块声明 + re-export）
  - 定义核心 trait / interface / 关键类型
  - 确认骨架能编译
  - 具体动作：新建文件 X.rs，写空 mod，添加 trait Y { fn foo() -> Result<Z, E> }
  - 验收：cargo build 通过；类型签名与计划一致

阶段 2: 纯逻辑函数（从不依赖 I/O 的内层开始）
  - 实现核心算法 / 纯转换函数 / 数据处理
  - 单元测试 happy + 边界 + 错误路径
  - 验收：新增单元测试全过；覆盖 happy + 至少 2 个边界

阶段 3: I/O 和集成层
  - 实现与外部交互的层（网络 / 磁盘 / DB / 子进程）
  - 加入观测（日志 / metrics / tracing）
  - 集成测试
  - 验收：集成测试全过；关键路径观测已加

阶段 4: 对外接口拼装
  - CLI / API / 库导出
  - 文档注释（公共 API 必含）
  - End-to-end 测试
  - 验收：对外能调；doc comment 完整
```

每个阶段的"具体动作"字段必须**足够细到每个文件 / 函数级别**，使 AI 一次能做对。

**禁止**：
- ❌ 一个阶段里同时做多个模块的实现
- ❌ 把测试放到"最后阶段"补——每个阶段的验收标准里应该就有测试要求
- ❌ "先搭架子不写实现" —— 架子本身应该是可编译可测试的
- ❌ 一个阶段内同时动核心抽象和 I/O 层——违反"由内向外"

### feat 的生产级重点（施工时必须贯穿）

8 维度里 feat 要特别强化这几项（即使议题小也不能跳）：

- **错误体系**：必须在计划阶段就定下错误类型，不允许"先用 anyhow 占位以后再改"。施工阶段 1 就必须把错误类型定死。
- **可观测性**：关键路径的日志级别和字段必须在计划中示例。施工阶段 3 必须加入观测，不是"以后再补"。
- **测试策略**：happy path + 至少 2 个边界 / 错误路径 —— 每个公共函数都要。施工每个阶段末必须有对应测试。
- **文档**：公共 API 必须有 doc comment。施工阶段 4 完成前必须补齐。

---

### feat 的施工纪律（写入计划的"施工纪律"字段）

- 小步提交：每个具体动作（新增 trait / 实现函数 / 加测试）一个独立 commit
- 每步后运行相关测试
- 不要为了赶进度跳过阶段——阶段间的拓扑序是生产级的保证
- 新增依赖必须在计划中已声明，否则停下和用户确认

---

### feat 的反模式（计划里出现即不合格）

- ❌ "这是 MVP，错误处理以后再补" —— MVP 也要生产级错误处理，只是场景更少
- ❌ "先做 happy path，边界 case 后期迭代" —— 边界 case 是阻断上线的杀手
- ❌ "观测性看后续需要" —— 加不加观测在设计阶段定，不是看心情
- ❌ 没有明确的 out of scope —— 没边界的计划不是计划
- ❌ 实施计划只有一个阶段 "实现全部功能" —— 违反每阶段独立可验证
- ❌ "具体动作"字段只写一句抽象话（"实现解析器"）—— 施工时 AI 会不知道从何下手

---

### feat 的风险与回滚

- 单阶段失败时的回滚粒度：回到上阶段末的 commit
- 预警信号：
  - 阶段 2 纯逻辑测试出现大量边界失败 → 核心抽象有问题，停下回 `/pax`
  - 阶段 3 集成时发现接口不合适 → 同上
  - 任何阶段出现循环依赖或抽象泄漏 → 停下，评估是否应该转 `--reshape`
