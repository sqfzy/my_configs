## 目的：doc（文档生成 / 一致性核查）

**适用**：为现有代码产出 / 更新文档——README、API 文档、CHANGELOG、inline doc comments、onboarding 指南、架构概览。

**doc 会生成文档**（不是可选）—— 分析文档需求的同时生成 / 更新。只想要"核查文档一致性不改"的场景，该走 `--review` 聚焦文档维度。

**铁律**：文档必须**从实际代码反向生成**，不允许凭想象写。

本文件只定义 doc 目的**独有**的约束，通用维度由 SKILL.md 规定。

---

### doc 独有的必展开章节

#### 1. 文档类型与范围

**必须在计划中明确**：

```
文档类型（可多选）：
  [ ] README（项目总览 / 快速开始）
  [ ] CHANGELOG（版本变更记录）
  [ ] API 文档（公共接口契约）
  [ ] Inline doc comments（源码内注释）
  [ ] Onboarding 指南（新人上手）
  [ ] 架构概览（模块 / 数据流 / 关键决策）
  [ ] 其他：<具体>

范围：
  [ ] 整个项目
  [ ] 某模块 / 某 crate：<名称>
  [ ] 某一类（如"所有公共 API 的 doc comment"）
```

#### 2. 现状扫描（必做）

**doc 不能凭印象写** —— 必须先系统扫描代码现状：

- **公共接口清单**：所有 pub / export / 导出的符号（精确到函数 / 类 / 类型）
- **已有文档清单**：每个公共符号是否已有 doc comment / 文档条目；若有，内容是否与实现一致
- **文档缺失清单**：没有文档的公共符号 / 模块
- **文档过期清单**：文档存在但与当前实现不一致的条目（如签名改了文档没改）
- **文档风格现状**：项目现有文档的风格（Rust rustdoc / TypeScript TSDoc / Google style / NumPy style…）——新生成必须沿用

**陷阱**：跳过现状扫描直接生成文档，会产生大量重复、冲突、不一致。

#### 3. 生成策略

必须明确：
- **文档来源**：直接从代码提取（函数签名、类型、错误类型、控制流）；不允许凭猜测补充代码中没有的行为
- **文档风格**：沿用项目既有（rustdoc / TSDoc / docstring / Markdown …）
- **文档层级**：
  - 公共 API 必须有 doc comment
  - 非 trivial 私有函数按需有注释（why 而非 what）
  - 模块级有模块 doc
  - 项目级有 README + 架构概览
- **示例代码**：doc 里的示例必须**可运行**（rustdoc doctest / Python doctest / Go Example）
- **链接**：跨项目链接（README → 模块 doc → API doc）必须正确

#### 4. 文档内容标准

每个公共 API 的 doc 必须包含：
- **一句话摘要**（出现在列表页）
- **详细说明**（背后逻辑、使用场景）
- **参数说明**（类型、约束、含义）
- **返回值说明**（类型、含义、特殊情况）
- **错误条件**（什么情况下返回 error / 抛异常）
- **Panic 条件**（若有）
- **使用示例**（至少一个，且可运行）
- **相关引用**（See also）

**陷阱**：只写一句话摘要就交差是最常见的低质量文档。

对于 CHANGELOG，每条必须：
- 版本号 + 日期
- 分类（Added / Changed / Deprecated / Removed / Fixed / Security）
- 一句话 + 必要的上下文（影响范围、迁移指引若有）

#### 5. 特殊文档类型的附加要求

**README** 必含：
- 项目一句话介绍
- Install / Quick Start（可复制粘贴的命令）
- 基本使用示例（最小可跑）
- 指向详细 API 文档的链接
- 许可证

**架构概览** 必含：
- 系统目的 + 边界
- 主要模块及职责（配 ASCII 图）
- 数据流 / 控制流（配 ASCII 图）
- 关键决策记录
- 已知约束 / 限制

**CHANGELOG**：
- 遵循 Keep a Changelog 格式（或项目既有格式）
- 最新版本在顶部
- 未发布改动放 `[Unreleased]` 节

---

### doc 必含 ASCII 图示（按文档类型选用）

- 架构概览 / README：**模块依赖图** / **数据流**
- API 文档（大模块）：**模块内部结构图**
- CHANGELOG：若某版本有重大结构变化，用**改动前后对比图**

### doc 实施计划的特殊约束（施工契约）

**doc 是施工类目的** —— 产出"修改文档文件的阶段骨架"，因为 doc 必然生成 / 更新文档。

**阶段骨架**：

```
阶段 0: 现状扫描
  - 具体动作：
    1. grep 所有公共符号（按项目惯例，如 Rust 的 `pub`、TS 的 `export`）
    2. 对照已有文档，统计覆盖率 / 缺失 / 过期
    3. 读取现有文档的风格样本
    4. 记录项目的文档风格惯例
  - 交付物：现状报告（见"现状扫描"章节）
  - 验收：清单完整、定位到具体符号

阶段 1: 删除与重建策略决策
  - 原则：对于 API 文档 / 架构概览等**从代码生成**的文档，倾向"delete-and-regenerate"而非增量更新（避免过期文档残留）
  - 对于 CHANGELOG / README / onboarding 等人工维护的，**必须增量更新**，不允许删除历史
  - 具体动作：明确本次每类文档的处理方式
  - 验收：每类文档有明确策略

阶段 2..N: 按类型 / 按模块生成（每类一个阶段）
  - 具体动作（以 API doc 为例）：
    1. 对阶段 0 清单中的每个公共符号，从代码提取签名、错误类型、示例
    2. 按项目风格写 doc comment，必含 4 要素（摘要 / 参数 / 返回 / 错误）+ 示例
    3. 运行 doctest（若可）确认示例可跑
    4. 一组（如一个模块）一个 commit
  - 验收：
    - 所有目标符号有 doc
    - 示例可跑（若适用）
    - 风格与项目一致

阶段 N+1: 交叉链接与索引
  - 具体动作：
    1. 确保 README → 模块 doc → API doc 链接正确
    2. 生成 / 更新文档索引（若项目有）
    3. 按需更新"相关引用"（See also）
  - 验收：无死链；索引完整

阶段 N+2: 最终验证
  - 具体动作：
    1. 运行文档构建（cargo doc / mkdocs / typedoc / …）
    2. 检查无警告 / 无死链
    3. 抽查几个公共 API 的 doc 可读性
  - 验收：
    - 构建通过无警告
    - 抽查的 doc 读起来清晰可操作
    - 示例运行通过
```

**每阶段独立可提交**。禁止"一次性改几十个文件的文档"大 commit。

### doc 的生产级重点（施工时必须贯穿）

- **从代码反向生成**：不允许凭想象补充代码没有的行为
- **示例可运行**：凡有示例必须是真能跑的，不能假装
- **一致性**：整个项目一个文档风格，不混搭
- **版本化**：文档版本和代码版本绑定（CHANGELOG 必有版本号）
- **无死链**：所有跨文档链接必须有效
- **无敏感信息**：不在文档中暴露内部路径、密钥、token、PII

---

### doc 的施工纪律（写入计划的"施工纪律"字段）

- 每一类（或每一模块）文档一个独立 commit（message 明确"docs: <范围>"）
- 每个 commit 后运行文档构建（cargo doc / mkdocs 等）
- 若文档含示例，必须运行 doctest 确认可跑
- 不允许"文档和代码分开 PR" —— 代码改必须同时改文档（若 API 改了，doc 必须跟着改）
- 不允许"复制已有文档到新位置"—— 每份文档必须从当前代码重新生成

---

### doc 的反模式（计划里出现即不合格）

- ❌ 没有现状扫描，直接生成文档（会产生大量冲突）
- ❌ 凭猜测写 API 行为（"这个函数大概会..."）
- ❌ 只有一句话摘要，没有参数 / 返回 / 错误 / 示例
- ❌ 示例代码不可运行（假代码）
- ❌ 文档风格和项目既有混搭
- ❌ README 里的 install 命令不能直接复制运行
- ❌ CHANGELOG 删除历史条目
- ❌ 文档里暴露敏感信息（路径 / 密钥 / PII）
- ❌ 改 API 不改 doc（或反之）
- ❌ 一个 commit 改几十个文件的文档（无法 review）

---

### doc 的风险与回滚

- 某阶段生成的文档和代码不一致 → 说明代码被误改或理解有误，回滚该阶段重新扫描
- 文档构建失败 → 通常是格式 / 链接问题，修复后继续；不允许关闭文档构建警告
- 示例运行失败 → 说明示例写错或代码有 bug，修例后继续；若反复失败，改简化版示例
- 整个文档生成后读起来"像堆砌材料" → 说明缺少关联叙事，回 `/pax --doc` 重新规划章节结构
