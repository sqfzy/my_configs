## 目的：review（代码审查 + 全项目审视 + 演进规划）

**适用**：对代码做质量核查——可以是局部（某段 diff / 模块 / PR）、也可以是全局（整个项目的技术债、生产级体检、演进规划）。

**范围自动分流**：

- 用户给了具体范围（diff / PR / 文件 / 模块） → **局部 review**：聚焦发现清单 + 逐项建议
- 用户说"全项目 / 体检 / 技术债 / 下一步做什么 / 缺什么" → **全局 review**：产出 Tier 分层元计划 + 演进路线，引导后续多次 pax

两种范围共享同一套扫描维度（生产级 8 维度 + 风格一致性），差别只在输出组织形式。

**默认只读**：不改代码，只产出发现 / 报告。

本文件只定义 review 目的**独有**的约束，通用维度由 SKILL.md 规定。

---

### review 独有的必展开章节

#### 1. 范围声明

**必须在计划最显眼位置写一句**：

```
review 范围：
  [ ] 局部：<具体范围：diff / PR#N / 模块 X / 文件 Y>
  [ ] 全局：整个项目
```

局部 review 后续产出"发现清单 + 建议"；全局 review 后续产出"元计划 + 演进路线"。

#### 2. 扫描维度（局部 + 全局共用，是 review 的核心工作）

**必须系统扫描**生产级 8 维度 + 风格一致性，不允许只凭印象：

**生产级 8 维度扫描**：
- **正确性与边界**：grep `unwrap()` / `panic!()` / `expect()` / 空 catch / 静默忽略 `Result`；边界条件是否覆盖
- **并发与资源**：共享状态的同步 / 锁 / 资源释放路径（含错误路径） / 超时 / 重试 / 背压
- **可观测性**：关键路径日志覆盖度 / 错误分支是否有日志 / 日志是否结构化可操作
- **安全**：外部输入校验 / 敏感信息（密钥 / token / PII）处理 / 注入风险 / 依赖 CVE
- **性能**：循环内 I/O / 无界集合 / 已知热点 / 不必要的 clone / 复杂度
- **可维护性**：大文件 / 长函数 / 深嵌套 / 职责混杂 / 抽象泄漏 / 重复代码
- **测试策略**：测试数量 / 覆盖率（若有工具）/ 未测模块清单 / 边界与错误路径覆盖
- **文档**：公共 API 缺 doc comment 数量 / 错误语义未文档化 / 关键决策未记录

**风格一致性扫描**（跨文件）：
- 命名约定（同概念用同一名字？动词 / 命名风格是否统一）
- 错误处理模式是否统一（是否多种错误体系并存）
- 日志 / tracing 风格是否一致
- 公共 API 风格是否统一（函数命名动词、参数顺序惯例）

**架构与 API 设计扫描**（局部 review 在改动涉及接口 / 模块边界时启用；全局 review 必扫）：

- **架构形态**：项目结构是否符合该规模 / 该领域已收敛的有效模式（分层 / 六边形 / 事件驱动 / actor / pipeline / DDD / clean）？识别现有特征，评判是否合适，**不强推**新模式
- **API 人体工程学**：命名 / 参数顺序 / 错误返回风格 / 异步形态 / batch vs single / config struct vs builder vs flat args / 默认值处理 / 类型安全程度（newtype / phantom / flag enum）
- **抽象层级**：是否有泄漏（底层细节穿透到上层）？是否有过度封装（一层薄薄的转发层）？

**现代化对标 + 同类项目对比**（局部 review 改 API 时启用；全局 review 必扫）：

- **当下最佳实践**（按语言 / 生态 2024-2026 收敛惯例）：
  - Rust：`thiserror`+`anyhow` / async-trait / `#[non_exhaustive]` / typed builders / `tracing` 而非 `log` / `Result<T, E>` 而非 panic
  - C++23：`std::expected` / `std::span` / `std::format` / concepts / coroutines / RAII guard
  - Python：type hints + pyright / pydantic v2 / `match` 替代 isinstance / async-first
  - 标注本项目**对齐 / 部分对齐 / 未对齐**，列出未对齐的具体位置
- **同类成熟项目对比**（必选 1-3 个对标对象）：
  - 同语言 + 同领域优先（写网络库 → tokio/hyper/mio/smoltcp；写 ORM → sea-orm/sqlx/diesel；写 LSM tree → RocksDB/sled/Pebble）
  - 对比维度：模块切分 / 命名 / 错误体系 / 公共类型导出 / 主接口形状 / 默认行为
  - 输出：**对比矩阵**（本项目 vs 对标 1 vs 对标 2，每维度一句结论），不写"差不多"这种空话
  - **命名校准**：本项目内同概念是否用了行业默认词？独创术语是否合理（有则保留，无则改）

**过度设计防线**（铁律 —— 防止 review 变成"最佳实践堆砌癖"）：

- "看着像最佳实践但本项目暂时用不上"的设计 → **不进 Critical / Major**
- 改记入新档"📎 参考点"：未来若 X 发生（明确触发条件），考虑引入 Y
- 例："目前单实例够用，不必引入 actor；若未来多核扩缩容，参考 actix 的 Arbiter 设计"
- 判据：当前痛点 / 风险 / 摩擦 ≠ 0 才进 Critical/Major；为"将来可能"提案的进 📎 参考点

每维度的发现必须：
- 定位到具体位置（文件:行 或 符号）
- 问题描述
- 严重程度：🔴 Critical / 🟡 Major / 🔵 Minor / 📎 参考点（未来触发条件命中再考虑）
- 建议的处理目的（feat / fix / reshape / upgrade / bench / test / doc）；📎 参考点不需立即对应处理目的

#### 3. 局部 review 独有的产出

若范围是局部（diff / 模块 / PR）：

**发现清单**（按严重程度分组）：

```
🔴 Critical（必改）
  1. <文件:行> — <问题描述> — <建议修复>
  2. ...

🟡 Major（应改）
  1. ...

🔵 Minor / Nit（可改）
  1. ...
```

**改动面评估**（针对 diff 类 review）：
- 本次改动的波及范围（调用方、公共 API、外部协议）
- 是否引入行为变化（哪怕微小）
- 是否有与改动无关的夹带代码

**未覆盖考虑点清单**：列出"本次改动应该考虑但没考虑"的维度（如生产级 8 维度的盲区）。

#### 4. 全局 review 独有的产出

若范围是全项目：

##### 4.1 项目画像

- **规模指标**：代码量（文件数 / 总行数）、模块数、测试数量、依赖数量
- **技术栈与依赖健康度**：主要依赖的版本 + 是否过时 / 有 CVE / 接近 EOL
- **成熟度判断**（五档之一）：
  - 原型（POC，能跑但不稳）
  - 早期（基本功能齐，仍在快速迭代）
  - 稳定（功能完整，在生产运行）
  - 成熟（功能稳定，主要在维护）
  - 僵化（技术债深，迭代困难）
- **主要资产**：已完成的关键能力 / 突出的优势点
- **明显短板**：一句话版的"最大的两三个问题"

##### 4.2 问题全景

把扫描维度找到的所有问题按严重程度汇总（Critical / Major / Minor），每条必须标注：
- 具体位置（文件 / 模块，精确到符号更好）
- 问题描述
- **建议的处理目的**（feat / fix / reshape / upgrade / bench / test / doc）
- 预估工作量（小 / 中 / 大）

##### 4.3 Tier 分层（全局 review 核心产出）

把问题全景**重新排序**为 Tier：

```
Tier 1（必做，不做项目不可用 / 有严重隐患）：
  1. <条目> — 建议目的: /pax --<目的> — 工作量: 中
  2. ...

Tier 2（应做，显著提升项目价值）：
  1. ...

Tier 3（可做，锦上添花）：
  1. ...

搁置（明确标为"目前不做"）：
  1. <条目> — 理由: <为什么现在不做>
```

Tier 划分原则：
- 是否阻塞正常使用 / 交付？是 → Tier 1
- 显著影响 ROI（可维护性 / 性能 / 团队效率）？是 → Tier 2
- 只是审美 / 长期优化？→ Tier 3
- 当前不值得做（成本 > 收益 / 等时机）？→ 搁置

##### 4.4 演进路线

- **从哪里开始**：Tier 1 的第一项，明确入口
- **依赖关系**：哪些项必须先做完才能做后续（画 DAG 或列清单）
- **节奏建议**：不强求时间估计，给"Tier 1 先做完再 Tier 2"这种顺序
- **依赖外部决策的项**：哪些项需要用户 / 团队决策才能继续（标注）

### review 必含 ASCII 图示（按范围选用）

**局部 review**：
- 若涉及跨模块 / 多文件 → **影响范围图**
- 若涉及行为变化 → **修改前后行为对比**

**全局 review**：
- **项目画像图**（模块依赖全景 + 耦合热点标注）
- **问题分布图**（按严重程度 / 按模块维度展示问题密度）
- **演进路线 DAG**（Tier 1 → Tier 2 → Tier 3 的依赖关系）
- **现代化对标矩阵**（必含若做了"同类成熟项目对比"扫描）：

  ```
  | 维度       | 本项目          | tokio 1.36     | hyper 1.x      | 评判          |
  |-----------|----------------|----------------|----------------|--------------|
  | 错误体系   | 自造 Error enum | thiserror      | thiserror      | 🟡 未对齐    |
  | 异步形态   | 半同步半 Future | async-await 全 | async-await 全 | 🟡 未对齐    |
  | 命名"任务" | task / job 混用 | task           | task           | 🔵 命名校准  |
  | ...       | ...            | ...            | ...            | ✓ / 🟡 / 🔴 |
  ```

### review 实施计划的特殊约束

**review 不是施工类目的** —— 不产出"修改代码的阶段骨架"。

review 的"实施计划"章节**等价于**：
- **局部 review**：发现清单 + 建议 → 针对 Critical / Major 条目，给出推荐的下一步 pax 调用

  ```
  针对发现的问题，推荐的下一步：
    1. 🔴 <条目> → /pax --fix "<描述>"
    2. 🟡 <条目> → /pax --reshape "<描述>"
    ...
  ```

- **全局 review**：演进路线 → 每一项推荐 pax 调用

  ```
  入口：从 <Tier 1 第一项> 开始

  Tier 1
  1. <条目> — 推荐下一步: /pax --<目的> "<描述>" — 依赖: 无
  2. ...
  ```

**不写**通用施工字段（目标 / 具体动作 / 交付物 / 验收 / 施工纪律 / 回滚点）——review 不改代码，那些字段由后续每个 pax 调用各自承担。

**审批通过后用户不应该说"完成整个计划"** —— review 的"整个计划"要求按建议 / 演进路线逐项单独 pax，不是一次性推进。Phase 5 会明确提示。

---

### review 的生产级重点

review **本身就是按生产级 8 维度扫描现有代码**，作为发现清单的骨架。无论局部还是全局，生产级 8 维度扫描都是 review 的核心工作。

---

### review 的反模式（计划里出现即不合格）

- ❌ 高大上但不可行动的抽象清单（"提升可维护性" —— 怎么提升？改哪里？）
- ❌ 发现清单里的条目没有"建议目的"
- ❌ 全局 review 没有 Tier 分层或 Tier 划分没有依据
- ❌ 全局 review 没有"从哪里开始"的明确入口
- ❌ review plan 里有施工阶段的"具体动作"字段 —— review 不直接施工
- ❌ 忽视风格一致性扫描
- ❌ 没有对生产级 8 维度的系统扫描
- ❌ 局部 review 没有做"改动面评估"（行为变化、波及范围、夹带代码）
- ❌ 凭"感觉"给结论，没有定位到具体文件 / 行 / 符号
- ❌ 评判 API 设计 / 架构却没列对标项目（成了"我以为应该这样"，不是 review）
- ❌ 把"未来可能要的最佳实践"当 Critical / Major（应进 📎 参考点，不阻塞当下）
- ❌ 对标项目选错（如评判 web 框架却拿 OS kernel 类项目对比）—— 必须同语言 + 同领域
- ❌ 对比矩阵单元格写"差不多" / "类似" 等空话 —— 必须给可比维度

### review 完成后的下一步

```
✅ Review 已产出。

这是一份核查报告，**不直接施工**——它指引后续多次 pax。

下一步建议：
  针对 <第一项 Critical / Tier 1 条目> 做：
    /pax --<目的> "<描述>"

完成该项后回到 review 报告，选下一项继续。

如需永久留存本次审查结果：
  /report decision
```
