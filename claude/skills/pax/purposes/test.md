## 目的：test（测试覆盖盲区分析 + 生成）

**适用**：为现有代码补齐测试——找覆盖盲区、生成边界 / 错误路径 / 属性 / 模糊测试。

**与 fix 目的的回归测试的区别**：
- `fix` 的回归测试是**针对某个已发现 bug** 的防护
- `test` 是**系统性补齐**某个模块 / 某段代码的测试覆盖，不针对具体 bug

**test 会生成测试代码**（不是可选）—— 分析覆盖盲区的同时生成对应测试。只想要分析不要生成，该走 `--review` 聚焦测试维度。

本文件只定义 test 目的**独有**的约束，通用维度由 SKILL.md 规定。

---

### test 独有的必展开章节

#### 1. 范围与现状

必须精确：
- **范围**：哪个模块 / 文件 / 函数 / 特性是本次补测试的对象
- **现有测试状况**：
  - 数量（单元 / 集成 / e2e）
  - 已覆盖的场景清单（简要枚举）
  - 覆盖率数据（若有工具支持）
  - 测试风格 / 框架（沿用项目既有，不引入新框架）

**陷阱**：不看现状直接补测试，容易重复、冲突、或风格不一致。

#### 2. 覆盖盲区清单（核心章节）

按维度系统扫描，每类盲区必须有具体条目或明确标注"本模块无此类路径"：

- **Happy path 盲区**：核心正常流程未覆盖的分支
- **边界条件盲区**：
  - 空输入 / 单元素 / 最大容量
  - 零 / 负数 / NaN / 溢出
  - 越界索引 / 空指针
  - 时间边界（闰年 / 跨时区 / 时钟回拨）
- **错误路径盲区**：每种错误类型的触发与处理
- **并发盲区**：竞态 / 死锁 / 顺序无关性（若适用）
- **资源盲区**：泄漏 / 已关闭资源 / 耗尽场景（若适用）
- **回归盲区**：已知修过但无测试锁定的 bug（从 git log / issue 扫）
- **集成盲区**：模块间交互 / 外部依赖的 contract test
- **属性 / 模糊测试盲区**：不变量是否该用 property-based / fuzz 覆盖

每条盲区标注：
- 具体位置（文件 / 函数 / 分支）
- 盲区类型
- 建议的测试类型（单元 / 集成 / 属性 / 模糊）
- 优先级（必补 / 应补 / 可补）

**陷阱**：只扫 happy path 盲区是最常见的失败。边界 + 错误路径盲区才是生产事故的真正来源。

#### 3. 测试生成策略

必须明确：
- **测试层级**：单元 / 集成 / e2e 如何分配（通常单元覆盖纯逻辑 + 边界，集成覆盖 I/O，e2e 覆盖关键业务路径）
- **测试风格**：沿用项目既有（AAA / Given-When-Then / 表驱动）
- **断言策略**：确切值 vs 属性 vs 快照（根据场景选）
- **测试隔离**：mock / fake / 真实依赖（偏向真实 I/O 的集成测试，除非成本过高）
- **测试命名**：描述场景的名字（`parse_empty_input_returns_error` 而非 `test1`）

#### 4. 测试质量自检

生成的测试必须达到生产级测试标准：
- 独立可运行，不依赖执行顺序
- 失败时消息可操作（看 diff 就知道哪里出错）
- 不引入 flaky（时序依赖、随机种子不固定、网络依赖）
- 覆盖率不是 KPI —— **有意义的覆盖 > 刷数字**
- 不 mock 自己在测的代码
- 不测实现细节，测外部可观测行为

### test 实施计划的特殊约束（施工契约）

**test 是施工类目的** —— 产出"修改代码的阶段骨架"，因为 test 必然生成测试代码。

**阶段骨架**：

```
阶段 0: 基线捕获
  - 具体动作：
    1. 运行现有全量测试，记录通过数量和耗时
    2. 若有覆盖率工具，记录当前覆盖率
    3. 记录现有测试数量（按层级）
  - 验收：基线存档

阶段 1: 盲区确认
  - 具体动作：对照"覆盖盲区清单"在代码中逐项验证（的确未覆盖）
  - 交付物：已确认的盲区最终清单
  - 验收：每条盲区已定位到具体代码分支

阶段 2..N: 按优先级逐项补齐测试（每个阶段一组相关盲区）
  - 具体动作：
    1. 先写失败的测试（证明盲区存在）
    2. 确认测试确实失败（若是 happy path / 边界 的正确场景，测试应通过 —— 验证它能捕获预期行为；若是未来 bug 防护，测试应通过表达不变量）
    3. 确保测试命名描述场景
    4. 每组一个独立 commit
  - 验收：
    - 新测试通过（或按预期失败捕获问题）
    - 现有测试仍全过
    - 无 flaky（重跑 3 次稳定）

阶段 N+1: 测试质量审视
  - 具体动作：逐项对照"测试质量自检"清单
  - 交付物：质量自检报告
  - 验收：每条自检项通过；任何发现的问题成为独立修复 commit

阶段 N+2: 全量验证
  - 具体动作：
    1. 运行全量测试
    2. 若有覆盖率工具，对比阶段 0 基线
    3. 检查 CI 配置是否需要更新
  - 验收：
    - 全量通过
    - 覆盖率上升（或至少不下降）
    - CI 正确运行新测试
```

**每阶段独立可提交、可回滚**。禁止"一口气写完全部测试再跑"。

**阶段的推进优先级**：必补 → 应补 → 可补。时间有限时完成必补即可停。

### test 的生产级重点（施工时必须贯穿）

- **测试必须独立**：不依赖执行顺序、不依赖外部服务（除非明确集成测试）
- **测试必须稳定**：无 flaky，无时序 / 网络 / 随机依赖
- **测试必须可读**：名字描述场景，失败消息可操作
- **测试必须维护友好**：不 over-mock，不测实现细节
- **测试数据必须贴近真实**：不用 `foo/bar/baz`，用接近业务含义的数据

---

### test 的施工纪律（写入计划的"施工纪律"字段）

- 每一组相关测试一个独立 commit（message 明确"test: add <scenario>"）
- 每组 commit 后跑一遍新测试（应通过或按预期失败）
- 每阶段末跑全量测试（确保没破坏现有）
- 测试代码质量也是生产级代码 —— 同样适用生产级 8 维度（可读、可维护、无魔数）
- 不允许"为了覆盖率加的垃圾测试"—— 每个测试必须有明确的场景意图
- 不允许"把实现拷到测试里验证" —— 测试验证外部行为，不验证内部实现

---

### test 的反模式（计划里出现即不合格）

- ❌ 没有"覆盖盲区清单"，凭感觉补测试
- ❌ 只扫 happy path 盲区，忽略边界 / 错误路径 / 并发
- ❌ 为了覆盖率刷数字，写无意义的 assertion
- ❌ 测试依赖执行顺序 / 共享可变状态
- ❌ Mock 到几乎没测真实代码
- ❌ 测试名字叫 `test1` / `test_foo` / `it_works`
- ❌ 把"测试生成"放到一个大阶段一次性做完 —— 违反独立可验证
- ❌ 引入和项目现有测试风格不一致的新框架 / 新约定
- ❌ 测试代码本身不达到生产级（有魔数、重复、难读）

---

### test 的风险与回滚

- 某阶段新测试大量失败于现有代码 → 说明发现了真实 bug，停下记录，回 `/pax --fix` 规划修复
- 某阶段新测试 flaky → 停下排查根因（通常是时序或共享状态），不允许加 retry 掩盖
- 覆盖率下降 → 某个现有测试被误删或影响，回滚
- 现有测试被意外改动 → 回滚该阶段，测试修改必须独立 commit 且有明确理由
