## 核心理念

调试的目标是**理解根因**，而不是急于修改代码。

一个合格的调试过程必须：

1. **精确复现**——确保错误可稳定触发
2. **分类定性**——明确错误类型，选择对应追踪策略
3. **追踪根因**——修根因的前提是找到根因，不是找到症状
4. **验证假设**——用证据确认，而非凭直觉猜测

本命令只做诊断，不做治疗。确认根因后，交由 `/fix` 或用户自行修复。

| 模式 | 格式 | 行为 |

| 参数 | 说明 |
|------|------|
| `[target: <path>]` | 将根因分析聚焦到指定文件或模块 |
| `[auto]` | 无人值守模式——跳过所有确认提示，自动完成全流程 |

## Phase 1: 错误捕获

解析 `$ARGUMENTS`，确定错误来源。

**自动捕获时**，根据构建命令获取策略（用户提供 > CLAUDE.md 声明 > 自动检测），确定并执行构建与测试命令。使用 `;` 分隔以收集所有输出。

若构建/测试**全部通过**：
```
✅ 构建和测试均通过，未发现错误。如需调试特定行为，请直接粘贴相关输出。
```
终止。

**若 `[target: <path>]` 已指定**：后续所有分析优先围绕该文件/模块展开，但不排除根因位于其依赖中的可能。

记录开始时间。

## Phase 2: 错误分类

| 类型 | 特征 | 后续策略 |
|------|------|----------|
| **编译错误** | 类型错误、未定义符号、语法错误 | 直接定位出错行 |
| **链接错误** | undefined reference、symbol not found | 检查依赖声明和构建配置 |
| **运行时 panic** | panic! / segfault / stack overflow / OOM | 追踪调用栈 |
| **测试失败** | assertion failed、expected/got | 比较预期与实际，追溯数据流 |
| **逻辑错误** | 输出不符合预期但无 crash | 追踪数据变换路径 |
| **间歇性错误** | flaky test、race condition | 识别并发/时序依赖 |

提取错误的**最深层 cause**（通常不是第一行），定位到具体文件、函数、行号。

声明：`错误类型：X | 直接位置：<file>:<line> | 疑似根因区域：<描述>`

---

## Phase 3: 根因追踪

从直接位置出发，向上追溯调用链，向下检查被调用的逻辑。

**追踪策略按错误类型选择**：

- **编译错误**：读取出错上下文（前后 20 行），追踪类型定义来源
- **链接错误**：检查构建配置中的依赖声明、链接目标、符号导出
- **运行时 panic**：逐帧分析调用栈，重点检查 `unwrap()`/`expect()`、数组越界、空指针
- **测试失败**：读取测试代码 → 追踪被测函数完整数据流 → 对比预期/实际差异
- **逻辑错误**：从输出点回溯，逐步检查每一次数据变换是否符合预期
- **间歇性错误**：检查共享状态访问模式，识别缺失的同步原语，检查测试隔离性

**若指定了 `[target: <path>]`**：优先在 target 文件及其直接依赖/被依赖中搜索，但若证据指向其他位置，仍然跟踪。

列出 1–3 个根因假设，按可能性排序：

```
假设 1（最可能）：<具体描述，指向代码位置>
  证据：<支持该假设的代码特征>

假设 2：<...>
  证据：<...>

假设 3（若有）：<...>
  证据：<...>
```

---

## Phase 4: 假设验证

对每个假设，用最小代价验证。**不改动项目代码**，仅做观察和诊断：

1. **静态分析**：重新阅读代码，逻辑推演，检查边界条件
2. **添加临时日志**：关键路径插入 `eprintln!` / `stderr`，重新运行（验证后**立即回滚**临时改动）
3. **最小复现**：构造能稳定触发的最小输入或最小测试用例
4. **注释隔离**：临时注释可疑代码块，确认错误消失（验证后**立即回滚**）

> 重要：本阶段的任何临时代码改动（日志、注释）仅用于验证，验证完成后必须 `git checkout` 恢复原状。调试不留痕迹。

**根因确认时**：

```
✅ 根因确认：<描述>
位置：<file>:<line> / <function>
触发条件：<什么情况下会触发>
影响范围：<仅此处 / 多处调用方受影响>
```

**根因无法确认时**：

若所有假设均被排除 → 扩展检查构建配置、依赖版本、环境差异。

若经过充分分析仍无法确认根因：

```
⚠️ 根因未确认。当前最可能的假设：
  <假设描述>
  已尝试的验证方法及结果：<...>

建议：
  - 使用 /discuss 对根因假设进行多角色对抗分析
  - 在更完整的环境中手动复现
```

仍然进入 Phase 5 生成报告，记录已有分析成果。

## 后续建议

根因确认后，根据情况向用户建议：

| 场景 | 建议 |
|------|------|
| 根因明确，修复方向清晰 | 使用 `/fix` 执行完整修复流程（定位→修复→测试→提交） |
| 根因涉及设计层面的权衡 | 使用 `/discuss` 进行多角色对抗分析，确定最佳修复策略 |
| 同类模式可能存在于其他位置 | 提示用户全局搜索类似模式，评估影响范围 |
| 根因未确认 | 建议补充信息、手动复现、或使用 `/discuss` 深入分析假设 |
