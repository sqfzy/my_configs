## 核心理念

功能开发的质量取决于设计讨论的深度是否匹配需求的模糊度。

- **需求清晰**（默认模式）：快速评审后直接实现——3 角色 2 轮，不拖泥带水
- **需求模糊**（`[deep]` 模式）：深度对抗讨论，生成候选方案，多轮收敛后再实现——避免"实现完发现方向错了"

两种模式共享同一套实现流程（Phase 3-5），区别仅在于设计阶段（Phase 2）的讨论深度。

---

## 参数解析

- **需求描述**（必填）：要实现的功能、要解决的问题
- `[deep]`：深度设计模式——Phase 2 使用 5 角色多轮对抗讨论 + 候选方案生成。适用于需求模糊、方案有根本分歧的场景
- `[no-commit]`：完成后不自动提交
- `[no-tests]`：跳过测试编写（不推荐，仅用于 spike/探索性实现）
- `[target: <path>]`：目标模块或目录
- `[rounds: N]`：讨论轮数（覆盖自动判断，仅 deep 模式有效）
- `[roles: N]`：参与角色数（默认：默认模式 3，deep 模式 5）
- `[auto]`：无人值守模式——需求不确定点按最保守理解处理，方案直接执行

### 模式自动推断

若用户未显式指定模式参数，从 prompt 中推断：

| 关键词/意图 | 推断模式 |
|-------------|----------|
| "实现"、"做一个"、"加一个"、"开发" | 默认模式 |
| "怎么设计"、"方案对比"、"不确定用什么方案"、"讨论一下怎么做" | deep |

- 多个关键词匹配多个模式时，按合理顺序组合执行
- 无法推断时使用默认模式
- 推断结果输出一行声明：`▶ 推断模式：<mode>（从"<关键词>"推断）`

### deep 模式（深度对抗讨论）

#### 2.0 复杂度评估与轮数确定

| 复杂度 | 特征 | 最低轮数 | 终止条件 |
|--------|------|----------|----------|
| 低 | 方案基本明确，只有少量细节需要讨论 | 3 | 连续 1 轮无新论点 |
| 中 | 2–3 个合理方案，存在非显然权衡 | 5 | 连续 1 轮无新论点 |
| 高 | 方案根本分歧、跨模块影响大、约束模糊 | 8 | 连续 2 轮无新论点 |

#### 2.1 候选方案生成

在角色讨论之前，先独立生成 2–3 个候选方案：

```
## 候选方案

### 方案 A：<标题>
- 核心思路：<一句话>
- 数据结构：<关键类型>
- 接口设计：<公共 API 签名>
- 优势：<...>
- 劣势：<...>
- 适用条件：<什么情况下这是最优解>

### 方案 B：<标题>
- ...
```

候选方案之间必须有**实质性差异**——不同的数据结构、不同的抽象层级、不同的性能/可维护性权衡。

#### 2.2 多角色对抗讨论

从角色库中选出指定数量（默认 5 个）角色，围绕候选方案展开对抗讨论。

每轮每个角色发言中额外包含：`支持方案：<A / B / C / 新方案>`

讨论结束后，将共识转化为可直接执行的设计文档：

```

- [分歧点] → [解决方式及理由]

### 未解决的权衡
- [冲突] → 选择：<最终选择>，原因：<...>

### 核心数据结构 / 模块结构 / 接口设计 / 实现步骤

### 关键设计决策
| 决策 | 选择 | 否决项 | 理由 |
|------|------|--------|------|
| ... | ... | ... | ... |

### 讨论摘要
- 参与角色：<列表>
- 讨论轮数：N
- 主要争议：<一句话>
- 收敛路径：<哪些论点最终说服了多数角色>
```

## Phase 3: 测试先行（TDD）

除非指定 `no-tests`，先写测试，再写实现。

### 3.0 Benchmark 基线（若存在）

检测并记录现有 benchmark 基线。若存在 benchmark，运行并按 bench-data 约定持久化。

### 3.1 测试用例设计

列出测试场景：
```
测试用例清单：
  [ ] happy path: <描述>
  [ ] 边界条件: <空输入 / 最大值 / 临界值>
  [ ] 错误路径: <各类错误的触发和处理>
  [ ] 集成场景: <与现有模块的交互>（若适用）
```

### 3.2 编写测试骨架

创建测试文件，写出所有测试函数的骨架（编译通过但全部 `#[ignore]` 或 `todo!()`），验证能编译通过。

## Phase 4: 增量实现

按设计方案分步实现，每完成一个独立单元即运行相关测试验证。

### 实现顺序原则

1. **核心数据结构优先**：先定义类型，让编译器辅助验证设计
2. **由内向外**：先实现纯逻辑函数，再实现 I/O 和集成层
3. **每步可编译**：每次改动后立即验证编译通过
4. **逐步解除 `todo!()`**：每实现一个函数，同步取消对应测试的 ignore，确认测试通过

### 语言特定要求

**Rust**：
- 错误处理使用 `std::error::Error` 或项目已有 error 类型，不使用裸 `unwrap()`
- 添加 `#[instrument(err)]`（若项目使用 `tracing`）
- unsafe 块必须附有 safety 注释

**C++**：
- 遵循项目已有的内存管理约定（RAII / 智能指针）
- 模板边界使用 concepts（C++20）
- 使用 `spdlog` 记录关键路径日志

**通用**：
- 非显然逻辑必须有注释，说明 **why** 而非 **what**
- 日志覆盖关键函数入口/出口和所有错误分支

## Phase 5: 全量验证

```
根据构建命令获取策略，依次执行构建、测试、lint。若存在 benchmark 基线，运行 benchmark 并按 bench-data 约定持久化。
```

- ✅ 全部通过 → Benchmark 回归检查（若有基线），然后进入 Phase 6
  - 无退化 → 继续
  - 退化 ≥5% → 告知用户，分析原因
  - 无基线 → 跳过
- ❌ 失败 → 修复后重新验证
- ⚠️ Clippy / lint 警告 → 逐一处理

[ ] 错误信息是可操作的

## 异常流程处理

### 讨论无法收敛（deep 模式）

若最后一轮仍有 2+ 角色坚持不同方案：
```
⚠️ 讨论未完全收敛。剩余分歧：
  - <角色A> 坚持方案 X，因为 <理由>
  - <角色B> 坚持方案 Y，因为 <理由>

请选择方案，或回复「加轮 N」追加讨论。
```

**`auto` 模式**：选获得最多角色支持的方案。票数相同选更保守的。

### 实现阶段发现设计缺陷

```
⚠️ 实现中发现设计缺陷：<具体问题>

选择：
  [1] 局部修补——不影响整体方案
  [2] 回到讨论——此缺陷可能影响方案选择
```

**`auto` 模式**：可局部修补则自动修补；涉及核心设计则自动终止并保存报告。
