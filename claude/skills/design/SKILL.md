## 核心理念

`/feature` 适合目标明确的功能——知道做什么、大致知道怎么做，缺的只是实现。

`/design` 面向更早期的阶段——需求有歧义、方案有根本分歧、多种技术路径各有利弊。这种情况下先动手写代码是浪费时间，因为大概率要推翻重来。

流程分三段：
1. **需求阶段**——把"要做什么"讨论清楚
2. **设计阶段**——把"怎么做"通过多角色对抗讨论透，收敛为可执行的设计文档
3. **实现阶段**——按收敛后的设计直接编码，不再重复评审

与手动 `/discuss` → `/feature` 的区别：discuss 的结论直接转化为 feature 的设计输入——不重复需求理解，不重复方案评审，上下文不断裂。

## 参数解析

- **需求描述**（必填）：要实现的功能、要解决的问题
- `[no-commit]`：完成后不自动提交
- `[target: <path>]`：目标模块或目录
- `[rounds: N]`：讨论轮数（覆盖自动判断）
- `[roles: N]`：参与角色数（默认 **5**）
- `[auto]`：无人值守模式——需求不确定点按最保守理解处理，方案直接执行，commit 用候选 1

## Phase 1: 需求拆解与边界澄清

### 1.1 理解现有代码库

在讨论需求之前，先阅读项目关键文件建立上下文：
- 构建配置（依赖、feature flags）
- 相关模块的现有接口
- 项目代码风格约定（命名、错误处理模式、注释风格）

### 1.2 需求分析

对用户的需求描述进行拆解：

```
## 需求理解

### 核心目标
<一句话：这个功能要解决什么问题>

### 功能边界
**In scope（本次实现）**：
- ...

**Out of scope（明确不做）**：
- ...

### 输入 / 输出
- 输入：<数据类型、来源、约束>
- 输出：<数据类型、格式、副作用>

### 边界条件 & 错误情况
- <空输入 / 超大输入 / 并发访问 / 权限不足 / ...>

### 与现有代码的集成点
- <需要修改或扩展的现有接口>
- <需要新增的模块或文件>

### 不确定点（需用户确认）
- [ ] <问题1>
- [ ] <问题2>
```

若存在不确定点，**在此处暂停，等待用户回答**，再继续。

**`auto` 模式**：不暂停，对每个不确定点选择最保守的理解（功能范围最小、约束最严格的解读），并在输出中标注 `[auto: 保守假设]`。

## Phase 2: 深度方案讨论

需求边界确认后，进入核心环节——通过多角色对抗讨论，从多个候选方案中收敛出最优设计。

### 2.0 复杂度评估与轮数确定

从需求中解析 `[rounds: N]`：

- **用户指定轮数**：严格执行，无上限。
- **未指定**：以**收敛为目标**，不设固定上限。按复杂度确定最低轮数，之后持续讨论直到所有角色在最后一轮无新的实质性论点：

| 复杂度 | 特征 | 最低轮数 | 终止条件 |
|--------|------|----------|----------|
| 低 | 方案基本明确，只有少量细节需要讨论 | 3 | 连续 1 轮无新论点 |
| 中 | 2–3 个合理方案，存在非显然权衡 | 5 | 连续 1 轮无新论点 |
| 高 | 方案根本分歧、跨模块影响大、约束模糊 | 8 | 连续 2 轮无新论点 |

声明：`复杂度：X → 讨论轮数：N 轮`

### 2.1 候选方案生成

在角色讨论之前，先独立生成 2–3 个候选方案，确保讨论有具体对象而非空谈：

```
## 候选方案

### 方案 A：<标题>
- 核心思路：<一句话>
- 数据结构：<关键类型>
- 接口设计：<公共 API 签名>
- 优势：<...>
- 劣势：<...>
- 适用条件：<什么情况下这是最优解>

### 方案 B：<标题>
- ...

### 方案 C：（可选）
- ...
```

候选方案之间必须有**实质性差异**——不同的数据结构、不同的抽象层级、不同的性能/可维护性权衡。变体级别的差异（换个命名、调整参数顺序）不算独立方案。

### 2.2 角色选择

从角色库中选出指定数量（默认 5 个）的角色，要求：
- 角色之间必须存在**真实的价值观冲突**
- 优先构建多维张力（保守 vs 激进、短期 vs 长期、性能 vs 可维护性等）
- 说明选择理由及预期冲突点

**角色库**：

| 代号 | 角色 | 核心偏好 | 适用场景 |
|------|------|----------|----------|
| R1 | 风险卫士 | 识别风险、边界条件、失败模式 | 架构决策、系统设计 |
| R2 | 极简主义者 | 反对一切不必要的复杂性 | 代码设计、API 设计 |
| R3 | 性能狂热者 | 吞吐量 / 延迟 / 资源效率优先 | 性能敏感场景 |
| R4 | 实用主义者 | 只接受实证支撑的方案，拒绝纯理论 | 技术选型 |
| R5 | 第一性原理者 | 从根本需求出发，质疑所有假设 | 需求分析、产品决策 |
| R6 | 维护性倡导者 | 三年后的可读性和可维护性优先 | 代码审查、长期项目 |
| R7 | 用户代言人 | 终端用户体验为唯一标准 | 产品/UX 决策 |
| R8 | 激进创新者 | 无视当前约束，追求理论最优解 | 架构探索 |
| R9 | 成本审计者 | 资源、时间、金钱成本为核心标准 | 资源分配 |
| R10 | 安全专家 | 安全性、隐私、合规为最高优先级 | 安全设计 |
| R11 | 怀疑论者 | 默认质疑一切方案，要求严格举证 | 逻辑验证 |
| R12 | 系统思维者 | 全局影响、二阶效应、跨系统依赖 | 复杂系统 |
| R13 | 测试驱动者 | 测试覆盖、可测试性、回归安全 | 接口设计 |

**自定义角色**（当预定义角色无法覆盖领域需求时）：

```
【自定义】<角色名>
核心偏好：<该角色的判断标准和优先级>
立场预期：<与其他角色的主要冲突点>
```

### 2.3 结构化讨论

每轮每个角色的发言格式：

```
【角色名 | 立场标签】
支持方案：<A / B / C / 新方案>
论点：...（须指向具体的技术决策，如数据结构选择、接口签名、模块划分）
反驳 [对方角色]：...   ← 第 2 轮起必填，须针对具体论点
联盟 [对方角色]：...   ← 可选
立场修正：...          ← 仅最后一轮必填
```

**轮次规则**：
- 第 1 轮：各角色独立评估候选方案，声明支持哪个并阐述理由
- 中间轮：必须直接反驳其他角色的具体论点；允许提出混合方案（取 A 的数据结构 + B 的接口设计）
- 最后一轮：每个角色声明立场修正——哪些反驳改变了想法、哪些坚持不让及原因

**质量要求**：
- 每个角色须有明确立场，"一方面…另一方面…"而无结论是失败模式
- 反驳必须指向对方的具体论点，不得打稻草人
- 讨论必须围绕候选方案展开，不能脱离成抽象理论

### 2.4 收敛为设计文档

讨论结束后，将共识转化为**可直接执行**的设计文档：

```
## 设计文档

### 已解决的分歧
- [分歧点] → [解决方式及理由]

### 未解决的权衡
- [冲突]：[角色A] vs [角色B]
  → 选择：<最终选择>，原因：<...>
  → 若 [条件X] 变化，应重新评估

### 核心数据结构
\```<lang>
// 关键类型定义（真实代码，非伪代码）
\```

### 模块结构
\```
新增 / 修改的文件：
  <path>  —  <职责>
  <path>  —  <职责>
\```

### 接口设计
\```<lang>
// 对外暴露的公共接口（完整签名）
\```

### 实现步骤
<按执行顺序编号，说明每步做什么、为什么这样拆解>
1. ...
2. ...

### 关键设计决策
| 决策 | 选择 | 否决项 | 理由 |
|------|------|--------|------|
| 数据结构 | <X> | <Y, Z> | <讨论中的关键论点> |
| 错误处理 | <X> | <Y> | <...> |
| ... | ... | ... | ... |

### 讨论摘要
- 参与角色：<列表>
- 讨论轮数：N
- 主要争议：<一句话>
- 收敛路径：<哪些论点最终说服了多数角色>
```

### 2.5 用户确认

输出：
```
✋ 设计文档已就绪（经 N 轮 M 角色对抗讨论收敛）。
   回复「继续」开始实现。
   回复并提出修改意见。
   回复「重新讨论 <焦点>」重开讨论。
```

**在此处暂停，等待用户确认。**

**`auto` 模式**：不暂停，直接采纳设计文档开始实现。

## Phase 3: 测试先行

### 3.0 Benchmark 基线（若存在）

```bash
find . -path "*/benches/*.rs" -o -name "*.cpp" -exec grep -l "BENCHMARK" {} \; -o -name "bench_*.py" 2>/dev/null | head -10
```

若存在 benchmark，运行并保存基线。

### 3.1 测试用例设计

基于设计文档中的接口设计和边界条件，列出测试场景：

```
测试用例清单：
  [ ] happy path: <描述>
  [ ] 边界条件: <从 Phase 1 边界条件列表推导>
  [ ] 错误路径: <从接口设计的错误返回类型推导>
  [ ] 集成场景: <与现有模块的交互>（若适用）
```

### 3.2 编写测试骨架

创建测试文件，写出所有测试函数的骨架（编译通过但全部 `#[ignore]` 或 `todo!()`）：

```
Rust：  src/<module>/tests.rs 或 tests/<feature>_test.rs
C++：   tests/<feature>_test.cpp
Python：tests/test_<feature>.py
Go：    <module>_test.go
```

验证骨架编译通过：

根据项目构建系统，验证测试代码能编译/收集通过（不实际运行）。

## Phase 4: 增量实现

按设计文档的实现步骤分步执行。

### 实现顺序原则

1. **核心数据结构优先**：先定义类型
2. **由内向外**：先纯逻辑，再 I/O 和集成层
3. **每步可编译**：每次改动后立即验证
4. **逐步解除 `todo!()`**：每实现一个函数，同步取消对应测试的 ignore

### 语言特定要求

**Rust**：
- 错误处理使用项目已有的 error 类型，不使用裸 `unwrap()`
- unsafe 块必须附 safety 注释

**C++**：
- 遵循项目内存管理约定（RAII / 智能指针）
- 模板边界使用 concepts（C++20）

**通用**：
- 非显然逻辑必须有注释，说明 **why**
- 日志覆盖关键函数入口/出口和所有错误分支

## Phase 5: 全量验证 + 提交

### 5.1 验证

若用户提供了构建/测试/lint 命令则优先使用；否则根据项目构建系统和配置，自行确定并执行构建、测试与静态检查命令；若项目无测试或 linter 则跳过对应步骤。

**结果处理**：
- ✅ 全部通过 → 运行 Benchmark 回归检查（若 Phase 3.0 记录了基线），继续提交
  - 退化 ≥5% → 告知用户，分析原因
- ❌ 失败 → 修复，重新验证
- ⚠️ Clippy / lint 警告 → 逐一处理

### 5.2 自查清单

```
[ ] 所有测试通过（包括已有测试，无回归）
[ ] 没有遗留的 todo!() / unimplemented!()
[ ] 没有调试用的临时 println! / console.log
[ ] 公共接口有文档注释
[ ] 错误信息是可操作的
[ ] 新文件已加入构建系统
```

### 5.3 提交

除非指定 `no-commit`，生成 Conventional Commits message：

```
feat(<scope>): <subject>

<body: 设计讨论的关键结论、为什么选择这个方案>
```

生成 **2 个候选**：

```
── 候选 1（推荐）──────────────────────────────
<详细版，body 包含设计决策摘要>

── 候选 2（最简）──────────────────────────────
<subject only>
```

询问用户：`使用候选 1？[1/2/e(自己输入)/回车默认1]`

**`auto` 模式**：直接使用候选 1，不询问。

```bash
git add -A
git commit -m "<confirmed message>"
```

输出：`✅ 已提交 <short-hash>: <subject>`

## 异常流程处理

### 讨论无法收敛

若最后一轮仍有 2+ 个角色坚持不同方案且无法调和：

```
⚠️ 讨论未完全收敛。剩余分歧：
  - <角色A> 坚持方案 X，因为 <理由>
  - <角色B> 坚持方案 Y，因为 <理由>

这是一个需要用户决策的权衡：
  → 若 <条件> 优先，选方案 X
  → 若 <条件> 优先，选方案 Y

请选择方案，或回复「加轮 N」追加 N 轮讨论。
```

**`auto` 模式**：自动选择获得最多角色支持的方案。若票数相同，选更保守（风险更低、复杂度更低）的方案。

### 实现阶段发现设计缺陷

若 Phase 4 实现过程中发现设计文档的方案有问题（类型不匹配、遗漏的约束、性能不可行）：

```
⚠️ 实现中发现设计缺陷：
  <具体问题>

选择：
  [1] 局部修补——调整设计的这一处，不影响整体方案
  [2] 回到讨论——此缺陷可能影响方案选择，需要重新评估
```

**`auto` 模式**：若缺陷可局部修补（不影响核心数据结构和接口），自动修补并继续。若缺陷涉及核心设计（需更换方案），自动终止并保存报告。
